TermMeaningHow to use it in our analysis
CodecStorage algorithm/container, e.g. WAV/PCM, FLAC, MP3, Opus.Controlled input-compression tests change this while holding utterance and ASR model fixed.
BitrateBits per second in the encoded audio stream, usually kbps.For lossy codecs, lower bitrate usually removes more acoustic detail.

Implication for the next experiment: Common Voice can support robustness stress tests, but it is already MP3, so re-encoding creates second-generation lossy artifacts. FairSpeech WAV is the cleaner starting point for a controlled codec/bitrate study now that the raw audio is available.

2.5 Batch-size policy

Do not fix one number of samples per batch.

Current inference code exposes a fixed --batch_size knob, but the better implementation is a pre-split duration-bucketed batch plan. FairSpeech clips range from 1.36 s to 65.62 s, and padded batches scale roughly with num_samples × longest_clip_in_batch, so each batch should contain a variable number of similar-duration samples under a per-model audio-seconds budget.

PolicyImplementation decision
Batch unitUse max audio seconds per batch plus a per-model max-samples cap, not a fixed sample count.
Pre-splittingSort or bucket files by duration, then form batches whose clips have similar duration and whose total seconds fit the budget.
CalibrationRun real batch-size/seconds profiling in the pilot gate, because VRAM depends on the VM, model implementation, and decoding path.
Safety marginChoose the largest budget that avoids OOM and keeps peak GPU memory below the VM target, e.g. ~80–85%.
FallbackIf dynamic batching is not ready, use conservative fixed batch sizes per model only as a fallback.

Example: a 40-second budget might batch 8×5 s clips, 4×10 s clips, 2×20 s clips, or 1×40–60 s long clip. The batch has variable sample count but roughly bounded compute/memory.
