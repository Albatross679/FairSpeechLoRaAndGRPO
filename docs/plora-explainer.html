PLoRA: Packed LoRA Fine-Tuning

How to run many LoRA HP configs simultaneously on one GPU — up to 7.5× faster sweeps

1. The Problem

When tuning LoRA hyperparameters (rank, alpha, learning rate, etc.), the standard approach is sequential: load the frozen base model, attach one adapter, train it, tear it down, repeat for the next config. This is wasteful because:

  • The frozen base model (often 7–70B parameters) must be loaded every time, but it never changes between configs
  • Each LoRA adapter is tiny — a rank-64 adapter on a 7B model is only ~3.4% of total parameters
  • The GPU sits with vast unused memory that could host additional adapters
  • HP sweeps of 20–120 configs take hours or days running one-at-a-time

2. Key Insight

In LoRA fine-tuning, the base model weights $W$ are frozen. Only the small adapter matrices $A$ and $B$ are trained. Since the base model is identical across all HP configs, PLoRA loads it once and attaches multiple adapters simultaneously. Each adapter gets its own data stream, its own gradients, and its own optimizer — but they all share the expensive base-model forward pass.

Core idea: A rank-64 LoRA adapter on Qwen-2.5-7B uses only 3.4% of GPU memory relative to the base model. That means you can fit up to 10 concurrent adapters on a single A100-40GB without running out of memory.

Why this works for HP sweeps

An HP sweep explores configs that differ only in adapter parameters (rank, alpha, LR, dropout). The base model never changes. PLoRA exploits this by amortizing the cost of the largest memory and compute consumer — the frozen base — across all configs.

3. How It Works

Standard LoRA forward pass

In standard LoRA, a single input $x$ passes through the frozen base weights $W$ and one adapter:

$$y = x \cdot W + \alpha \cdot x \cdot B \cdot A$$

Where $B \in \mathbb{R}^{d \times r}$ and $A \in \mathbb{R}^{r \times d}$ are the low-rank adapter matrices, $r$ is the rank, and $\alpha$ is the scaling factor.

Packed LoRA forward pass

PLoRA batches inputs from different adapter configs into a single forward pass. Given $N$ packed adapters with inputs $X = [x_1, x_2, \ldots, x_N]$:

$$y_i = x_i \cdot W + \alpha_i \cdot x_i \cdot B_i \cdot A_i \qquad \text{for } i = 1, \ldots, N$$

The critical structure: all inputs share the same $W$ multiplication, but each gets routed to its own adapter $(B_i, A_i)$ with its own scaling $\alpha_i$.

Gradient handling

Gradients remain naturally isolated because each adapter has its own $(A_i, B_i)$ weight matrices and its own input stream. The base model $W$ is frozen (no gradients flow to it), so there is no gradient interference between adapters. PLoRA implements custom CUDA kernels that:

  1. Partition the gradient computation along the output dimension $k$ to correctly associate each adapter's slice with its corresponding input/output
  2. Reduce gradients over the appropriate dimensions per-adapter
  3. Maintain separate optimizer states (momentum, variance) for each config

4. The Math

Memory budget

The key constraint is fitting everything on the GPU. For a job $j$ packing $K$ adapters:

$$M_\text{base} + \sum_{k=1}^{K} M_{\text{lora},k} \;\leq\; C \cdot M_\text{gpu} \cdot d_j$$

Where $M_\text{base}$ is the base model memory, $M_{\text{lora},k}$ is adapter $k$'s memory (weights + optimizer + activations), $C$ is a safety margin, $M_\text{gpu}$ is GPU capacity, and $d_j$ is the number of GPUs allocated.

Why so many adapters fit

Consider the memory breakdown for a rank-64 adapter on Qwen-2.5-7B:

ComponentMemory% of Base
Frozen base model (bf16)~14 GB100%
One rank-64 LoRA adapter (weights)~0.48 GB3.4%
Adapter optimizer states (AdamW)~0.96 GB6.9%
Adapter gradients~0.48 GB3.4%
Total per adapter~1.9 GB13.7%

On an A100-40GB: $40 - 14 = 26$ GB free for adapters. At ~1.9 GB each, you can pack up to 10–13 concurrent adapters (depending on activation memory sharing). Lower ranks use even less — a rank-4 adapter (like our best config) would be ~30× smaller.

Optimization objective

PLoRA minimizes makespan (total wall-clock time to finish all configs):

$$\min \; T_\text{makespan} = \max_j \; t_{\text{end},j}$$

Subject to memory constraints (above), GPU allocation limits ($\sum_j d_j \leq G$), and job non-overlap constraints on shared devices. This is NP-complete, so PLoRA uses a heuristic called Decomposed Throughput Maximization (DTM) that greedily maximizes instantaneous job throughput.

5. System Design

PLoRA operates in two phases:

Phase 1: Offline — Packing Planner

  1. Takes the full HP search space as input (e.g., 120 configs)
  2. Profiles each config's memory footprint (rank determines adapter size, batch size determines activation memory)
  3. Uses the DTM algorithm to group compatible configs into packed jobs — configs that fit together within the GPU memory budget
  4. Outputs a job schedule: which configs go in which job, how many GPUs per job

Phase 2: Online — Execution Engine

  1. Loads the base model once per GPU
  2. Attaches all packed adapters for the current job
  3. Routes each data sample to its corresponding adapter via custom CUDA kernels
  4. Computes gradients and updates each adapter independently
  5. When a job finishes, swaps in the next batch of packed adapters (base model stays loaded)

Custom CUDA kernels

The performance gains come from fused kernels that handle the packed forward and backward passes natively on the GPU, avoiding the overhead of Python-level loops or separate CUDA streams per adapter. The kernels use tiling strategies to:

  • Batch the $x \cdot W$ computation across all adapters in one matmul
  • Use block-level parallelism to compute each $x_i \cdot B_i \cdot A_i$ concurrently
  • Partition gradient accumulation to prevent cross-adapter contamination

Model quality preserved

PLoRA produces bit-identical results to sequential training — it changes how configs are scheduled and computed, not the training dynamics. The best configs found via PLoRA sweeps outperformed default HPs by up to 23.4% accuracy (CoLA on Qwen-2.5-3B), demonstrating the value of exploring more configs in the same wall-clock budget.

7. Relevance to Our Project

What we did

Our Phase 2 HP sweep ran 20 Optuna trials sequentially on Qwen3-ASR-1.7B, taking 1h 32min (~4.5 min per trial). Each trial trained for 1 epoch on a 500-sample subset.

What PLoRA could offer

Base model loading 20 times 4–7 times (once per packed batch)
Practical note: PLoRA requires custom CUDA kernels and is currently a research prototype (not yet available as a pip-installable library). For our class project's scale (20 trials, ~1.5 hours), the overhead of setting up PLoRA would likely exceed the time saved. PLoRA becomes compelling at 100+ configs or when sweeping larger models (7B+) where each trial takes 30+ minutes. Our Optuna TPE approach is appropriate for our scale — but PLoRA shows where the field is heading.
