METHODOLOGY
Testing Process
Three test prompts, each run with and without the caveman skill by independent subagents. Graded programmatically against 15 assertions.
Step 1
Design Test Prompts
Selected 3 diverse prompts: CUDA OOM debugging (technical troubleshooting), Python GC explanation (conceptual), and a Python function request (code generation). Each stresses different aspects of the skill.
Step 2
Parallel Execution
Spawned 6 subagents simultaneously — 3 with the caveman skill loaded, 3 baselines with no skill. All ran on the same model (Claude Opus 4.6) to ensure fair comparison.
Step 3
Draft Assertions
While runs were in flight, drafted 15 quantitative assertions across 5 categories: token reduction, filler elimination, article dropping, technical accuracy, and code boundary respect.
Step 4
Programmatic Grading
Wrote a Python grading script that counts words (excluding code blocks), detects filler phrases, checks for technical terms, and validates code quality. No subjective human grading in this pass.
Step 5
Benchmark & Review
Aggregated timing, token usage, and word counts. Generated an interactive eval viewer for side-by-side output comparison.
EVAL RESULTS
Per-Test Findings
All 3 evals passed all assertions. The skill reliably produces terse, technically accurate responses.

ANALYSIS
Key Insights
What the numbers tell us, and what the repo's marketing doesn't.

Readability Win, Not a Cost Win

The skill excels at information density. You get the same technical substance in half the words. For developers scanning output in a terminal, this is genuinely valuable — less scrolling, faster comprehension.

CONFIRMED

"Cuts 65% of Tokens" Is Misleading

The repo claims ~65% token reduction. This is true for output word count, but total API tokens (input + output) barely change. The skill prompt itself adds ~1K input tokens per call, partially offsetting any output savings.

CAVEAT

Code Boundaries Fully Respected

Across all evals, code blocks stayed in clean, idiomatic Python. Caveman grammar was applied only to surrounding explanation text. Variable names, docstrings, and comments remained untouched. The boundary rule works perfectly.

CONFIRMED

Technical Accuracy Preserved

No technical terms were simplified or rephrased. "Gradient checkpointing" stayed "gradient checkpointing," not "save less stuff." All expected diagnostic methods and fixes appeared in the caveman responses.

CONFIRMED

Duration Is a Wash

Caveman responses weren't consistently faster. In one eval the caveman agent was slower (CUDA OOM: 106s vs 90s), in another faster (Python fn: 32s vs 48s). The differences are noise, not signal — confounded by subagent permission-retry overhead.

INCONCLUSIVE

Filler Elimination Is 100%

Zero instances of "Sure!", "I'd be happy to", "basically", "certainly", or other filler phrases in any caveman response. The skill does exactly what it promises on the verbosity front — no pleasantries, no hedging, no preamble.

OBSERVATION
CONCLUSION
Final Verdict
B+

Good Skill, Overpromised Savings

Caveman mode delivers on readability: 53% fewer words, zero filler, full technical accuracy, and clean code boundaries. But the "saves tokens" marketing is misleading — total API cost is essentially unchanged. Use this skill when you want denser, more scannable output. Don't use it expecting cost savings.
