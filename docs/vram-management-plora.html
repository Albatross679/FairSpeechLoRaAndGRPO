VRAM Management in PLoRA

How the packed LoRA sweep fits 25 adapters on a 15 GB T4

1. The Problem

PLoRA attaches N parallel LoRA adapters to a single frozen base model and trains them simultaneously. Each adapter has its own weights, optimizer states, and gradients, so VRAM grows linearly with pack size. But different trial configs consume different amounts of memory—a rank-32 adapter targeting attention + MLP uses ~4× more VRAM than a rank-4 attention-only adapter. The sweep must decide how many adapters to pack per batch without knowing the exact configs until Optuna suggests them.

3. Layer 1: Theoretical Estimation

Before any adapter is loaded, estimate_adapter_vram_mb() computes a theoretical VRAM cost from the adapter config. The formula accounts for every component that consumes GPU memory:

$$\text{VRAM}_\text{adapter} = \underbrace{P \times 12\text{ bytes}}_{\text{weights + optimizer + gradients}} + \underbrace{300\text{ MB}}_{\text{activation overhead}}$$

where $P$ is the total number of LoRA parameters:

$$P = 2 \times h \times r \times t \times L$$
SymbolValueMeaning
$h$1,536Hidden dimension (Qwen3-ASR-1.7B)
$r${4, 8, 16, 32}LoRA rank
$t$4 or 7Target modules (attn-only = 4, attn+MLP = 7)
$L$28Transformer layers

The 12 bytes per parameter breaks down as:

ComponentBytesFormat
Weight (A + B matrices)2bf16
AdamW momentum ($m$)4fp32
AdamW variance ($v$)4fp32
Gradient2bf16

The 300 MB activation overhead is a flat constant calibrated from empirical profiling. It covers intermediate activations during forward/backward passes that gradient checkpointing does not eliminate.

4. Layer 2: Dynamic Pack Sizing

The sweep loop dynamically decides pack size for each batch. The process runs every iteration:

Ask Optuna for 1 trial
→
Estimate VRAM
→
Compute max_pack
→
Ask N−1 more trials
→
Recompute with actual configs
→
Trim excess
  1. Ask one trial from Optuna to get the first config (rank, target_mlp, etc.)
  2. Estimate its VRAM via estimate_adapter_vram_mb()
  3. Compute max_pack = $\lfloor 9{,}600 \,/\, \text{per\_adapter\_mb} \rfloor$, capped at remaining trials
  4. Ask additional trials to fill the pack (up to max_pack − 1)
  5. Recompute compute_pack_size() with the actual heterogeneous configs, since different trials may have different ranks and MLP settings
  6. Trim excess: if the recomputed pack size is smaller than the batch, extra trials are returned to Optuna as pruned and re-asked in the next iteration

This means rank-4 batches might pack 25+ adapters, while rank-32 batches pack only a handful. The pack size is never hardcoded—it adapts to whatever Optuna suggests.

Why two rounds of estimation? The first estimate (step 2) determines how many trials to ask. But Optuna's TPE sampler generates different configs each time, so the actual batch may mix rank-4 and rank-32 adapters. The second pass (step 5) catches this and trims the batch to what actually fits.

5. Layer 3: OOM Recovery

Theoretical estimates are approximations. If a batch triggers torch.cuda.OutOfMemoryError despite passing the estimate, the sweep recovers:

  1. The (max_rank, has_mlp) combination that caused the OOM is added to a blacklist
  2. All trials in the failed batch are marked as FAIL and re-asked (the completed count does not increment)
  3. Future batches whose first trial matches a blacklisted combo are forced to pack_size = 1

This creates a learning loop: the sweep starts optimistic (large packs), and if any config exceeds the estimate, it permanently downgrades that config class to sequential execution. The sweep never crashes—it gracefully degrades.

In practice, OOM recovery was never triggered. The 2 GB safety margin and conservative activation overhead (300 MB flat) meant theoretical estimates were always sufficient. Peak VRAM across all 220 trials was 9,744 MB—well under the 13 GB cap.

At low ranks, the theoretical estimate tracks closely (2–6% error). At rank 32 with MLP targeting, the flat 300 MB activation overhead significantly underestimates the actual cost (724 MB empirical vs 364 MB theoretical). However, because the sweep uses a 2 GB safety margin and the base model cost is conservatively estimated, packing decisions remained safe even for the worst-case config.

The three-layer approach worked conservatively. The theoretical estimator was accurate enough to avoid OOMs, the dynamic packer maximized throughput within the budget, and the OOM recovery layer was never needed. The 2 GB safety margin was the key design decision—it absorbed the estimation error at high ranks without sacrificing pack density at low ranks.

8. Limitations

What it does NOT doWhy
No runtime VRAM monitoring during training steps Pack size is decided once per batch; no mid-batch adjustment if memory spikes
No empirical profiling during the sweep itself Profiling was a one-time validation step (Stage 0); the sweep relies on theoretical estimates + safety margin
No per-adapter gradient checkpointing toggle Gradient checkpointing is always on globally; cannot selectively enable for memory-heavy adapters
No cross-batch memory reuse Each batch loads fresh adapters; no warm-starting from previous batch's shared activations
Activation overhead is a flat constant (300 MB) Does not scale with rank or batch size; underestimates at rank-32 + MLP (see profile table)
