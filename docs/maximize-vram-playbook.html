Overview

Bottom line:

Diagnose the actual bottleneck before optimizing. Applying VRAM techniques to an I/O-bound pipeline wastes hours tuning memory flags while the GPU sits idle waiting for data.

This playbook is task-agnostic (LLMs, ASR, vision, diffusion) and framework-agnostic (PyTorch, HF Trainer/TRL, DeepSpeed, FSDP, Lightning). The four phases are not strictly sequential — a quick nvidia-smi tells you which phase to start in.

Identify whether you are VRAM-, CPU-, or I/O-bound.

Tools: nvidia-smi, nvitop, PyTorch Profiler

Common mistake:

Using bf16=True on T4 or V100. These Turing/Volta GPUs have no native bf16 Tensor Cores — it falls back to slow software emulation or errors outright. Always match precision to compute capability (T4/V100 → fp16; A100/H100/RTX30+ → bf16).

Good rule of thumb:

Never reduce batch size just to increase accumulation — that wastes GPU parallelism. batch=8, accum=4 is always better than batch=1, accum=32 for the same effective batch of 32.

Deep dive: what is gradient checkpointing?

Gradient checkpointing (sometimes called activation checkpointing) is the single most impactful memory-saving technique after mixed precision. To understand why, you need to understand where activation memory comes from in the first place.

Why the backward pass needs activations

During the forward pass, every layer L computes an output tensor aL from its input aL−1. To compute gradients during the backward pass, PyTorch's autograd engine needs to know the value of aL−1 — because the gradient of most operations depends on their input. The naive strategy is the simplest one:

Store every activation from every layer during forward; read them back during backward.

This is fast (no recomputation) but memory-hungry. For a transformer with n layers and batch size b at sequence length s, activation memory scales as O(n · b · s · hidden_dim). On a 24-layer model with long sequences, this easily dwarfs the model weights, gradients, and optimizer states combined — which is why activations are listed as "variable, often dominant" in the VRAM breakdown table above.

The checkpointing trick: trade compute for memory

Gradient checkpointing replaces "store everything" with "store a few anchors, recompute the rest on demand." The model is partitioned into segments (commonly one segment per transformer block or group of blocks). During forward:

  • Save only the input to each segment (the "checkpoint"), not the intermediate activations inside it.
  • When the backward pass reaches a segment, re-run its forward starting from the saved checkpoint to reconstruct the missing intermediates just in time, compute the gradients, and then discard them again.

The classic paper by Chen et al. (2016), "Training Deep Nets with Sublinear Memory Cost," showed that if you place checkpoints every √n layers, activation memory drops from O(n) to O(√n) while the forward work roughly doubles (one original pass + one recomputation pass during backward). In practice, framework implementations place a checkpoint at every transformer block, which is close enough to the theoretical optimum for deep networks.

Visual intuition

The memory / compute tradeoff

Strategy Activation Memory Forward Compute Total Step Time
Store all activations (default) O(n) 1× Baseline
Gradient checkpointing (√n segments) O(√n) — ~4–6× less ~2× during backward ~20% slower
Full recomputation (no activations saved) O(1) O(n²) — quadratic blowup Impractically slow

The 20% wall-clock slowdown is almost always worth it, because the memory you free up gets spent on a larger batch size, which recovers much of that throughput back via better GPU utilization. In most real workloads, gradient checkpointing actually makes total training faster at a fixed VRAM budget.

Always pass use_reentrant=False:

The older reentrant variant is deprecated. It breaks with torch.compile, interacts poorly with certain autograd functions (multi-output layers, non-tensor inputs), and can silently disable gradient flow through frozen parameters — which is catastrophic for LoRA, because your adapters stop training.

Gotchas to watch for:

(1) Gradient checkpointing is incompatible with model.eval() / torch.no_grad() blocks inside the checkpointed region — the recomputed forward won't build a graph and gradients vanish. (2) If you use output_hidden_states=True on a HuggingFace model, disable it: the framework forces activation storage to return intermediates, defeating the optimization. (3) BatchNorm's running stats can update twice per step (once in each forward pass) — use GroupNorm, LayerNorm, or RMSNorm instead, which is standard in modern transformers anyway.

Bottom line:

Gradient checkpointing trades ~20% wall-clock time for a 4–6× reduction in activation memory on a typical 24-layer transformer. That freed memory lets you fit larger batches, which recovers most of the throughput loss via GPU parallelism — making it the highest-ROI memory technique after mixed precision, and a near-mandatory default for any non-trivial fine-tuning run.

Deep dive: what is FlashAttention?

FlashAttention (Dao et al., 2022; FlashAttention-2, 2023; FlashAttention-3, 2024) is an exact, IO-aware attention algorithm. "Exact" means the output is numerically equivalent to standard attention — there's no approximation, no accuracy loss, no hyperparameter to tune. The only thing that changes is how the computation is scheduled against the GPU memory hierarchy, and that change alone buys you a massive memory and speed win.

The problem with standard attention

Given queries Q, keys K, and values V of shape (N, d), the textbook formula is:

Attention(Q, K, V) = softmax(Q Kᵀ / √d) V

A naive implementation executes this literally: compute S = Q Kᵀ, store it, compute P = softmax(S), store it, then compute O = P V. Each of those intermediate matrices has shape (N, N) — the full attention matrix. That's where the O(N²) memory cost comes from, and it's why long-context training OOMs so spectacularly: doubling the sequence length quadruples activation memory.

But the hidden cost is worse than memory. On modern GPUs, compute is essentially free compared to memory bandwidth. The naive attention kernel spends most of its time shuttling the (N, N) matrices between HBM (the big, slow GPU DRAM — 40–80 GB at ~2 TB/s) and SRAM (the tiny, fast on-chip cache — ~200 KB per SM at ~20 TB/s). The tensor cores sit idle waiting for HBM reads.

The FlashAttention trick: tile, fuse, never materialize

FlashAttention makes three changes that work only in combination:

  1. Tiling. Split Q, K, V into small blocks that each fit in SRAM. Instead of computing the full (N, N) score matrix at once, compute it one tile at a time.
  2. Kernel fusion. Matmul → softmax → matmul are fused into a single CUDA kernel. Nothing writes back to HBM between those three operations, so the (N, N) intermediates live their whole life in SRAM and vanish.
  3. Online softmax. The softmax normalization constant is computed incrementally across tiles using a numerically-stable running-max trick (Milakov & Gimelshein, 2018). This is the key mathematical insight — it's what lets you softmax a row piecewise instead of needing the whole row in memory at once.

The result: activation memory for attention drops from O(N²) to O(N), and wall-clock speed improves by 2–4× because HBM traffic collapses. Both wins come from the same source — you never materialize the full score matrix anywhere.

Visual intuition

Memory and speed impact

Variant Attention Memory Speed vs. Naive Accuracy
Naive torch.nn.functional.softmax(Q@Kᵀ) @ V O(N²) 1× Reference
PyTorch scaled_dot_product_attention (SDPA) O(N) ~2× Exact
FlashAttention-2 (Ampere+) O(N) ~2–4× Exact
FlashAttention-3 (Hopper only) O(N) ~1.5–2× over FA2 Exact (optional fp8)

For a sequence length of N = 8192, a single attention layer with naive attention needs roughly 8192² × 4 bytes ≈ 256 MB per head per batch element just for the score matrix. Multiply by layers × heads × batch, and you see why long-context training OOMs before you even get to the backward pass. FlashAttention makes that cost essentially vanish.

Compatibility caveats:

(1) FlashAttention-2 requires Ampere or newer (A100, H100, RTX 30/40-series). It will not run on T4 or V100 — fall back to sdpa there, which still gets you the memory-efficient path via xFormers-style kernels. (2) Must be fp16 or bf16; fp32 attention is not supported. (3) Custom attention masks with arbitrary shapes can fall back to the slow path — stick to causal or padding masks when possible. (4) Some head dimensions (e.g., 256) were unsupported in earlier FA2 releases — upgrade to the latest flash-attn wheel if you see dispatch failures.

Bottom line:

FlashAttention is the rare optimization with no tradeoff — it's mathematically exact, faster, and uses less memory. Always enable it (via SDPA at minimum, FlashAttention-2 explicitly when you can) before you start tuning anything else related to long-sequence training. Combined with gradient checkpointing, it's what makes 8k+ context fine-tuning on a single GPU actually feasible.

Phase 3 · Fix the Data Pipeline

If GPU-Util is low, fix these before touching any VRAM optimization. No amount of memory tricks helps when the GPU sits idle 60% of the time.

Setting Starting Value Notes
num_workers 4 Tune up until GPU-Util stops improving
pin_memory True Always — enables fast H2D transfer
persistent_workers True Avoid worker respawn between epochs
prefetch_factor 2 Each worker prefetches 2 batches ahead
drop_last True Avoid variable-size final batch
non_blocking=True on .cuda() Required Overlaps H2D with prior step's compute

What are "workers," and how do they relate to VRAM?

A DataLoader worker is a separate OS process (spawned by PyTorch via multiprocessing) whose sole job is to load the next batch from disk and run all CPU-side preprocessing on it — decoding images, resampling audio, tokenizing text, applying augmentations, collating into tensors. With num_workers=0, all of that runs in the main training process, so every time the GPU finishes a step it has to stop and wait while the CPU fetches and prepares the next batch. With num_workers=N, you have N parallel pipelines preparing batches ahead of time and dropping them into a shared queue that the GPU pulls from.

Important caveat:

Workers do not directly reduce VRAM usage. They live in CPU RAM, not GPU memory. What they do is make sure your GPU can actually use the VRAM headroom you worked so hard to fit into in Phase 2.

The indirect link between workers and VRAM optimization

Phase 2 is about fitting the largest possible batch into VRAM. But a huge batch is only useful if the GPU actually processes it continuously. Without enough workers, the pipeline looks like this:

# num_workers=0 — GPU keeps stalling
[GPU busy]──▒▒▒──[GPU busy]──▒▒▒──[GPU busy]──▒▒▒──
            ↑                 ↑                 ↑
         CPU prep          CPU prep          CPU prep
       (GPU idle, VRAM sitting unused)

# num_workers=4 — GPU stays saturated
[GPU busy][GPU busy][GPU busy][GPU busy][GPU busy]──
     ↑         ↑         ↑         ↑         ↑
  worker 1  worker 2  worker 3  worker 4  worker 1
  (batches prepared in parallel, queued ahead)

More workers help VRAM optimization in three concrete ways:

EffectWhy it matters for VRAM
Keep GPU saturated A batch of 32 filling 90% of VRAM is wasted if the GPU crunches it in 80 ms and then sits idle for 400 ms waiting on the next one. Workers close that gap so the VRAM you reserved is actually doing work every millisecond.
Enable honest batch-size search When binary-searching per_device_batch_size to near-OOM (Phase 2, Technique 4), slow single-worker loading can trick you into thinking training is slow because of the batch size — leading you to shrink it unnecessarily. With enough workers, the VRAM-bound ceiling becomes the true limit you measure against.
Overlap H2D transfer with compute Combined with pin_memory=True and non_blocking=True, workers prepare the next batch in pinned (page-locked) host memory. This lets CUDA copy it to VRAM while the current step's kernels are still running — no idle gap between steps.
Rule of thumb for tuning N:

Start at num_workers=4. Increase until GPU-Util stops improving, then stop. Too many workers wastes CPU RAM (each worker holds a copy of the dataset index and prefetched batches) and can cause contention on shared resources. A reasonable upper bound is one worker per physical CPU core, or slightly below that if you share the machine with other jobs.

Watch out:

If workers themselves are slow (heavy augmentation, decoding a 50 GB audio dataset from a spinning disk every epoch), adding more workers hits diminishing returns quickly. Before cranking N up, consider pre-processing offline (save Arrow/NPY shards) so each worker does less work per batch. That's usually a bigger win than parallelizing expensive work.

Advanced Techniques (2024–2026)

The four phases above cover the classic playbook through late 2023 — mixed precision, gradient checkpointing, QLoRA, FSDP, flash attention. This section covers the next layer of techniques that have landed in PyTorch, torchao, torchtitan, and the research community since then. Reach for them when the four core phases have been applied and you still need more — or when you are starting a new project on modern hardware (H100/H200/Blackwell) and want to skip straight to the current state of the art.

Advanced Parallelism (beyond FSDP/ZeRO-3)

FSDP1 and DeepSpeed ZeRO are the "legacy" multi-GPU recipe. The PyTorch-native path since 2024 is built on DTensor — a distributed-tensor abstraction that lets FSDP2, tensor parallelism, sequence parallelism, context parallelism, and pipeline parallelism compose cleanly on a single DeviceMesh. This is the foundation of torchtitan, the reference training framework Meta/PyTorch publishes for scale-out training.

FSDP2 · per-parameter sharding via fully_shard

The successor to FSDP1. Instead of flattening and concatenating groups of tensors before chunking (FSDP1), FSDP2 chunks each parameter individually on dim-0 across the DP workers. Every parameter becomes its own DTensor. Result: ~7% lower peak GPU memory than FSDP1 at the same loss curve on Llama-2-7B, ~1.5% MFU gain on Llama-7B / 8×H100 in torchtitan, and deterministic memory usage without FSDP1's limit_all_gathers=True blocking pattern.

# Per-layer sharding — FSDP2 recipe
from torch.distributed.fsdp import fully_shard

for layer in model.layers:
    fully_shard(layer)
fully_shard(model)  # shard top-level too

With HuggingFace Accelerate, set fsdp_version: 2 in the FSDP plugin config (available in accelerate >= 1.0).

Why it matters:

Every new multi-GPU training project should start on FSDP2. It's a drop-in ~7% memory win and the prerequisite for every other modern technique in this section (TP, SP, CP, PP, fp8 all-gather). Treat FSDP1 as legacy.

Tensor Parallelism via torch.distributed.tensor.parallel

Shards individual matmul ops (attention heads, MLP hidden dim) across GPUs so each GPU holds a slice of the weights and does a slice of the compute. Uses ColwiseParallel / RowwiseParallel to partition Linear-layer weights along columns or rows; an all-reduce combines results. Unlike FSDP, which temporarily all-gathers full weights for each op, TP keeps weights sharded during compute — so peak parameter memory per layer is permanently lower.

In torchtitan 3D runs: 12.59% speedup on Llama 3.1 70B / 256 GPUs (2D), 30% on Llama 3.1 405B / 512 GPUs (3D) over optimized baselines.

from torch.distributed.tensor.parallel import (
    parallelize_module, ColwiseParallel, RowwiseParallel)
from torch.distributed.device_mesh import init_device_mesh

mesh = init_device_mesh("cuda", (dp_size, tp_size),
                        mesh_dim_names=("dp", "tp"))
tp_mesh = mesh["tp"]

for layer in model.layers:
    parallelize_module(layer, tp_mesh, {
        "attention.wq": ColwiseParallel(),
        "attention.wk": ColwiseParallel(),
        "attention.wv": ColwiseParallel(),
        "attention.wo": RowwiseParallel(),
        "feed_forward.w1": ColwiseParallel(),
        "feed_forward.w2": RowwiseParallel(),
        "feed_forward.w3": ColwiseParallel(),
    })
TP requires NVLink.

Tensor parallelism all-reduces on every matmul. On NVLink-connected GPUs (A100 SXM, H100 SXM, DGX systems) this is fast; on PCIe-only boxes it is a throughput disaster. Never enable TP without a high-bandwidth intra-node interconnect.

Sequence Parallelism (paired with TP)

In pure TP, attention and MLP weights are sharded, but LayerNorm and Dropout operate on the full activation tensor — which gets replicated across TP ranks, wasting memory. Sequence Parallelism splits these "replicated" regions along the sequence dim so each rank only holds seq_len / tp_size tokens of the norm/dropout activations. All-gather and reduce-scatter handle the transitions at TP region boundaries.

Impact (Korthikanti et al. 2022, Reducing Activation Recomputation): roughly half the activation memory of plain TP for layernorms/dropouts. Combined with selective activation recomputation (below), 5× total activation reduction while still running >90% of compute without recomputation.

from torch.distributed.tensor.parallel import SequenceParallel

parallelize_module(layer, tp_mesh, {
    "attention_norm": SequenceParallel(),
    "ffn_norm":       SequenceParallel(),
    # ... plus TP on the linear layers as above
})

SP is only meaningful on top of TP — on its own it does nothing for DP-only training.

Context Parallelism · Ring Attention

For sequences so long that per-token activation × seq_len doesn't fit on one GPU even with FlashAttention, Context Parallelism shards the sequence dimension across GPUs. Each GPU holds only seq_len / CP_degree tokens. Each GPU computes local attention against its Q shard, then receives remote KV shards in a ring pattern, updating the running softmax — the same tile-based online softmax FlashAttention already uses inside one GPU, just now across GPUs.

Scale pointMeasured result
Llama3-8B · 32× H100 · torchtitan Enables sequence lengths up to 1M tokens
DeepSpeed Ulysses variant 2.5× speedup at 4× longer sequences vs the prior SP baseline 
from torch.distributed.tensor.experimental import context_parallel

cp_mesh = mesh["cp"]
with context_parallel(cp_mesh, buffers=[q, k, v], buffer_seq_dims=[2, 2, 2]):
    out = F.scaled_dot_product_attention(q, k, v, is_causal=True)

Naive ring attention is load-unbalanced for causal masks (early ranks idle). "Pass-KV" variants fix this — use torchtitan's default schedule. Only worth the complexity past ~32k tokens.

Pipeline Parallelism · Zero-Bubble 1F1B

Splits the model by layers across GPUs (stage 1 = layers 0–7, stage 2 = 8–15, …). Microbatches flow through the pipeline. The naive "GPipe" schedule has bubbles where GPUs idle; 1F1B interleaves one forward and one backward per microbatch; Zero-Bubble splits backward into "grad-input" and "grad-weight" so the optimizer step bypasses sync barriers.

Reported (Qi et al., ICLR 2024): up to 23% throughput gain over 1F1B at the same memory budget, 31% with relaxed memory.

from torch.distributed.pipelining import Schedule1F1B, pipeline, SplitPoint

pipe = pipeline(model, mb_args=sample_input, split_spec={
    "layers.8":  SplitPoint.BEGINNING,
    "layers.16": SplitPoint.BEGINNING,
    "layers.24": SplitPoint.BEGINNING,
})
stage = pipe.build_stage(stage_idx, device, pp_group)
schedule = Schedule1F1B(stage, n_microbatches=8, loss_fn=loss_fn)
# Also: ScheduleInterleaved1F1B, ScheduleZBVZeroBubble

Requires n_microbatches ≥ pp_size × 2 or the bubble reappears. Reach for PP only when TP+FSDP has hit its ceiling on a single node, typically cross-node training with high inter-node latency where TP's all-reduce cost is prohibitive.

HSDP · Hybrid Sharded Data Parallel

Middle ground for multi-node training where inter-node bandwidth is slow. Uses a 2D device mesh: within a node, full FSDP (all-gather weights for each op — cheap over NVLink at 600–900 GB/s); across nodes, DDP (all-reduce gradients once per step — fits in slow Ethernet budget at 25–50 GB/s).

from torch.distributed.device_mesh import init_device_mesh
from torch.distributed.fsdp import fully_shard

mesh = init_device_mesh("cuda", (num_nodes, gpus_per_node),
                        mesh_dim_names=("replicate", "shard"))

for layer in model.layers:
    fully_shard(layer, mesh=mesh)
fully_shard(model, mesh=mesh)
When to pick HSDP over full FSDP:

Use HSDP when inter-node bandwidth is below ~100 Gbps (most non-Infiniband clusters). On Infiniband or NVLink-Switch clusters, pure FSDP usually wins because you actually want full sharding across all GPUs for peak memory reduction. Check ibstat or nvidia-smi topo -m before deciding.

Activation Memory — Deeper Cuts

The original Phase 2 technique — full gradient checkpointing — trades ~20% wall-clock slowdown for 4–6× activation reduction. These techniques refine that tradeoff in both directions: recompute less (selective) or recompute not-at-all and instead move activations off GPU (offload).

Selective Activation Recomputation

Instead of recomputing all checkpointed activations on backward, profile which ops have a bad operation-per-byte ratio and recompute only those. Attention's softmax/dropout/QK-dot have memory cost O(seq²) but modest compute → cheap to recompute. The linear projections are expensive to recompute but small → better to keep them. Selective recomputation checkpoints only the inputs to attention blocks.

StrategyActivation memRecompute overheadThroughput (530B model)
Full recomputation (Phase 2 default) O(√n) ~20% Baseline
Selective recomputation O(√n) — nearly same ~2% +29%, 54.2% MFU 
# torchtitan exposes two modes:
#   selective_op    — checkpoint only matmul ops (skip elementwise/softmax)
#   selective_layer — checkpoint every Nth transformer layer
# CLI: --activation_checkpoint.mode=selective_op

from torch.distributed.algorithms._checkpoint.checkpoint_wrapper import (
    checkpoint_wrapper, CheckpointImpl, apply_activation_checkpointing)

# NO_REENTRANT is mandatory for selective

Via Megatron-LM: --recompute-granularity selective. Source: Korthikanti et al., Reducing Activation Recomputation in Large Transformer Models (arXiv 2205.05198).

Activation CPU Offload

Pairs with gradient checkpointing: after the forward pass stashes a checkpoint's input, that input is copied to CPU pinned memory on a dedicated CUDA stream so GPU compute continues. Before backward reaches the segment, the activation is prefetched back asynchronously. Combined with activation checkpointing + parameter offload, FSDP can train models up to 20× larger than plain DDP on the same hardware.

from torch.distributed.algorithms._checkpoint.checkpoint_wrapper import (
    checkpoint_wrapper, CheckpointImpl)

def wrap(module):
    return checkpoint_wrapper(
        module,
        checkpoint_impl=CheckpointImpl.NO_REENTRANT,
        offload_to_cpu=True,   # ← the key flag
    )
# Apply to transformer layers before FSDP wrap

Axolotl exposes this as activation_offloading: true. Throughput cost is modest with async overlap on (roughly 1.3–1.5× slower typical); without overlap it's much worse. Does not stack with recomputation on the same layer — pick one per layer.

Attention Kernels — FA3 & FlexAttention

The existing Phase 2 section covers SDPA and FlashAttention-2. Two newer kernels push further.

FlexAttention · programmable masks without Triton

torch.nn.attention.flex_attention takes score_mod and mask_mod callables plus Q/K/V tensors. Under torch.compile, PyTorch generates a block-sparse FlashAttention kernel that skips fully-masked blocks via a BlockMask. You write the mask in plain Python; you get FlashAttention-quality performance. Enables sliding-window attention, ALiBi, document-packing masks, tanh soft-cap, prefix-LM — each as a one-line change.

Impact: ~2.4× faster training than manual-mask attention in torchtune. Fully-masked blocks are skipped — for sliding-window attention with window << seq_len, memory and compute scale by the mask sparsity ratio.

from torch.nn.attention.flex_attention import flex_attention, create_block_mask

def sliding_window_mask(b, h, q_idx, kv_idx):
    return (q_idx - kv_idx).abs() <= 512

block_mask = create_block_mask(sliding_window_mask, B=None, H=None,
                                Q_LEN=seq_len, KV_LEN=seq_len)

out = flex_attention(q, k, v, block_mask=block_mask)
Compile is mandatory.

FlexAttention's eager path is slow — the whole point is that torch.compile fuses your Python callable into a block-sparse kernel. Always wrap the model (or at least the attention call site) in torch.compile. Requires PyTorch ≥ 2.5.

fp8 Training & MX Formats

Phase 2 stops at bf16. Modern Hopper and Blackwell GPUs have hardware support for 8-bit floats, and the PyTorch-native path (via torchao) has matured to the point where fp8 is a serious production option.

torchao float8 + FSDP2 + torch.compile

torchao.float8.convert_to_float8_training recursively replaces nn.Linear with Float8Linear. Matmul weights, inputs, and grad outputs are quantized to E4M3 / E5M2 per step with rowwise scaling; attention and layernorm stay in bf16. Pairs with FSDP2's fp8 all-gather to also speed cross-GPU communication.

    model,
    config=config,

# torch.compile is MANDATORY for performance:

Three non-negotiables:

(1) Exclude lm_head and embeddings from fp8 — huge accuracy drop otherwise. (2) torch.compile is mandatory — without it, fp8 actually regresses vs bf16. (3) Hopper or newer — A100 has no fp8 Tensor Cores; using fp8 there falls back to simulation. Also: QLoRA is conceptually incompatible with fp8 training (QLoRA already quantizes).

MXFP8 / MX Formats (Blackwell)

Block-scaled FP8 / FP6 / FP4 formats native to Blackwell (B100, B200, RTX 50-series). A shared 8-bit exponent per 32-element block — finer-grained than Hopper's tensor-level or rowwise scaling, giving better numerics and simpler recipes. Lets you drop to MXFP6 or MXFP4 for inference and potentially training.

Torchtitan has an evolving MXFP8 recipe in docs/mxfp8.md. Hardware-locked to Blackwell; on Hopper, use the torchao fp8 recipe above instead.

torchao CPUOffloadOptimizer — modern paged_adamw replacement

PyTorch-native, torch.compile-friendly replacement for bitsandbytes' paged_adamw_8bit. Stashes optimizer state and gradients on CPU pinned memory; runs the optimizer step on CPU. Benchmark: 39.1 GB peak vs 51.6 GB fused AdamW vs 39.3 GB bitsandbytes 8-bit — matches bitsandbytes in memory while being pure PyTorch. Expect ~1.5–2× slower per step than an in-GPU optimizer.

from torchao.prototype.low_bit_optim import CPUOffloadOptimizer

optimizer = CPUOffloadOptimizer(
    model.parameters(),
    optimizer_class=torch.optim.AdamW,
    fused=True,           # CPU-side fused AdamW
    offload_gradients=True,
)

torchao low-bit AdamW (8-bit, 4-bit)

Pure-PyTorch + torch.compile quantized optimizer state. Benchmarks match bitsandbytes at 8-bit; 4-bit halves state again, unlocking something bitsandbytes doesn't ship. Easier install, no bnb version drama, works with FSDP.

from torchao.prototype.low_bit_optim import AdamW8bit, AdamW4bit
optimizer = AdamW4bit(model.parameters(), lr=1e-4)  # ~8× savings vs fp32 state

Stochastic rounding + Kahan summation (full-FT only)

When updating bf16 weights in-place (no fp32 master copy), nearest-rounding kills small updates. Stochastic rounding makes the update unbiased in expectation; Kahan summation carries the rounding error in a separate 16-bit accumulator. Net: drop the 4P-bytes fp32 master copy → gain 4P, pay 2P back for Kahan accumulator → save 2P bytes. Only worth it for full-param training ≥ 1B. The optimi library implements this cleanly.

from optimi import AdamW
optimizer = AdamW(model.parameters(), lr=1e-4, kahan_summation=True)

PEFT Variants — beyond LoRA / QLoRA / AdaLoRA

Modern alternatives to vanilla LoRA. Most of these are quality upgrades, not memory wins — it's important to know which are which. The table below makes that explicit.

Compiler & CUDA Graphs

torch.compile · three modes to know

The existing Phase 2 playbook mentions torch.compile only as a footnote. It deserves its own section with the three-mode distinction:

Mode Use case Speedup Memory Compile time
default General training 1.2–1.5× base fast
reduce-overhead Small-batch / launch-bound training 1.3–1.8× slightly more (CUDA graph workspace) moderate
max-autotune Inference / very long training runs 1.5–2.0×+ base slow (minutes)

# Or via HF Trainer
training_args.torch_compile_mode = "reduce-overhead"
CUDA-graph gotchas:

CUDA graph capture fails on dynamic shapes. Pin sequence lengths (pad_to_multiple_of + fixed max seq, or frame-budget batching that pads to a max) before enabling reduce-overhead. Shape changes trigger recompile storms that destroy throughput. Also: bitsandbytes 4-bit matmul under torch.compile is still version-sensitive — verify before shipping.

Thunder (Lightning AI) — niche alternative fusion backend

thunder.jit(model) traces a Python program and dispatches ops across multiple executors (nvFuser, torch.compile, cuDNN, TransformerEngine). Reports ~40% speedup vs eager, up to 3.42× faster text generation, 1.69× faster fwd+bwd on DeepSeek-R1-Distill-Llama-1.5B. Interops with torch.compile (can be stacked). Niche; most users can stay on torch.compile. Include as "worth trying if curious."

Allocator & System-Level Tuning

Full PYTORCH_CUDA_ALLOC_CONF cheat sheet

The existing playbook mentions expandable_segments:True only. Several other options address different failure modes:

Option Effect When to use
expandable_segments:True Reduces fragmentation by using growable VMM segments OOM after many steps (the default fix)
max_split_size_mb:N Native allocator won't split blocks larger than N MB Large contiguous allocations failing despite reported free memory
garbage_collection_threshold:0.8 Proactively frees old unused blocks past 80% usage Slow memory creep without a clear leak
backend:cudaMallocAsync Switches to CUDA's async allocator Experimental — different fragmentation profile; ignores garbage_collection_threshold
roundup_power2_divisions:[N] Round allocation sizes to power-of-2 / N Allocator-heavy workloads with lots of odd sizes
pinned_use_background_threads:True Async pinned CPU-mem allocation Large pinned allocations blocking the main thread

Recommended default for training:

export PYTORCH_CUDA_ALLOC_CONF="expandable_segments:True,garbage_collection_threshold:0.8"

Gotcha: garbage_collection_threshold is ignored when backend:cudaMallocAsync is set. Pick one backend.

NCCL buffer tuning

Per-GPU NCCL comm buffers consume VRAM. The default NCCL_BUFFSIZE is 4 MiB; cutting it to 2 MiB saves a few hundred MB per rank when multi-GPU training is VRAM-tight. Benchmark before committing — too small can tank comms throughput.

export NCCL_BUFFSIZE=2097152   # 2 MiB, half of default

DeepSpeed ZeRO-Infinity · NVMe offload (last-resort tier)

Extension of ZeRO-3 that offloads parameters, gradients, and optimizer states to NVMe SSDs instead of just CPU RAM. Async prefetching overlaps NVMe→CPU→GPU movement with compute. Enables trillion-parameter training on a single DGX-2 node; 30+ trillion on 32 nodes. Bandwidth-limited by NVMe, so per-step is much slower than in-GPU training.

Reach for this only when (params + grads + optim) exceeds total CPU RAM — rare outside pretraining research. For 99% of users, FSDP2 + CPU offload is sufficient. DeepSpeed ecosystem only; FSDP2 has no NVMe-tier offload as of early 2026.

Advanced Diagnostics · nsys

PyTorch Profiler (Phase 1 Step 4) is sufficient for most bottleneck analysis. When it isn't — GPU-Util oscillating but the profiler breakdown shows "nothing obvious" — NVIDIA Nsight Systems (nsys) captures CPU, GPU, Python, CUDA, and NCCL activity on one timeline, revealing whether the gap is launch overhead, Python GIL, H2D transfer, or NCCL comm.

Axolotl defaults

Axolotl's default configuration contains several subtleties the core playbook doesn't call out:

  • sample_packing: true paired with pad_to_sequence_len: true — Axolotl intentionally pads packed sequences to a fixed length. Reason: varying packed lengths cause VRAM spikes even if each batch is smaller than fully padded. This contradicts the usual "dynamic padding wins" advice — but packing's whole point is that everything is already the same length.
  • flash_attention: true automatically routes packed sequences through the FA2 varlen API.

  • eval_sample_packing: false — packing during eval is often broken; disable explicitly.
  • sequence_parallel_degree: N — Ulysses-style SP, currently in beta.

The canonical modern recipe for scale-out PyTorch training, maintained by Meta/PyTorch. Combines everything in this section:

Pre-flight Checklist

Run through this before launching any expensive training run. Click to check off — state persists in your browser.

Core checklist (applies to every run):

  • Ran nvidia-smi — no other job using this GPU
  • GPU memory usage below 80% before launch
  • Mixed precision matches GPU architecture (fp16 on T4/V100, bf16 on Ampere+)
  • Gradient checkpointing enabled with use_reentrant=False
  • SDPA or FlashAttention-2 enabled for long sequences
  • Binary-searched per_device_batch_size to near-OOM
  • Gradient accumulation set to reach target effective batch
  • DataLoader num_workers tuned against GPU-Util
  • pin_memory + non_blocking transfers enabled
  • Peak VRAM is 80–90% of total (Phase 4 check)
  • GPU-Util steady > 90%, no oscillation
  • W&B logging + babysit-training monitoring set up
  • Checkpoint recovery tested (kill + resume works)

Advanced checklist (if you're using the 2024–2026 techniques):

  • Multi-GPU runs use FSDP2 (fully_shard), not legacy FSDP1
  • Gradient checkpointing uses selective recomputation where possible
  • PYTORCH_CUDA_ALLOC_CONF exports expandable_segments:True,garbage_collection_threshold:0.8
  • torch.compile mode chosen deliberately (default / reduce-overhead / max-autotune)
  • If using fp8: lm_head and embeddings are excluded, torch.compile is on, hardware is Hopper+

FAQ

Should I use fp16 or bf16?

Bf16 if your GPU is Ampere or newer (A100, H100, RTX 30/40 series). Fp16 on older (T4, V100). Bf16 has the same dynamic range as fp32 so it avoids the gradient scaling headaches of fp16 — but Turing/Volta don't have the Tensor Cores for it and will silently fall back to software emulation.

Why is my GPU at 100% utilization but training is slow?

You are compute-bound, not VRAM-bound. Your GPU kernels are the bottleneck. Try torch.compile, FlashAttention-2, or fused kernels (Liger/Unsloth). If none of those help, the model is simply compute-heavy for this hardware.

My peak VRAM is 60% — should I increase batch size?

Yes, until you hit 80–90%. Binary search: double the batch until OOM, then back off. The exception is when your effective batch is already above what the task needs (e.g., using > 32 effective for a small LR-sensitive task).

When should I reach for QLoRA vs regular LoRA?

Regular LoRA first — it's simpler and has no quantization error. Reach for QLoRA (NF4) when LoRA alone won't fit, typically for 7B+ models on consumer GPUs. The 4-bit base weights cut frozen memory by ~4×, at the cost of slightly slower forward passes.

Do I need FSDP or DeepSpeed?

Almost always the last resort. For single-GPU LoRA on models under 7B, every technique above is simpler and enough. Multi-GPU full fine-tuning is where sharding actually pays off — prefer FSDP over DeepSpeed for new PyTorch projects since it's native and better integrated with Accelerate/Trainer.

FSDP1 or FSDP2 — which one should I use?

FSDP2. Treat FSDP1 as legacy for new code. FSDP2's per-parameter fully_shard API gives ~7% lower peak memory on Llama-2-7B at the same loss curve, ~1.5% MFU gain on Llama-7B / 8×H100, and much cleaner composition with TP / SP / CP / PP via a DeviceMesh. It's also the prerequisite for fp8 all-gather and modern torchtitan recipes. The only reason to stay on FSDP1 is backward compatibility with existing sharded checkpoints — and migrating those is a one-time job. See the FSDP2 section.

Does fp8 training actually help me?

Only on Hopper (H100/H200) or Blackwell, and only for models ≥ ~1B parameters, and only if you're willing to run torch.compile. Under those conditions, torchao float8 + FSDP2 can give up to 50% throughput over bf16. If any of those conditions fails — A100, small model, no compile — bf16 is still the right answer. Always exclude lm_head and embeddings from fp8 conversion, or accuracy collapses. See torchao fp8.

Selective vs full activation recomputation — which should I pick?

Selective if your framework supports it (torchtitan, Megatron, or a manual FSDP2 wrap). It keeps nearly all of full checkpointing's memory win but at ~2% compute overhead instead of ~20%. The trick is that it only recomputes the cheap-to-recompute, expensive-in-memory ops — primarily attention softmax and dropout — and keeps the expensive matmuls. For HF Trainer without patching, you're stuck on full checkpointing. See selective recomputation.

My LoRA isn't good enough but full fine-tuning doesn't fit — now what?

In order of effort: (1) Try rsLoRA with a higher rank (one-flag fix to unlock r=64/128). (2) Try PiSSA or LoftQ init if you're on QLoRA — they absorb quantization error. (3) Switch to a full-param technique with small optimizer state: APOLLO (MLSys'25, best throughput), Q-GaLore (consumer GPU pretraining), or BAdam (block coordinate descent — full Llama-3-8B on a single 3090). (4) As a last resort, add FSDP2 + CPU offload or a second GPU. Do not default to DoRA — it increases training VRAM, it doesn't decrease it.

My ASR pipeline uses torchaudio.load — is that still fine?

For now, yes — but plan the migration. torchaudio.load is deprecated and will be removed in torchaudio 2.9. The replacement is torchcodec, which is also the PyTorch-native path for video decoding and has a CUDA decode path for compressed formats. The mechanical drop-in is torchaudio.load_with_torchcodec(...), which calls torchcodec under the hood; the cleaner long-term path is using torchcodec.decoders.AudioDecoder directly.
