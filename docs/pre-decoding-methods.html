WebDataset, FLAC, raw PCM, log-mels, and quantization — when each one wins

Why this document exists.

Phase 03-02's VRAM tuning on the A6000 capped mean GPU utilization at 41–42%, well below the 55% launch floor for Plan 03-03. Diagnosis: the dataloader is I/O-bound on MP3 decode plus per-file seeks across 1.16M small files on overlayfs with 4 CPU cores. Pre-decoding is the root-cause fix. This page is the decision reference for how to pre-decode. See gpu-occupancy-handling.html for the diagnosis and webdataset-explained.html for a deeper WebDataset primer.

1. The two-part problem

"Pre-decoding" actually bundles two independent problems. Conflating them is the most common planning mistake — you can fix one without the other, and each fix has its own cost.

Problem A

The decode cost

MP3 decompression on CPU is the dominant per-sample cost. Single-threaded per file, competing with every other dataloader operation. On a 4-core host, decoder threads starve the training loop.

Fixed by: picking a storage format that's cheaper to decode (FLAC, PCM, log-mels).

Problem B

The file-layout cost

2.2M tiny files in a flat directory means millions of open() / seek() / read() syscalls per epoch. Overlayfs amplifies the cost. Page cache fragments. Sequential I/O on a tar shard is 10–100× faster than random I/O across small files.

Fixed by: packaging into large sequential shards (WebDataset, FFCV, tar).

The common mistake: assuming "switch to WebDataset" solves everything. WebDataset fixes only Problem B (file layout). The storage format inside the shards determines whether Problem A (decode cost) is also fixed. A WebDataset tar full of MP3 files has the same decode bottleneck as a flat MP3 directory.

3. Sample-rate options

Independent of storage format: what target sample rate should the pre-decode land at? CommonVoice-25 sources are heterogeneous (63% @ 48 kHz, 37% @ 32 kHz, trace @ 44.1 kHz — see project-overview.html §2).

Source → Target Train-split size Quality impact Verdict
Native (mixed 32/48 kHz) → keep native ~400 GB none wasteful
Any → 16 kHz mono int16 ~192 GB none for Qwen3-ASR (matches pre-train) yes
Any → 8 kHz mono int16 ~96 GB +10–30% relative WER, demographically uneven no (fairness)
Any → 16 kHz mono + log-mel ~96 GB none yes (frozen cfg)
Why 8 kHz is disqualified for a fairness study.

Bandwidth reduction to 8 kHz cuts everything above 4 kHz: sibilants (/s/, /ʃ/, /f/, /θ/), stop bursts, and high formants. These features degrade unequally across demographics — female voices, children, and speakers of accents with high-frequency consonant distinctions lose more information. Introducing 8 kHz would confound the fairness metric directly: any measured WER gap could come from the codec, not the model. Unacceptable for a study whose entire point is to measure those gaps.

4. Packaging layer — why WebDataset

Independent of storage format: how are the pre-decoded samples arranged on disk? This is Problem B from §1.

Layout A

Individual files

One file per clip. 2.2M small files, random I/O, millions of open() calls per epoch. Overlayfs amplifies the syscall overhead. Page cache fragments because each file is tiny. Workers contend for inode lookups.

Layout B

WebDataset tar shards

~1 GB per shard, ~1000–5000 clips each. Sequential I/O within a shard, one open() per shard, page cache works, workers stream one shard at a time. Standard POSIX tar format — portable, inspectable, shareable.

WebDataset doesWebDataset does not
Solve the file-layout cost (Problem B) Change storage size
Give shard-level parallelism across workers Change decode cost
Improve page-cache behavior via sequential reads Eliminate the need to pick a storage format
Standard format across PyTorch, JAX, TensorFlow Make the dataset smaller or faster to decode on its own

6. Upfront cost of each option

What you pay before the first training step, regardless of how long the subsequent training run is.

Option Upfront CPU Upfront disk Code changes Reversibility
int16 PCM + WebDataset ~4–6 h (1.16M clips, 4 cores) ~192 GB medium: shard writer + loader fully reversible
FLAC lossless + WebDataset ~4–6 h (decode + re-encode) ~100–115 GB medium fully reversible
Log-mel fp16 + WebDataset ~5–8 h (decode + filterbank) ~96 GB medium-hard: match processor cfg exactly; bypass model front-end reversible but must freeze feature cfg
μ-law + WebDataset ~4–6 h ~96 GB medium reversible but fewer benefits than int16

CPU estimates assume ~10–15 ms MP3 decode per clip on the A6000 host's 4 cores. Your mileage will vary with decoder settings and clip length distribution.

7. Why we rejected int8 audio and 8 kHz

Rejected

int8 linear audio

Dynamic range. int16 gives ~96 dB; int8 linear gives ~48 dB — the span between a whisper and a normal voice. Quiet consonants land in the noise floor, loud stops clip.

No memory win. The model expects float32 at load time, so you dequantize back to float anyway. int8 saves disk but adds noise.

Fairness confound. Quantization noise hits quiet consonants and aspirated stops unevenly across accents and genders — you'd be baking a demographic artifact into the dataset.

Rejected

8 kHz sample rate

Bandwidth loss. Nyquist = 4 kHz. Sibilants (/s/, /ʃ/, /f/, /θ/), stop bursts, and high formants all live above 4 kHz.

Measurable WER cost. Published 16 kHz → 8 kHz degradation is +10–30% relative, and it is demographically uneven.

Distribution shift. Qwen3-ASR was pre-trained on 16 kHz input. Feeding it 8 kHz (even upsampled back) means zeroed high-frequency bins the encoder has never seen during pre-training.

Bottom line: neither int8 audio nor 8 kHz is a valid optimization for this project. Both save disk but trade quality for size in ways that directly confound the fairness measurement that is the entire point of the study.

8. Common misconceptions

"WebDataset reduces storage."

No. WebDataset is a packaging format, not a compression format. Storage size depends on what you put inside the shards. A WebDataset of MP3s is still ~84 GB; a WebDataset of int16 PCM is still ~192 GB. The tar overhead is negligible (< 1%). WebDataset speeds up I/O layout (Problem B), not storage size.

"Log-mels are the same as mel-spectrograms."

Close but not identical. A mel-spectrogram is the output of a mel filterbank applied to a STFT magnitude. A log-mel spectrogram is that spectrogram with a log (or log-plus-small-epsilon) applied. Qwen3-ASR expects log-mels with a specific n_mels, sample_rate, hop_length, win_length, window, power, and whether the log is natural or base-10. If you pre-compute, you must match the processor config exactly, or training will see a feature distribution the model has never seen.

"FLAC is like MP3 but bigger."

Very different. FLAC is lossless — it bit-exact reconstructs the original PCM samples. MP3 is a lossy perceptual codec that throws away ~90% of the data. FLAC is ~2× bigger than MP3 but ~2× smaller than int16 PCM, and decodes 10–20× faster than MP3 because its math is simpler. For an I/O-bound ASR dataloader, FLAC is often the sweet spot between storage and decode cost.

"Pre-decoded shards make the dataset immutable."

Yes, and for the most part that's a feature. Shards are reproducible, shareable across hosts, and hash-identifiable. The downside: if you change the feature-extraction config (e.g., switch n_mels from 80 to 128), you have to re-dump the shards from source. Budget that cost into any decision to pre-compute features rather than raw audio.

"OPUS at 24 kbps is the sweet spot — smaller than FLAC, better than MP3."

Smaller, yes. Better quality per bit, yes. But OPUS is still a perceptual lossy codec, and chaining it with MP3-sourced audio compounds the perceptual loss (the first lossy pass has already discarded information, so the second pass is operating on an already-degraded signal). For a fairness study you cannot afford unknown perceptual artifacts that might favor some accents over others. OPUS is fine for general ML but not here.
