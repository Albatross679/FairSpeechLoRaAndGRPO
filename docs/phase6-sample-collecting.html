Big Picture

Phase 5 produces a fairness-aware GRPO checkpoint, but RL inference is expensive. Phase 6 distills that policy into a standard SFT model by generating many transcription candidates from the frozen GRPO checkpoint, filtering them for joint accuracy + fairness, then fine-tuning a fresh LoRA adapter on the curated set. The result is a model with fairness baked into its training data, deployable without RL at inference time.

The core move: candidate generation happens after GRPO training finishes — from the frozen best checkpoint, not from intermediate policies during training. This matches the DeepSeek-R1 / Qwen3 / Llama alternating-training paradigm.
Source model
Best GRPO adapter from Phase 5 (selected by composite WER + fairness metric)
Target model
Fresh LoRA on the base model — not a continuation of the GRPO adapter
Corpus
~5,000 utterance representative subset of the Phase 3 training manifest
Candidates per utterance
N = 10 (RLHF book recommends 10–30; ASR has less diversity headroom)
Sampling
do_sample=True, temperature=0.8 — diversity over individual quality
Output artifact
Filtered CSV (~3–5K rows) → fresh LoRA adapter trained for 1 epoch

Pipeline Overview

Three independent scripts connected by CSV intermediates. The separation is load-bearing: Stage 1 is the expensive step, Stages 2 and 3 are cheap and iterable.

Stage 1

Generate Candidates

Load frozen GRPO checkpoint. For each utterance in the 5K subset, sample 10 transcriptions at temperature 0.8. Compute WER per candidate. Write one row per (utterance, candidate).

In: GRPO adapter + manifest
Out: 50K rows CSV
→all_candidates.csv
Stage 2

Filter

Select best candidate per utterance (lowest WER). Apply WER threshold. Check per-group WER; rebalance demographic groups by relaxing or tightening per-group thresholds.

In: candidates CSV
Out: ~3–5K rows CSV
→filtered_dataset.csv
Stage 3

SFT

Load base model. Apply a fresh LoRA adapter with Phase 3's locked hyperparameters. Train 1 epoch on the filtered dataset. Reuses train_standard_lora.py infrastructure wholesale.

In: filtered CSV
Out: Fresh LoRA adapter

Stage 1 — Generate Candidates

The single expensive step in the pipeline. Each utterance is run through the frozen GRPO checkpoint ten times with stochastic sampling, then WER is computed inline for every candidate so the filtering stage has nothing to recompute.

Generation Knobs

KnobValueWhy
N candidates 10 per utterance RLHF book: 10–30 is the standard range. ASR has less diversity, so lower end.
Decoding mode do_sample=True Greedy/beam produce high-quality but nearly identical candidates — nothing to filter between.
Temperature 0.8 Diversity knob. Raise to 1.0–1.2 if the pilot shows near-duplicate candidates.
Inner batch num_return_sequences=2 T4's 15 GB KV cache OOMs at 10 parallel sequences. Loop 5 times per utterance.
Audio batch 1 utterance at a time Keeps KV cache arithmetic simple. Generation is the bottleneck, not audio encoding.
GC hygiene torch.cuda.empty_cache() per inner loop Prevents drift-OOM after hundreds of utterances.

Why write WER into the candidate CSV? It front-loads the one computation that depends on Stage 1's actual text outputs. Stage 2 can then iterate on thresholds without ever re-decoding audio — the filter becomes a pandas operation, not a model call.

Stage 2 — Filter

Two rules that matter. Everything else is threshold tuning.

Rule 1: Per-Utterance Best, Not Global Top-K

What to do
  • Group candidates by utterance_id
  • Keep the single lowest-WER row per group
  • Result: one row per utterance (~5K rows)
Why not global top-K
  • Short/clear audio has low WER naturally
  • Easy utterances are overrepresented in majority groups
  • Global top-K → filtered set is less fair than original

Rule 2: Fairness-Aware Two-Pass Threshold

PassActionOutput
Pass 1 — WER ceiling Drop utterances whose best-candidate WER exceeds global threshold (e.g. 0.30) High-quality subset, possibly demographically skewed
Pass 2 — per-group gap Compute WER per demographic group. If gap > tolerance, relax threshold for underperforming groups and tighten for overperforming ones Fairness-balanced SFT set
Pass 3 — coverage check Verify each group has at least min_group_size utterances; if not, keep the N lowest-WER candidates regardless of threshold Guaranteed demographic coverage
Target acceptance rate: 60–80%. Above 95% means the filter is doing nothing; below 40% means thresholds are starving the SFT set.

Stage 3 — SFT on Filtered Data

The cheapest surprise in Phase 6: Stage 3 is almost entirely reused Phase 3 code. Swap the CSV path and reduce epochs; everything else — model loading, data collator, LoRA config, Trainer setup, evaluation — is unchanged.

What Changes vs. Phase 3

SettingPhase 3 (full SFT)Phase 6 (rejection SFT)Reason
Training data Full manifest (~41K) Filtered CSV (~3–5K) Whole point of rejection sampling
Starting weights Base Qwen3-ASR Base Qwen3-ASR (not the GRPO adapter) Fresh LoRA avoids RL weight artifacts; stacking adapters causes instability
LoRA rank / alpha From locked config Same locked config Comparability with Phase 3 baseline
Epochs 3–5 1 (maybe 2) Small dataset + LoRA = fast overfit risk
Dropout 0.05 0.05–0.10 Bump if filtered set is < 50% of Phase 3 size
Evaluation evaluate_adapter.py evaluate_adapter.py Zero change — reuses fairness bridge
Anti-pattern: continuing the GRPO adapter. Do not load the Phase 5 adapter and run SFT on top of it. The GRPO adapter carries RL-specific weight distributions and stacking two LoRA adapters causes gradient interference. The point of rejection sampling is to distill the GRPO policy's outputs, not its weights.

Why Three Stages — The Decoupling Argument

A single monolithic script would be shorter, but rejection sampling is a threshold-tuning problem and the threshold choice depends on data you don't have until Stage 1 finishes. The CSV handoffs separate the one expensive step from the many cheap ones.

StageCost per runHow often you re-run itWhat you iterate on
1 — Generate ~48 hours (T4) Once (maybe twice if temp changes) N candidates, temperature, model checkpoint
2 — Filter Seconds Dozens of times WER threshold, fairness gap tolerance, min group size
3 — SFT 1–3 hours 3–5 times per filter config Epochs, LR, dropout
Concrete failure avoided: if you fuse Stage 1 and Stage 2 into one script, a bad threshold choice forces re-running the 48-hour generation step. The CSV intermediate makes that mistake cost seconds instead of days.

Key Risks & Mitigations

Risk 1: Unique-Answer Problem Critical

ASR transcription has effectively one correct answer per utterance. Unlike open-ended text where 10 samples produce 10 different valid completions, ASR candidates may be 10 near-duplicates. If that happens, the filter has nothing to select between and "rejection sampling" reduces to straightforward GRPO inference.

Warning signs
  • >80% of utterances have all 10 candidates with identical WER
  • Median pairwise edit distance < 1 word
  • Filtered set is > 95% the size of the original
Mitigations
  • Run a pilot on 100 utterances first before the 48-hour full run
  • Raise temperature to 1.0 or 1.2 if diversity is too low
  • Mix in candidates from the pre-GRPO base model for structural variety

Risk 2: Threshold Bias Against Hard Utterances High

A uniform WER ceiling disproportionately rejects utterances from demographic groups the GRPO model already struggles with — the exact groups that need more data, not less. The two-pass fairness filter (Rule 2 above) exists specifically to prevent this.

Risk 3: VRAM Exhaustion on T4 High

num_return_sequences=10 forces 10 parallel KV caches on a 1.7B model — instant OOM on 15 GB. Mitigation: loop with num_return_sequences=2 five times per utterance, with torch.cuda.empty_cache() between loops.

Risk 4: SFT Overfitting on a Small Set Medium

A filtered set of 3–5K samples with Phase 3's epoch schedule will overfit quickly. Mitigations: reduce epochs to 1, bump dropout to 0.10, monitor eval loss every few hundred steps.
