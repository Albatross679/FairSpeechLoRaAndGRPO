6-Cell Tuning Grid

Each cell ran as an isolated subprocess with an nvidia-smi poller at 1 Hz. Per-cell metrics.json + training_config.json captured runtime-effective values (optimizer actually used, gradient checkpointing actually enabled) — not just requested values — to catch silent fallbacks like fused → torch AdamW.

Caveat on tok/s for dynamic cells (D, E, F). The reported tokens_per_sec field uses batch_size × grad_accum = 4 from TrainingArguments, not the real per-step sample count (which is avg sampler batch × grad_accum ≈ 30–160). Dynamic-cell tok/s is therefore ~20× too low. This does not change the winner — util is the primary sort key and the Occam tiebreak dominates tok/s — but it distorts any quick throughput read across fixed vs dynamic rows. Flagged as a known stub, fix deferred.

Winner Selection & Occam Tiebreak Fix

Selection rule (from PLAN.md): sort by (−mean_gpu_util, −tok/s, +step_time, +complexity), then apply an Occam tiebreak — among all cells within 2 pp of the util leader, prefer the lowest complexity score.

StepActionOutcome
1Primary sortD and E tied at 42.6%, C at 41.0% — all three within 2 pp of leader.
2Build tie bandTie band = {C, D, E}.
3Complexity scoresC = 0 (fixed batching) · D = 1 (dynamic) · E = 1 (dynamic).
4Occam tiebreakCell C wins — only fixed-batching cell in the band.
Rule-1 bug auto-fixed in this session. The original tune_vram.select_winner() only compared the top two sorted cells — it would have silently promoted cell E (runner-up) and never considered cell C (third in the sort). Fix widens the Occam check to scan every cell within the 2 pp tie band, then picks the lowest complexity deterministically. Committed as e537ab2. Re-ran selection against the existing metrics.json files — winner flipped from E → C. No re-training needed.

Why BLOCKED — I/O Is the Binding Constraint

Four independent observations from the grid converge on "dataloader I/O is the bottleneck." Compute, optimizer, and memory are all ruled out.

Observation 1

Util flat across 8× compute variance

Cell C steps in 2.01 s. Cell E steps in 16.08 s. Per-step compute differs by 8×. Mean GPU util is within 1.6 pp. If compute were the bottleneck, heavier cells would pull higher util. They don't.

Observation 2

Disabling GC lowered util

Cell F (no gradient checkpointing) is faster per step than cell D but util drops from 42.6% → 35.9%. A faster step leaves the dataloader less time to prefetch, so the GPU waits longer on data. Textbook I/O-bound signature.

Observation 3

Fused AdamW is a no-op

A → B swaps adamw_torch for adamw_torch_fused. Util gain: +0.7 pp. The optimizer is not on the critical path.

Observation 4

Only microbatch size moved the needle

A → C doubles the microbatch (4 → 8) and gains +6 pp util. That's consistent with amortizing data transfer over a larger batch — exactly what you'd expect from a dataloader-bound training loop.

VRAM and loss reproduce deterministically, but GPU util sits on the edge of the tolerance window because the steady-state sampling window is sensitive to when the training loop actually starts firing. One noisy run near 50% is not enough to upgrade the verdict to go_with_warning. Conservative call: keep verdict = BLOCKED and force an I/O mitigation before the expensive full run.

6 I/O Mitigation Options

Task 3 fired automatically when the verdict came back blocked. It produced io_remediation_report.md with six ranked options, each analyzed for impact, cost, and risk.

# Option Impact Cost Recommended
1 Pre-decode audio to WebDataset tar shards 3–5× throughput ~50 GB disk · 4–6 h preprocessing · medium code primary (root-cause)
2 Stage rotating subset into /dev/shm 1.5–2.5× ~15 GB RAM · small code companion only

6 Accept 42% util, launch Plan 03-03 as-is none ~73 h GPU time (2 epochs) no — 3 days blocked

Recommended Path Forward

Two-step strategy:

(1) Zero-cost pre-flight first. Re-run cell C on this host with --dataloader_num_workers 2 and a warm page cache (Options 3 + 4). ~10 min of work. If the new mean GPU util clears 55%, upgrade the verdict to go_with_warning and launch Plan 03-03 unchanged.

(2) If still below floor — fix the root cause. Pick one of:

  • Option 1 (WebDataset pre-decode) — unblocks this run and every future ASR training phase. Recommended if you're staying on this host.
  • Option 5 (migrate host) — faster wall-clock to unblock, but doesn't improve any downstream phase.

Do not pursue Option 6 as the primary path — burning 73 h of GPU time at 41% util is cheaper to avoid than to endure.
