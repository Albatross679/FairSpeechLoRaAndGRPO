# Phase 2.1: Advanced HP Sweep — PLoRA + ASHA + RsLoRA - Context

**Gathered:** 2026-04-07
**Status:** Ready for planning

<domain>
## Phase Boundary

Upgrade the Phase 2 HP sweep with three incremental techniques — PLoRA (packed parallel adapters), ASHA pruning (early stopping), and RsLoRA (rank-stable scaling) — to explore a larger HP space (9 dimensions, 100 trials) in less wall-clock time than Phase 2's 20-trial sequential sweep. Produces a new locked config that may replace Phase 2's rank=4 winner.

</domain>

<decisions>
## Implementation Decisions

### Incremental rollout (3 stages)
- **D-01:** Stage 1 — PLoRA only. Validate packing correctness against Phase 2 sequential results.
- **D-02:** Stage 2 — PLoRA + ASHA. Add HyperbandPruner, increase trial count to 100.
- **D-03:** Stage 3 — PLoRA + ASHA + RsLoRA. Enable `use_rslora=True`, re-sweep to test if optimal rank changes.
- **D-04:** Each stage must pass a gate before advancing: top-3 configs must share ≥2 entries with the prior stage's top-3 (matched by rank + target_mlp). This proves the new technique doesn't break ranking.

### PLoRA implementation
- **D-05:** True packing — single forward pass through frozen base model, then branch into N parallel LoRA adapter paths. Not sequential weight-swapping.
- **D-06:** Dynamic auto-detect packing count. Profile VRAM for one adapter at the trial's rank/target_mlp setting, compute remaining headroom, pack as many as fit.
- **D-07:** VRAM safety margin: 2 GB headroom (pack until estimated VRAM ≤ 13GB on T4's 15GB). Leaves room for allocation spikes and fragmentation.

### ASHA pruning (Stage 2+)
- **D-08:** Switch from Phase 2's `MedianPruner` to `optuna.pruners.HyperbandPruner`.
- **D-09:** Config: `min_resource=25` (no pruning before step 25), `reduction_factor=3` (keep top 1/3 per rung).
- **D-10:** Rung schedule: report at steps {25, 50, 75} for 100-step trials.
- **D-11:** 100 trials total. ASHA prunes ~2/3 per rung, so effective compute ≈ 33 full trials.

### HP search space (expanded from Phase 2)
- **D-12:** 9 HP dimensions total. Original 6 from Phase 2 plus 3 new:
  - Original: learning rate (1e-5 to 1e-3 log), LoRA rank {4, 8, 16, 32}, alpha ratio {1, 2}, dropout (0.0-0.15), target MLP {true, false}, weight decay (1e-6 to 1e-2 log)
  - New: LR scheduler {linear, cosine}, warmup ratio {0.0, 0.03, 0.1}, gradient accumulation steps {1, 2, 4}
- **D-13:** Batch size is NOT a sweep dimension — it's a VRAM lever used by the dynamic packing logic. Per-device batch size stays at 2; effective batch varies via gradient accumulation.
- **D-14:** Same subset size (500 samples) and steps per trial (100) as Phase 2 for apples-to-apples comparison.

### RsLoRA (Stage 3)
- **D-15:** Add `use_rslora=True` to LoraConfig. Changes scaling from α/r to α/√r.
- **D-16:** Same 9 HP dimensions and ranges. The only change is the RsLoRA flag — isolates its effect.
- **D-17:** If RsLoRA changes the optimal rank (e.g., rank=16 beats rank=4), the new config replaces Phase 2's locked config and feeds into Phase 3.

### Phase 3 integration
- **D-18:** Phase 3's locked config will be updated to use the best config from Phase 2.1 (whichever stage produces the final winner). Phase 2's rank=4 config is not sacred.

### Claude's Discretion
- Exact VRAM profiling implementation (how to measure single-adapter footprint)
- Optuna study persistence (in-memory vs SQLite)
- Whether to use W&B for sweep tracking or keep Optuna-only
- Exact adapter isolation implementation (separate optimizers vs grouped params)

</decisions>

## Existing Code Insights

### Phase 2 Results (baseline for validation gates)
- Best trial (sweep): trial 15, eval_loss=0.159, rank=4, alpha_ratio=2, target_mlp=true, lr=9.97e-4
- Locked config (after validation): rank=4, alpha_ratio=1, dropout=0.0, lr=8.2e-4, target_mlp=false
- Phase 2 sweep: 20 trials × 100 steps, ~1h 32min on T4
- Peak VRAM during training: 9.97 GB (rank=4, target_mlp=false)

### Key Patterns to Preserve
- `patch_outer_forward()` — delegates to `model.thinker.forward()` for HF Trainer compatibility
- Audio loading via `soundfile.read()` (not torchaudio — torchcodec dependency issue)
- Encoder freeze + re-freeze after PEFT wrapping (PEFT may add LoRA to audio_tower q/k/v_proj)
- Gradient checkpointing with `use_reentrant=False`

### Hardware
- T4 (15 GB VRAM), currently at 0 MB usage
- VRAM budget for packing: ≤13 GB (2 GB safety margin)

### PLoRA Implementation Considerations
- Frozen base model: ~3.4 GB (1.7B params × bf16)
- Each LoRA adapter (rank=4, attn only): ~50-100 MB weights + optimizer state
- Activation memory is the bottleneck — each parallel adapter needs its own backward pass activations
- Gradient checkpointing reduces per-adapter activation cost significantly

</code_context>

<specifics>
## Specific Ideas

- The 3-stage rollout de-risks the complex PLoRA implementation — if Stage 1 fails validation, we debug before adding ASHA complexity
- Dynamic packing means rank=32 trials may only pack 2 adapters while rank=4 trials pack 4-5 — this is expected and correct
- The expanded HP space (9 dims, 100 trials) is well within TPE+ASHA's effective exploration capacity
- Gradient accumulation as a sweep dimension lets us test whether larger effective batches improve convergence without touching per-device batch size or VRAM

</specifics>

<deferred>
## Deferred Ideas

- Multi-GPU PLoRA (shard frozen model across GPUs) — not applicable on single T4
- Adapter merging / adapter soup from top-K sweep results — interesting but out of scope for sweep infra
- Population-based training (PBT) as alternative to TPE+ASHA — higher complexity, save for future work

</deferred>
