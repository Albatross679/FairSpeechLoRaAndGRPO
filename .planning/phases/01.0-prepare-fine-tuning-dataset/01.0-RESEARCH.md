# Phase 1.0: Prepare Fine-Tuning Dataset - Research

**Researched:** 2026-04-05
**Domain:** Dataset splitting, demographic-stratified data loading, audio preprocessing for ASR fine-tuning
**Confidence:** HIGH

## Summary

Phase 1.0 produces train/eval splits for Fair-Speech and Common Voice 24 with demographic labels preserved, plus PyTorch data loaders yielding `(audio_tensor, reference_transcript, demographic_group)` tuples. The existing codebase already has manifest creation scripts (`prepare_fairspeech.py`, `prepare_dataset.py`) with all label standardization maps. The new work is: (1) splitting the Fair-Speech manifest into train/eval at the speaker level, (2) retaining the Common Voice test split as-is for accent evaluation, (3) building a PyTorch Dataset/DataLoader with demographic-stratified sampling and oversampling for small groups, and (4) a validation script following the established `validate_*.py` pattern.

The key technical challenge is **speaker-level splitting** for Fair-Speech. The dataset has 593 participants, each identified by a de-identified unique number. The existing `prepare_fairspeech.py` does not preserve a speaker identifier -- it uses `hash_name` (per-utterance) as `utterance_id`. The Fair-Speech metadata TSV likely contains a participant identifier column that must be discovered at runtime and used for `GroupShuffleSplit` to prevent data leakage. This is a medium-risk item that needs runtime verification.

**Primary recommendation:** Extend `prepare_fairspeech.py` to produce train/eval manifests with speaker IDs preserved, use `sklearn.model_selection.StratifiedGroupKFold` (or `GroupShuffleSplit`) for the split, then build a lightweight PyTorch `Dataset` class that reads manifest CSVs and loads audio via `soundfile`.

### Pattern 1: Speaker-Level Stratified Splitting
**What:** Split Fair-Speech utterances into train/eval ensuring no speaker appears in both splits, while maintaining ethnicity distribution proportions.
**When to use:** Always for Fair-Speech train/eval split (D-01).
**Example:**
```python
# Source: sklearn docs + project CONTEXT.md D-01
import pandas as pd
from sklearn.model_selection import GroupShuffleSplit

df = pd.read_csv("fs_manifest.csv")

# Need speaker_id column from metadata TSV
# Fair-Speech uses de-identified participant numbers (593 speakers)
splitter = GroupShuffleSplit(n_splits=1, test_size=0.2, random_state=SEED)

# groups=speaker_id ensures no speaker in both splits
# Stratify by ethnicity for proportional representation
train_idx, eval_idx = next(splitter.split(df, y=df["ethnicity"], groups=df["speaker_id"]))

df_train = df.iloc[train_idx]
df_eval = df.iloc[eval_idx]
```

**Important caveat:** `GroupShuffleSplit` groups by speaker but does NOT guarantee stratification by `y`. For true stratified-group splitting, use `StratifiedGroupKFold` with k=5 and take one fold as eval:

```python
from sklearn.model_selection import StratifiedGroupKFold

sgkf = StratifiedGroupKFold(n_splits=5, shuffle=True, random_state=SEED)
# Take first fold as eval (20%), rest as train (80%)
for train_idx, eval_idx in sgkf.split(df, y=df["ethnicity"], groups=df["speaker_id"]):
    break  # first fold only
```

### Pattern 2: PyTorch Dataset for ASR with Demographics
**What:** Custom Dataset that loads audio via soundfile, returns (waveform, transcript, demographic_dict).
**When to use:** For all training and evaluation data loading.
**Example:**
```python
# Source: PyTorch docs + project architecture
import torch
from torch.utils.data import Dataset
import soundfile as sf
import pandas as pd

class ASRFairnessDataset(Dataset):
    def __init__(self, manifest_csv: str, demographic_axis: str = "ethnicity",
                 sample_rate: int = 16000):
        self.df = pd.read_csv(manifest_csv)
        self.demographic_axis = demographic_axis
        self.sample_rate = sample_rate

    def __len__(self):
        return len(self.df)

    def __getitem__(self, idx):
        row = self.df.iloc[idx]
        # Load audio as float32 numpy, then convert to tensor
        audio, sr = sf.read(row["audio_path"], dtype="float32")
        if sr != self.sample_rate:
            # Resample if needed (Fair-Speech is 16kHz, CV may vary)
            import torchaudio.functional as F
            audio = torch.from_numpy(audio)
            audio = F.resample(audio, sr, self.sample_rate)
        else:
            audio = torch.from_numpy(audio)

        return {
            "audio": audio,
            "transcript": row["sentence"],
            "demographic_group": row.get(self.demographic_axis, ""),
            "utterance_id": row["utterance_id"],
        }
```

### Pattern 3: Demographic-Stratified BatchSampler
**What:** Custom sampler ensuring each batch contains utterances from multiple demographic groups, with oversampling for small groups.
**When to use:** During GRPO training to ensure fairness reward is meaningful per batch.
**Example:**
```python
# Source: PyTorch docs + D-09 requirement
from torch.utils.data import Sampler
import numpy as np

class DemographicStratifiedSampler(Sampler):
    def __init__(self, demographics: pd.Series, batch_size: int,
                 min_per_group: int = 2, seed: int = 42):
        self.demographics = demographics.values
        self.batch_size = batch_size
        self.min_per_group = min_per_group
        self.rng = np.random.RandomState(seed)

        # Build per-group index lists
        self.group_indices = {}
        for idx, group in enumerate(self.demographics):
            if group:  # skip missing labels
                self.group_indices.setdefault(group, []).append(idx)

        # Compute oversampling weights
        group_sizes = {g: len(idxs) for g, idxs in self.group_indices.items()}
        max_size = max(group_sizes.values())
        self.oversample_weights = {
            g: max_size / size for g, size in group_sizes.items()
        }

    def __iter__(self):
        # Implementation: round-robin from groups with oversampling
        ...

    def __len__(self):
        return len(self.demographics)
```

### Anti-Patterns to Avoid
- **Utterance-level splitting without speaker grouping:** Causes data leakage. Same speaker in train and eval means the model memorizes speaker characteristics, not demographic patterns. [CITED: CONTEXT.md D-01]
- **Loading full audio into memory at Dataset.__init__:** Fair-Speech is ~26K utterances of ~7.4s average. Loading all audio upfront wastes RAM. Load lazily in `__getitem__`. [ASSUMED]
- **Using torchaudio.load() directly:** Broken in this environment (requires torchcodec). Use soundfile.read() instead. [VERIFIED: runtime test]
- **Modifying existing prepare_*.py scripts:** Per ARCHITECTURE.md integration principle: "Do not modify existing scripts." New splitting logic goes in a new script. [CITED: .planning/research/ARCHITECTURE.md]

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Speaker-level stratified splitting | Manual group assignment + random shuffle | `sklearn.model_selection.StratifiedGroupKFold` | Handles edge cases (speakers with single ethnicity), deterministic, well-tested |
| Audio loading + resampling | Custom WAV parser | `soundfile.read()` + `torchaudio.functional.resample()` | Handles formats, sample rates, endianness correctly |
| Text normalization | Regex-based normalizer | `whisper_normalizer.english.EnglishTextNormalizer` | Matches Whisper's internal normalization exactly -- critical for WER consistency with existing pipeline |
| Demographic distribution stats | Manual counting loops | `pandas.DataFrame.groupby().size()` + `value_counts()` | Cleaner, handles NaN/empty correctly |

**Key insight:** The existing pipeline already solved label standardization (ETHNICITY_MAP, ACCENT_MAP, AGE_MAP, L1_KEEP). Reuse these maps by importing from existing scripts or copying the constants. Do not redefine them.

## Common Pitfalls

### Pitfall 1: Missing Speaker ID in Fair-Speech Manifest
**What goes wrong:** The existing `prepare_fairspeech.py` does not include a speaker/participant identifier in its output CSV. The `hash_name` is per-utterance, not per-speaker. Without speaker ID, GroupShuffleSplit cannot enforce speaker-disjoint splits.
**Why it happens:** The original script was built for evaluation (no split needed), so speaker grouping was irrelevant.
**How to avoid:** The Fair-Speech metadata TSV likely has a participant identifier column (the paper states "Participants de-identified with unique numbers"). The new splitting script must read this column from the raw TSV. Discover the actual column name at runtime by inspecting TSV headers.
**Warning signs:** If all `hash_name` values are unique (they should be -- they're utterance IDs), do NOT use them as speaker IDs. Look for a column with ~593 unique values.

### Pitfall 2: StratifiedGroupKFold vs GroupShuffleSplit Confusion
**What goes wrong:** `GroupShuffleSplit` respects groups but does NOT stratify by label. The split may have disproportionate ethnicity representation if speakers from certain ethnicities cluster together.
**Why it happens:** The names are confusingly similar. "Group" means "don't split groups across train/eval." "Stratified" means "preserve label proportions."
**How to avoid:** Use `StratifiedGroupKFold(n_splits=5)` which does BOTH: keeps speaker groups intact AND stratifies by ethnicity. Take one fold as eval (20%), remaining as train (80%).
**Warning signs:** Large ethnicity ratio differences between train and eval splits.

### Pitfall 3: torchaudio.load() Fails in This Environment
**What goes wrong:** `torchaudio.load()` raises `ImportError: TorchCodec is required`. Code that uses `torchaudio.load()` will crash.
**Why it happens:** torchaudio 2.11 dropped the `sox_io` and `soundfile` backends, requiring `torchcodec` which is not installed.
**How to avoid:** Use `soundfile.read()` directly: `audio, sr = sf.read(path, dtype='float32')` then `torch.from_numpy(audio)`. Works reliably.
**Warning signs:** Any import of torchaudio for audio I/O.

### Pitfall 4: Text Normalization Inconsistency
**What goes wrong:** If the new split manifests use different text normalization than the existing pipeline, WER numbers will be incomparable.
**Why it happens:** `prepare_fairspeech.py` uses `EnglishTextNormalizer` from whisper/whisper_normalizer. If that package is not installed, the new script may skip normalization or use a different one.
**How to avoid:** Install `whisper-normalizer` package (`pip install whisper-normalizer`). Use the exact same normalization path as existing scripts. The existing `prepare_fairspeech.py` already normalizes `sentence` -- so if reading from `fs_manifest.csv`, normalization is already done.
**Warning signs:** WER values that differ from existing baseline evaluations on the same data.

### Pitfall 5: Common Voice MP3 vs WAV Format
**What goes wrong:** Common Voice stores audio as `.mp3` files. The data loader must handle both `.wav` (Fair-Speech) and `.mp3` (Common Voice) formats.
**Why it happens:** Different datasets use different formats. `soundfile` handles WAV natively but may need additional codecs for MP3.
**How to avoid:** Test `soundfile.read()` on an MP3 file. If it fails, use `librosa.load()` as fallback (already installed). Or convert MP3 to WAV during manifest creation.
**Warning signs:** `RuntimeError` when loading CV audio files.

### Pitfall 6: Oversampling Changes Effective Epoch Length
**What goes wrong:** With stratified oversampling (D-09), minority groups are repeated multiple times per epoch. The effective epoch is longer than the dataset size, and the model sees minority samples more often. This is intentional for training fairness but must be accounted for in learning rate scheduling and epoch counting.
**Why it happens:** `WeightedRandomSampler` with `replacement=True` creates an effectively infinite stream.
**How to avoid:** Set `num_samples` explicitly on the sampler to define epoch length. Document the oversampling ratio per group in the validation output.

import pandas as pd
from sklearn.model_selection import StratifiedGroupKFold

SEED = 42

### Audio Loading with soundfile (Environment-Safe)
```python
# Source: verified in this environment
import soundfile as sf
import torch

def load_audio(path: str, target_sr: int = 16000) -> torch.Tensor:
    """Load audio file, return float32 tensor at target sample rate."""
    audio, sr = sf.read(path, dtype="float32")
    audio = torch.from_numpy(audio)
    if sr != target_sr:
        # torchaudio.functional.resample works even though torchaudio.load doesn't
        import torchaudio.functional as F
        audio = F.resample(audio, sr, target_sr)
    return audio
```
