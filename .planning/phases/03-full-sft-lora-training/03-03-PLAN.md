<objective>
Execute full-scale SFT LoRA training on the complete ~1.16M sample dataset using validated hyperparameters from Phase 2.1 and the frozen launch config from Plan 03-02. This produces the production-quality baseline adapter.

Purpose: Train the definitive accuracy-only SFT LoRA baseline that all fairness methods will be compared against.
Output: Trained adapter in `outputs/full-lora/adapter/`, training metadata in `outputs/full-lora/training_config.json`, train/eval split CSVs.

**Scope boundary:** No trainer-level flag decisions live in this plan. Batching strategy, gradient checkpointing, optimizer choice, dataloader workers, frame budget — all of those were tuned and frozen in Plan 03-02 and are consumed via `--vram_config outputs/full-lora-vram-tune/vram_config.json`. This plan's job is just to launch, monitor, and checkpoint the long run; any deviation from the frozen config would invalidate the Plan 03-02 measurements.
</objective>

**Hardware:** NVIDIA RTX A6000, 48 GB VRAM, torch 2.10.0+cu128. CPU quota 3.96 cores (cgroup-limited). Storage on overlayfs — I/O ceiling is a known constraint surfaced in Plan 03-01 and addressed (or explicitly accepted) in Plan 03-02.

**Launch config:** fully sourced from `outputs/full-lora-vram-tune/vram_config.json`, written by Plan 03-02's grid runner. All trainer flags (batching strategy, gradient checkpointing, optimizer, batch_size, grad_accum, frame_budget, max_batch_size, dataloader workers) come from there. This plan does not pick any trainer-level flags on its own — changing any of them here would invalidate Plan 03-02's measurements and the projected wall-clock.

CLI for full-scale training:
```bash
PYTORCH_CUDA_ALLOC_CONF=expandable_segments:True \
python scripts/training/train_standard_lora.py \
    --mode train \
    --full_dataset \
    --fs_manifest outputs/manifests/fs_train.csv \
    --cv_manifest outputs/manifests/cv_train.csv \
    --output_dir outputs/full-lora \
    --locked_config_path outputs/standard-lora/locked_config.json \
    --vram_config outputs/full-lora-vram-tune/vram_config.json \
    --lr 9.8e-4 \
    --num_epochs 2 \
    --save_total_limit 3 \
    --wandb_project asr-fairness-lora
```

Note: `--lr 9.8e-4` is kept as an explicit CLI flag (not in vram_config.json) because it is an HP decision driven by D-16 sqrt-scaling, not a VRAM tuning choice. The vram_config.json `flags.batch_size × flags.grad_accum` product IS still effective_batch=16 (cross-checked by Plan 03-02), so the sqrt-scaled LR stays valid regardless of which cell won.

**Expected scale:**
- ~1,159,789 total samples → 90/10 split → ~1,043,810 train + ~115,979 eval
- Steps per epoch: 1,043,810 / 16 = ~65,238 (effective batch=16 held constant across Plan 03-02 cells)
- **Wall-clock estimate:** read from Plan 03-02's SUMMARY — do NOT use the 36h/epoch number from the pre-tuning plan. Plan 03-02 projects wall-clock from `winner.median_step_time_s × 65,238 steps/epoch × epochs_until_early_stop`. If Plan 03-02's verdict was `go_with_warning`, the wall-clock will be longer than the "ideal GPU-Util ≥ 70%" projection — the SUMMARY documents the I/O-bound caveat.
- Early stopping (patience=3, eval every ~5000 steps) should terminate within 1–2 epochs
- **Peak VRAM budget:** < 44 GB (A6000 has 48 GB; 4 GB safety margin). Plan 03-02 already confirmed the winning cell's peak VRAM is well inside this budget.

**Launch training:**
```bash
# expandable_segments:True reduces fragmentation OOMs on variable-length audio batches
# over a long run — cheap insurance, no perf cost.
# --vram_config is the Plan 03-02 frozen launch config (batching strategy, GC, optim,
# workers, frame_budget/max_batch_size, etc). --lr stays explicit because it's a D-16
# HP decision, not a VRAM tuning choice.
PYTORCH_CUDA_ALLOC_CONF=expandable_segments:True \
python scripts/training/train_standard_lora.py \
    --mode train \
    --full_dataset \
    --fs_manifest outputs/manifests/fs_train.csv \
    --cv_manifest outputs/manifests/cv_train.csv \
    --output_dir outputs/full-lora \
    --locked_config_path outputs/standard-lora/locked_config.json \
    --vram_config outputs/full-lora-vram-tune/vram_config.json \
    --lr 9.8e-4 \
    --num_epochs 2 \
    --save_total_limit 3 \
    --wandb_project asr-fairness-lora
```

The `--lr 9.8e-4` implements D-16 (sqrt-scaled LR from 3.465e-4 → 9.8e-4 to match the 8× larger effective batch relative to Phase 2.1's sweep scale of eff=2). It cannot be read from `locked_config.json` because the locked config captures the sweep scale (lr=3.47e-4), and Phase 3 deliberately diverges from it. It is NOT in `vram_config.json` either because Plan 03-02's grid held effective_batch=16 constant across cells — the sqrt-scaled LR is valid for every cell, so the LR choice belongs to the HP layer, not the VRAM layer.

All other trainer flags (per-device batch, grad_accum, optim, frame_budget, max_batch_size, num_workers, GC on/off) come from `vram_config.json`. Do NOT add them explicitly on the command line — if the CLI override and the config disagree, the CLI wins and the Plan 03-02 measurements become unreliable.

**Monitoring during training (check periodically):**
- Watch for loss decreasing in log output (logging_steps=10)
- Watch for eval_loss at eval checkpoints (~every 2000 steps)
- Watch for peak VRAM staying within Plan 03-02's measured peak + 2 GB. If it drifts higher, investigate — could indicate a runtime difference from the tuning run.
- If early stopping triggers, the script will print it and stop automatically
- If GPU-Util drops substantially below Plan 03-02's measured value for a sustained period, suspect I/O degradation (page cache cold on a new epoch, disk pressure) — check `iostat`/`free -h`

**If training crashes or is interrupted:**
Resume from last checkpoint. Because the launch config is locked in `vram_config.json`, the resume command is identical to the initial launch plus `--resume_from_checkpoint`:
```bash
PYTORCH_CUDA_ALLOC_CONF=expandable_segments:True \
python scripts/training/train_standard_lora.py \
    --mode train \
    --full_dataset \
    --fs_manifest outputs/manifests/fs_train.csv \
    --cv_manifest outputs/manifests/cv_train.csv \
    --output_dir outputs/full-lora \
    --locked_config_path outputs/standard-lora/locked_config.json \
    --vram_config outputs/full-lora-vram-tune/vram_config.json \
    --lr 9.8e-4 \
    --num_epochs 2 \
    --save_total_limit 3 \
    --wandb_project asr-fairness-lora \
    --resume_from_checkpoint outputs/full-lora/checkpoint-XXXX
```
Replace `checkpoint-XXXX` with the latest checkpoint directory found in `outputs/full-lora/`. The single source of truth for trainer flags is `vram_config.json` — never hand-edit the resume command to add or remove trainer flags.
