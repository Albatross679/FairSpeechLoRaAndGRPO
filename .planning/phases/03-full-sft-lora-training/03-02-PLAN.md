**Launch training:**
```bash
# expandable_segments:True reduces fragmentation OOMs on variable-length audio
# batches over a 36h+ run — cheap insurance, no perf cost.
PYTORCH_CUDA_ALLOC_CONF=expandable_segments:True \
python scripts/training/train_standard_lora.py \
    --mode train \
    --full_dataset \
    --fs_manifest outputs/manifests/fs_train.csv \
    --cv_manifest outputs/manifests/cv_train.csv \
    --output_dir outputs/full-lora \
    --locked_config_path outputs/standard-lora/locked_config.json \
    --num_epochs 2 \
    --save_total_limit 3 \
    --batch_size 4 \
    --grad_accum 4 \
    --lr 9.8e-4 \
    --dataloader_num_workers 4 \
    --dataloader_prefetch_factor 2 \
    --wandb_project asr-fairness-lora
```

The `--batch_size 4 --grad_accum 4 --lr 9.8e-4` overrides implement D-06 (effective batch = 16, up from the sweep's eff=2) and D-16 (sqrt-scaled LR from 3.465e-4 → 9.8e-4 to match the 8× larger effective batch). These cannot be read from `locked_config.json` because the locked config captures the Phase 2.1 sweep scale (grad_accum=1, lr=3.47e-4), and Phase 3 deliberately diverges from it.

The dataloader flags (`--dataloader_num_workers 4 --dataloader_prefetch_factor 2`) plus the Plan 01 TrainingArguments changes (pin_memory=True, persistent_workers=True, SDPA attention) are the VRAM-playbook Phase 3 fixes: without them, a 1.16M-sample audio dataset decoded from disk will starve the GPU and stretch the 36h/epoch estimate well beyond that. Plan 01's dry-run already gated on mean GPU-Util ≥ 70% with this exact config, so the same flags transfer directly. **If Plan 01 required bumping workers to 8 or 12 to clear the gate, use that same value here — do not silently fall back to 4.**

**Optional: If Plan 01 dry-run showed heavy padding waste** (variance in step time, or you inspected sample durations and saw wildly variable lengths), add `--dynamic_batch --frame_budget 120 --max_batch_size 24` to activate the FrameBudgetBatchSampler. This packs short utterances into larger batches and long utterances into smaller ones, keeping activation memory roughly constant. Trades off some setup time (one-time duration computation, cached in output_dir) for better throughput on the 1.16M variable-length dataset.

**Monitoring during training (check periodically):**
- Watch for loss decreasing in log output (logging_steps=10)
- Watch for eval_loss at eval checkpoints (~every 2000 steps)
- Watch for VRAM staying < 14GB
- If early stopping triggers, the script will print it and stop automatically

Replace `checkpoint-XXXX` with the latest checkpoint directory found in `outputs/full-lora/`.
