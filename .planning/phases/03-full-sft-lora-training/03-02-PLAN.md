<objective>
Find a provably good VRAM/throughput configuration for the Wave 3 full-scale training run BEFORE committing to it, then freeze that configuration as `vram_config.json` so Plan 03-03's launch is a single-flag one-liner.

**Why this plan exists:** Plan 03-01's dry-run peaked at 7.87 GB on an A6000 with a 48 GB budget (16% utilization) and achieved only 30–35% mean GPU-Util on this host. The upcoming Plan 03-03 is a long training run (duration unknown until we measure). Launching it without first measuring where the real bottleneck is — compute, VRAM capacity, or I/O — would waste many GPU-hours. This plan is the "tune before scale" gate: cheap (~20 min of GPU time), delivers a frozen config, and produces explicit go/no-go evidence.

**Scope boundary:** this plan tunes *trainer-level* flags (batching strategy, gradient checkpointing, optimizer, dataloader). It does NOT modify learning rate, LoRA rank, model architecture, or anything that affects convergence (effective batch is held constant at 16 across cells). It also does NOT pre-decode the audio dataset — that is a separate, much larger data-engineering concern left for a follow-up.

**Output:** a committed `vram_config.json` + grid results + plan 03-03 updated to consume it.
</objective>

**Grid definition** (6 cells — `configs/vram_tune_grid.json` or inline in `tune_vram.py`):

| Cell | Batching | GC | Optim | per_device_batch × grad_accum | frame_budget | max_batch_size | Rationale |
|---|---|---|---|---|---|---|---|
| **A** | fixed | on | adamw_torch | 4 × 4 | — | — | **Baseline** — reproduces Plan 01/previous-02 config verbatim for anchoring |
| **B** | fixed | on | adamw_torch_fused | 4 × 4 | — | — | Isolate fused-adam delta |
| **C** | fixed | on | adamw_torch_fused | 8 × 2 | — | — | Doubled per-device microbatch (eff=16 unchanged), tests "bigger batch reduces per-sample overhead" hypothesis without dynamic batching |
| **D** | dynamic | on | adamw_torch_fused | 1 × 4 | 180 | 64 | Dynamic batching (FrameBudgetBatchSampler), conservative budget |
| **E** | dynamic | on | adamw_torch_fused | 1 × 4 | 300 | 96 | Dynamic batching aggressive — uses more of the 48 GB |
| **F** | dynamic | **off** | adamw_torch_fused | 1 × 4 | 180 | 64 | GC off + dynamic. Expected 2.5–4× activation memory vs cell D. Still within 44 GB budget based on cell D peak projection. |

All cells share: `--full_dataset --fs_manifest outputs/manifests/fs_train.csv --cv_manifest outputs/manifests/cv_train.csv --locked_config_path outputs/standard-lora/locked_config.json --lr 9.8e-4 --dataloader_num_workers 4 --dataloader_prefetch_factor 2 --max_steps 50 --wandb_project none`.

**Pass criteria per cell:**
- Hard: `peak_vram_gb < 44.0` AND no OOM AND `final_loss` is finite AND `final_loss < 3.0` (loss-decreasing sanity — prevents a broken cell from "winning" on util alone).
- Hard: `median_step_time_s < 30.0` (catches cells where something is catastrophically wrong).

**Winner selection:**
Among passing cells, pick the one with the **highest mean_gpu_util (steps 10–49, skipping warmup)**. Tiebreak on `tokens_per_sec`, then `median_step_time_s` (lower is better). If multiple cells tie within 2 percentage points on GPU-Util, prefer the cell with the *simpler* config (fixed batching over dynamic; GC on over GC off) — this is an "Occam tiebreaker" to avoid adopting complexity for a marginal gain.

**GPU-Util gate for the winner (host-aware):**
- **Target:** mean_gpu_util ≥ 70% (the original maximize-vram gate)
- **Acceptable:** mean_gpu_util ≥ 55% (lowered for this host because the 4-core cgroup + overlayfs I/O bottleneck is a hardware constraint, not a tuning failure)
- **Blocker:** if the best cell is < 55%, Task 3 runs and reports I/O mitigation options before Plan 03-03 can launch. Do NOT let Plan 03-03 launch into a sub-55% regime — the 36h (or whatever it projects to) would be cut in half by fixing I/O first.

**Step 1.3 — Create `scripts/training/tune_vram.py`:**

A self-contained script (no imports from training internals beyond what `subprocess.run` needs). Structure:

```python
#!/usr/bin/env python
"""VRAM/throughput tuning grid runner.

Runs a sequence of training cells as fresh subprocesses, measures each
with the Plan 01 gpu.log + timing.txt + training_config.json artifacts,
picks a winner, and freezes it as vram_config.json.

Usage:
    python scripts/training/tune_vram.py \
        --output_dir outputs/full-lora-vram-tune \
        --grid_spec scripts/training/vram_tune_grid.json \
        --max_steps 50 \
        --gpu_poll_interval_s 1

Non-zero exit if no cell passes (lets CI gate fail).
"""
```

Core responsibilities:

1. **Grid spec loading** — either read `--grid_spec PATH` (JSON with a `cells` array) or fall back to an embedded default grid matching the 6 cells in this plan. Each cell is a dict like:
   ```json
   {"id": "A", "label": "baseline (fixed, GC on, torch)",
    "flags": {"batch_size": 4, "grad_accum": 4, "optim": "adamw_torch"}}
   ```

2. **Per-cell execution** — for each cell:
   - Create `{output_dir}/cell-{id}/` directory.
   - Start `nvidia-smi --query-gpu=utilization.gpu,memory.used --format=csv,noheader,nounits -l {poll_interval}` as a background subprocess, redirecting stdout to `{cell_dir}/gpu.log`. Append a timestamp column from Python's side if nvidia-smi doesn't provide one (Plan 01's validate_dryrun_gates.py handles both formats).
   - Build the `python train_standard_lora.py` command from the shared base + per-cell `flags` dict. Set `--output_dir {cell_dir}`. Set `--max_steps {args.max_steps}`.
   - Run the training subprocess with `subprocess.run(..., env={...'PYTORCH_CUDA_ALLOC_CONF': 'expandable_segments:True'...}, capture_output=True)`. Kill the nvidia-smi subprocess once training returns.
   - Parse `{cell_dir}/training_config.json` for `peak_vram_gb`, `final_train_loss`, `gradient_checkpointing_enabled`, `optim`, `effective_learning_rate`.
   - Call `validate_dryrun_gates.py` functions (import it, don't shell out) to compute `mean_gpu_util` (from steps 10–49, using `timing.txt` markers to clip warmup), `median_step_time_s`, `tokens_per_sec`.
   - Write `{cell_dir}/metrics.json` with all fields + verdict (`pass`/`fail_oom`/`fail_vram`/`fail_loss`/`fail_step_time`).

3. **Isolation invariants:**
   - Each cell is a fresh `subprocess.run` — no shared Python process state.
   - After each cell's training process exits, wait 3 seconds then log `nvidia-smi --query-gpu=memory.used` to confirm VRAM returned to ~0 before launching the next cell. If not (>1 GB residual), sleep another 5s and retry. Fail the sweep if VRAM doesn't drain — something is leaking.
   - Each cell writes to its own output directory; cells never share paths.

4. **Reporting:**
   - After all cells complete, write `grid_results.csv` (one row per cell, columns: cell_id, label, flags_json, peak_vram_gb, mean_gpu_util, median_step_time_s, tokens_per_sec, final_loss, verdict).
   - Write `grid_results.md` with a formatted comparison table + the winner highlighted + explicit rationale for the winner selection.

5. **Winner selection + config freeze (factored out into a function for testability):**
   - Filter cells where `verdict == "pass"`.
   - Sort by: `-mean_gpu_util`, `-tokens_per_sec`, `+median_step_time_s`, `+complexity_score` (where `complexity_score` = 0 for fixed+GC-on, 1 for fixed+GC-off, 1 for dynamic+GC-on, 2 for dynamic+GC-off).
   - The head of the sorted list is the winner. Write `{output_dir}/vram_config.json`:
     ```json
     {
       "source": "phase-03 VRAM tuning, cell {id}",
       "target_host": "{nvidia_smi_name} ({memory_total_gb}GB)",
       "target_host_cpu_cores": {nproc_effective},
       "target_host_storage": "overlayfs",
       "tuned_at": "{iso_timestamp}",
       "grid_spec_sha256": "{sha of grid_spec.json contents}",
       "flags": { ...winning cell's flags, merged with the shared base flags... },
       "measured": {
         "peak_vram_gb": ...,
         "mean_gpu_util": ...,
         "median_step_time_s": ...,
         "tokens_per_sec": ...,
         "final_loss": ...
       },
       "verdict": "go" | "go_with_warning" | "blocked",
       "verdict_reason": "mean_gpu_util {x}% >= 70% target" | "...>= 55% acceptable, below 70% target — I/O is the ceiling on this host" | "...< 55% — I/O remediation required before Plan 03-03"
     }
     ```
