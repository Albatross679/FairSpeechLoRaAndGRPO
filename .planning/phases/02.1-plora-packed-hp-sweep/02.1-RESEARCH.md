# Phase 2.1: Advanced HP Sweep -- PLoRA + ASHA + RsLoRA - Research

**Researched:** 2026-04-07
**Domain:** LoRA hyperparameter sweep optimization (packing, pruning, rank-stable scaling)
**Confidence:** MEDIUM

## Summary

This phase upgrades the Phase 2 sequential Optuna sweep with three techniques: (1) packed adapter training to share a single frozen base model forward pass across multiple LoRA configs, (2) ASHA/Hyperband pruning to kill unpromising trials early, and (3) RsLoRA scaling to enable fair comparison across rank values. The critical finding is that PLoRA (Chen et al., arXiv:2508.02932) is a research prototype with custom CUDA kernels and **no public code or pip-installable library** [VERIFIED: web search, no GitHub repo found]. We must implement a simplified "packed" approach ourselves using PEFT's multi-adapter API (`add_adapter` / `set_active_adapters`) combined with a custom training loop. ASHA and RsLoRA are both available out-of-the-box: Optuna 4.8.0 ships `HyperbandPruner` [VERIFIED: installed] and PEFT 0.18.1 supports `use_rslora=True` in `LoraConfig` [VERIFIED: installed].

VRAM is the binding constraint. Phase 2 peaked at ~10 GB on the T4 (15 GB total), leaving ~5 GB for additional adapters. A rank-4 adapter (our current winner) costs ~77 MB for weights+optimizer+gradients, but activation memory adds ~0.3-0.5 GB per adapter with gradient checkpointing. Conservative packing capacity: **3-5 rank-4 adapters** or **2-3 rank-32 adapters** total. This is far fewer than the 10-13 adapters PLoRA achieves on A100-40GB, but still delivers a meaningful 3-5x throughput improvement over sequential execution.

**Primary recommendation:** Implement a custom multi-adapter training loop that loads the base model once, attaches N adapters via PEFT, and round-robins per-adapter forward/backward passes within each training step. Combine with HyperbandPruner for early stopping and RsLoRA for rank-stable scaling.

### Pattern 1: Multi-Adapter Round-Robin Training

**What:** Load base model once, attach N adapters with different HP configs, iterate through adapters within each training step. Each adapter gets its own data batch, forward pass (shared base + own adapter), backward pass, and optimizer step.

**When to use:** Every packed training step.

**Example:**
```python
# Source: PEFT multi-adapter API + custom training loop
from peft import LoraConfig, get_peft_model

# Step 1: Create base model and first adapter
model = load_base_model()
config_0 = LoraConfig(r=4, lora_alpha=8, use_rslora=True,
                       target_modules=["q_proj","k_proj","v_proj","o_proj"])
model = get_peft_model(model, config_0, adapter_name="trial_0")

# Step 2: Add additional adapters
config_1 = LoraConfig(r=16, lora_alpha=32, use_rslora=True,
                       target_modules=["q_proj","k_proj","v_proj","o_proj"])
model.add_adapter("trial_1", config_1)

# Step 3: Round-robin training step
for adapter_name in ["trial_0", "trial_1"]:
    model.set_adapter(adapter_name)
    batch = next(data_iters[adapter_name])
    outputs = model(**batch)
    loss = outputs.loss
    loss.backward()
    optimizers[adapter_name].step()
    optimizers[adapter_name].zero_grad()
```

### Pattern 3: Packed Trial Execution with ASHA

**What:** Run N adapters per "pack". Each pack is one Optuna trial that internally manages N sub-trials. When a sub-trial is pruned, its adapter is removed and replaced with the next pending config.

**When to use:** Main sweep execution.

**Example:**
```python
# Simplified packed execution
def run_packed_sweep(study, n_trials, pack_size=3, steps=100, rung_steps=[25,50,75]):
    pending_trials = list(range(n_trials))
    
    while pending_trials:
        # Load base model ONCE for this pack
        model, processor = load_model_and_processor()
        
        # Attach pack_size adapters
        pack = pending_trials[:pack_size]
        pending_trials = pending_trials[pack_size:]
        
        adapters = {}
        for trial_idx in pack:
            trial = study.ask()
            config = sample_lora_config(trial)
            adapter_name = f"trial_{trial_idx}"
            if trial_idx == 0:
                model = get_peft_model(model, config, adapter_name=adapter_name)
            else:
                model.add_adapter(adapter_name, config)
            adapters[adapter_name] = {
                "trial": trial, "optimizer": create_optimizer(model, adapter_name),
                "active": True
            }
        
        # Train all adapters in pack
        for step in range(steps):
            for name, info in adapters.items():
                if not info["active"]:
                    continue
                model.set_adapter(name)
                # ... forward, backward, optimizer step ...
                
                if step in rung_steps:
                    eval_loss = evaluate_adapter(model, name)
                    info["trial"].report(eval_loss, step)
                    if info["trial"].should_prune():
                        study.tell(info["trial"], state=optuna.trial.TrialState.PRUNED)
                        info["active"] = False
        
        # Report final results for surviving adapters
        for name, info in adapters.items():
            if info["active"]:
                final_loss = evaluate_adapter(model, name)
                study.tell(info["trial"], final_loss)
        
        # Cleanup
        del model
        torch.cuda.empty_cache()
```

### Pattern 4: VRAM-Aware Packing Planner

**What:** Estimate VRAM per adapter config and determine pack size dynamically.

**When to use:** Before each pack execution.

**Example:**
```python
def estimate_adapter_vram_mb(rank, n_targets, hidden_dim=1536, n_layers=28):
    """Estimate VRAM for one adapter (weights + optimizer + gradients)."""
    n_params = n_layers * n_targets * 2 * hidden_dim * rank
    weights_mb = n_params * 2 / 1e6       # bf16
    optimizer_mb = n_params * 8 / 1e6     # AdamW: 2 fp32 states
    gradients_mb = n_params * 2 / 1e6     # bf16
    activation_overhead_mb = 300          # conservative estimate with grad checkpointing
    return weights_mb + optimizer_mb + gradients_mb + activation_overhead_mb

def compute_pack_size(configs, available_vram_mb=5000):
    """Determine how many adapters fit in available VRAM."""
    total = 0
    for i, cfg in enumerate(configs):
        cost = estimate_adapter_vram_mb(cfg["rank"], cfg["n_targets"])
        if total + cost > available_vram_mb:
            return max(1, i)
        total += cost
    return len(configs)
```

### Anti-Patterns to Avoid

- **Loading model per trial:** The whole point of packing is to avoid this. Load once, attach multiple adapters.
- **Using HuggingFace Trainer for packed training:** Trainer manages one model/one optimizer. Use a custom training loop.
- **Sharing optimizer across adapters:** Each adapter MUST have its own optimizer instance for gradient isolation.
- **Ignoring VRAM when packing:** Always estimate and respect the memory budget. OOM crashes waste more time than running fewer adapters per pack.

**Key insight:** PLoRA's custom CUDA kernels are out of reach, but the core speedup (shared base model + multi-adapter) is achievable with PEFT's existing API and a custom training loop. We sacrifice the fused kernel throughput but retain the memory-sharing benefit.

## Common Pitfalls

### Pitfall 1: Gradient Cross-Contamination Between Adapters

**What goes wrong:** If you don't properly isolate gradients, adapter A's backward pass accumulates gradients into adapter B's parameters (or the shared base model).
**Why it happens:** PEFT's `set_adapter()` changes which adapter is "active" but all adapter parameters remain part of the model graph. Without careful `zero_grad()` scoping, stale gradients persist.
**How to avoid:** (1) Call `optimizer_i.zero_grad()` before each adapter's forward pass. (2) After backward, immediately step the active adapter's optimizer before switching. (3) Verify with a gradient isolation test: train adapter A, check adapter B params haven't changed.
**Warning signs:** All adapters converge to the same loss curve, or loss is NaN/unstable.

### Pitfall 2: VRAM OOM from Rank-32 Packing

**What goes wrong:** Attempting to pack 3+ rank-32 adapters with MLP targets on T4 causes OOM.
**Why it happens:** Rank-32 with 7 targets (attn+MLP) uses ~614 MB per adapter for weights+optimizer+gradients, plus activation memory. Three rank-32 adapters could exceed available ~5 GB headroom.
**How to avoid:** Use the VRAM-aware packing planner. Start with 2 adapters for rank-32 configs, scale up only after empirical measurement. Always try `torch.cuda.max_memory_allocated()` after first pack.
**Warning signs:** `torch.cuda.OutOfMemoryError`, or GPU utilization drops to 0% (process killed).

### Pitfall 3: HyperbandPruner Killing Slow Starters

**What goes wrong:** High-rank configs (16, 32) may start with higher loss than rank-4 configs but converge later. Aggressive pruning kills them before they can demonstrate potential.
**Why it happens:** HyperbandPruner with reduction_factor=3 kills 2/3 at each rung. If the first rung is too early (step 10), configs that need warmup get pruned.
**How to avoid:** Set `min_resource=25` (not lower) so first rung occurs after meaningful training. Test with reduction_factor=2 for gentler pruning. Also, RsLoRA helps here by stabilizing gradient magnitudes at higher ranks.
**Warning signs:** No rank-16/32 configs survive past the first rung; top-k results are all rank-4.

### Pitfall 4: RsLoRA Scaling + Existing Learning Rates

**What goes wrong:** The Phase 2 winning LR (~1e-3) was tuned under standard scaling (alpha/r). RsLoRA changes effective scaling from alpha/r to alpha/sqrt(r), which at rank-4 means 2x larger effective gradients.
**Why it happens:** At rank-4: standard scaling = alpha/4 = 2.0; RsLoRA = alpha/2 = 4.0. The adapter contribution doubles.
**How to avoid:** When comparing RsLoRA vs standard, include learning rate in the sweep space. Don't assume Phase 2's winning LR transfers to RsLoRA. The sweep should find new optimal LRs.
**Warning signs:** RsLoRA runs diverge or produce worse results than standard scaling at the same LR.

### Pitfall 5: Optuna ask/tell API Misuse with Packing

**What goes wrong:** Using `study.optimize()` expects a simple `objective(trial)` function. Packing requires managing multiple trials concurrently, which doesn't fit this pattern.
**Why it happens:** `study.optimize()` runs trials sequentially by design. For packing, you need the ask/tell API to manage trial lifecycle manually.
**How to avoid:** Use `trial = study.ask()` to get trial params, then `study.tell(trial, value)` or `study.tell(trial, state=TrialState.PRUNED)` to report results. This gives full control over concurrent trial management.
**Warning signs:** Optuna complains about trial states, or trials are double-reported.

### Area Under Learning Curve (ALC) Metric

```python
# Source: standard metric computation
import numpy as np

def compute_alc(losses, steps):
    """Compute area under the learning curve (lower = faster convergence)."""
    return np.trapezoid(losses, steps) / (steps[-1] - steps[0])
```

### Per-Adapter Optimizer Creation

```python
# Source: PyTorch + PEFT pattern
def create_adapter_optimizer(model, adapter_name, lr, weight_decay):
    """Create optimizer for a specific adapter's parameters only."""
    model.set_adapter(adapter_name)
    adapter_params = [
        p for n, p in model.named_parameters()
        if p.requires_grad and adapter_name in n
    ]
    return torch.optim.AdamW(adapter_params, lr=lr, weight_decay=weight_decay)
```

## Open Questions

1. **Does PEFT's gradient isolation actually work for training multiple adapters simultaneously?**
   - What we know: PEFT supports `add_adapter()` and `set_adapter()` for switching active adapters. Inference-time multi-adapter is well-documented.
   - What's unclear: Whether backward passes through non-active adapters remain correctly isolated during training. No official documentation on simultaneous training of multiple adapters.
   - Recommendation: Build a gradient isolation test as the FIRST task: attach 2 adapters, train only adapter A for 10 steps, verify adapter B weights haven't changed.

2. **Actual VRAM cost per additional adapter on T4?**
   - What we know: Phase 2 peaked at ~10 GB with one rank-4 adapter. Theoretical overhead per adapter is ~77 MB + activation memory.
   - What's unclear: Exact activation memory per adapter with gradient checkpointing on this specific model architecture.
   - Recommendation: Empirical profiling task. Load model, add 1-5 adapters, measure `torch.cuda.max_memory_allocated()` after one training step with each count.

3. **Does RsLoRA actually change the optimal rank for Qwen3-ASR?**
   - What we know: Phase 2 found rank=4 optimal with standard scaling. Theory says RsLoRA should help higher ranks.
   - What's unclear: Whether Qwen3-ASR-1.7B (a relatively small model) benefits from higher-rank adapters at all, regardless of scaling.
   - Recommendation: This is what the sweep answers. Not a blocker -- it's the experiment.
