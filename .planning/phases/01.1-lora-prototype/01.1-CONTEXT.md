# Phase 1.1: LoRA Prototype - Context

**Gathered:** 2026-04-05
**Status:** Ready for planning

<domain>
## Phase Boundary

Attach LoRA adapters to Qwen3-ASR-1.7B decoder, validate forward pass produces valid transcriptions within 14GB VRAM budget on RTX A4000, and test adapter save/load/merge lifecycle end-to-end. This is a prototype validation phase — not full training.

</domain>

<decisions>
## Implementation Decisions

### LoRA Configuration
- **D-01:** Attention-only LoRA targeting decoder self-attention projections (q_proj, k_proj, v_proj, o_proj) via regex `thinker\.model\.layers\.\d+\.self_attn\.(q_proj|k_proj|v_proj|o_proj)`
- **D-02:** Rank=16, alpha=32, dropout=0.05 — ~2.5M trainable parameters (~0.15% of total)
- **D-03:** Audio encoder (`thinker.audio_tower`) frozen entirely — no LoRA on encoder modules
- **D-04:** Attention+MLP targeting deferred to Phase 1.2 for empirical benchmarking

### Prototype Data Subset
- **D-05:** Both datasets used: 100 stratified samples from Fair-Speech (ethnicity axis) + 100 stratified samples from Common Voice (accent axis) = 200 total
- **D-06:** Validates both code paths early — Fair-Speech controlled prompts + Common Voice crowd-sourced audio with variable duration
- **D-07:** Subset selection via stratified groupby sampling from Phase 1.0 train splits

### Validation Criteria
- **D-08:** Extend existing validate_test_run.py pattern — issue collection, summary reporting, exit codes
- **D-09:** Transcription quality: WER < 80% threshold (catches garbage output), chatty pattern detection, empty hypothesis checks
- **D-10:** VRAM profiling via `torch.cuda.max_memory_allocated()` — must confirm peak < 14GB
- **D-11:** Adapter round-trip: exact tensor comparison (torch.equal on output tokens) between original adapter and save→load adapter on fixed input batch
- **D-12:** Training sanity: 50-100 steps must show decreasing loss trend

### Training Framework
- **D-13:** HuggingFace Trainer for prototype — mirrors official Qwen3-ASR SFT script (`qwen3_asr_sft.py`)
- **D-14:** Trainer handles gradient checkpointing (`use_reentrant=False`), device placement, logging, and checkpoint management
- **D-15:** Forward patch required (`patch_outer_forward()`) for Trainer compatibility — outer model has no `forward()` by default
- **D-16:** Custom PyTorch training loop deferred to Phase 2 (GRPO requires multi-candidate generation that breaks Trainer paradigm)

### Claude's Discretion
- Exact batch size for prototype training steps (likely 2-4 given VRAM)
- Learning rate and optimizer choice for prototype
- Checkpoint save frequency during 50-100 step run
- Specific samples selected for the 200-sample subset (as long as stratification holds)

</decisions>

- CSV manifests as data contracts between pipeline stages
- Module-level constants for paths (DATA_DIR, RESULTS_DIR)
