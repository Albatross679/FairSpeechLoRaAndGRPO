## LoRA Hyperparameters

| Option | Description | Selected |
|--------|-------------|----------|
| Attention-only, r=16/alpha=32 | ~2.5M trainable params, max VRAM headroom for GRPO later. Research default. Dropout=0.05. | ✓ |
| Attention+MLP, r=8/alpha=16 | Covers all module types at similar param budget to attention-only r=16. Broader adaptation. | |
| Attention+MLP, r=16/alpha=32 | Maximum capacity (~3x params). Tightest VRAM. Best if domain vocabulary adaptation needed. | |

**User's choice:** Attention-only, r=16/alpha=32 (Recommended)
**Notes:** Keeps VRAM comfortable on A4000 and focuses adaptation on decoder generation behavior where fairness bias manifests. Phase 1.2 can benchmark attention+MLP.

## Prototype Data Subset

| Option | Description | Selected |
|--------|-------------|----------|
| Both datasets, 100 each | 200 total samples. Validates both Fair-Speech (ethnicity) and Common Voice (accent) code paths before Phase 1.2 full training. | ✓ |
| Fair-Speech only, 500 samples | Exercises the stratified sampler properly (~70 per ethnicity group). Good match for 50-100 training steps. | |
| Fair-Speech only, 100 samples | Absolute minimum. Fastest iteration. Only validates one dataset path. | |

**User's choice:** Both datasets, 100 each (Recommended)
**Notes:** Catches Common Voice-specific issues (variable audio duration, missing demographics, accent-axis handling) before they become blockers in Phase 1.2.

## Validation Criteria

|--------|-------------|----------|
| Extend existing validate pattern | WER<80% threshold + chatty/empty checks + torch.cuda.max_memory_allocated for VRAM + exact output comparison for adapter round-trip. Matches validate_test_run.py conventions. | ✓ |
| Automated + manual spot-check | Same automated checks plus nvidia-smi polling for total GPU memory and 5-sample manual eyeball. Better VRAM accuracy but not automatable. | |
| You decide | Let Claude pick the validation approach during implementation. | |

**User's choice:** Extend existing validate pattern (Recommended)
**Notes:** Reuses proven codebase patterns. torch.cuda.max_memory_allocated is sufficient since all memory consumers are PyTorch-managed.

## Training Framework

| Option | Description | Selected |
|--------|-------------|----------|
| HF Trainer for prototype | Mirrors official Qwen3-ASR SFT script. Auto grad checkpointing, logging, device placement. Phase 2 will rewrite to custom GRPO loop — bounded rewrite cost. | ✓ |
| Custom PyTorch loop from start | Single paradigm across all phases. More boilerplate now but no Phase 2 rewrite. Full control over forward/backward/generation. | |
| Trainer subclass with custom training_step | Trainer infra + custom loss. Middle ground but GRPO still breaks the paradigm entirely. | |

**User's choice:** HF Trainer for prototype (Recommended)
**Notes:** Official Qwen3-ASR SFT script already uses Trainer. GRPO (Phase 2) cannot use Trainer due to multi-candidate generation with non-standard audio-in model — rewrite cost is bounded.
