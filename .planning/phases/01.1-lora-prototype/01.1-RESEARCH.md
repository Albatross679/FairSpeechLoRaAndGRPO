# Phase 1.1: LoRA Prototype - Research

**Researched:** 2026-04-05
**Domain:** LoRA fine-tuning of Qwen3-ASR-1.7B multimodal speech-language model
**Confidence:** HIGH

## Summary

Qwen3-ASR-1.7B is a multimodal encoder-decoder ASR model with an audio encoder (24 transformer layers, 1024 hidden) and a Qwen3 text decoder (28 transformer layers, 2048 hidden, GQA 16 heads / 8 KV heads). The model is loaded via the `qwen-asr` package (v0.0.6), which registers custom modeling code with HuggingFace AutoModel. The official fine-tuning script (`qwen3_asr_sft.py`) already demonstrates extracting the inner HuggingFace model from the wrapper and training it with the standard HuggingFace Trainer.

The key architectural insight is the model hierarchy: `Qwen3ASRForConditionalGeneration.thinker.audio_tower` (encoder) and `Qwen3ASRForConditionalGeneration.thinker.model` (text decoder) + `thinker.lm_head`. For LoRA, we target only the decoder's attention and MLP projections via PEFT regex patterns, freeze the audio encoder entirely, and use gradient checkpointing to fit within the 14GB VRAM budget. The model at bf16 is ~3.4GB; LoRA rank 16 adds ~2.5M trainable params (~0.15% of total). With gradient checkpointing, peak VRAM should be well under 14GB.

**Primary recommendation:** Use the official fine-tuning script pattern (`qwen_asr.Qwen3ASRModel.from_pretrained()` then extract `.model`), apply PEFT LoRA to decoder-only modules via `target_modules` regex, freeze encoder explicitly, and use HuggingFace Trainer with gradient checkpointing.

## Architecture Patterns

### Model Hierarchy (Critical for LoRA Targeting)

```
Qwen3ASRModel (inference wrapper - NOT a nn.Module)
  .model = Qwen3ASRForConditionalGeneration (outer HF model)
    .thinker = Qwen3ASRThinkerForConditionalGeneration
      .audio_tower = Qwen3ASRAudioEncoder          # 24 layers, FREEZE
      .model = Qwen3ASRThinkerTextModel             # 28 decoder layers, LoRA HERE
        .embed_tokens                                # Token embeddings
        .layers[0..27] = Qwen3ASRThinkerTextDecoderLayer
          .self_attn.q_proj  (2048 -> 2048)          # LoRA target
          .self_attn.k_proj  (2048 -> 1024, GQA)     # LoRA target
          .self_attn.v_proj  (2048 -> 1024, GQA)     # LoRA target
          .self_attn.o_proj  (2048 -> 2048)           # LoRA target
          .mlp.gate_proj     (2048 -> 6144)           # Optional LoRA target
          .mlp.up_proj       (2048 -> 6144)           # Optional LoRA target
          .mlp.down_proj     (6144 -> 2048)           # Optional LoRA target
        .norm = RMSNorm
      .lm_head = nn.Linear(2048, 151936)            # Keep frozen
```

### Pattern 2: Forward Patch (from official SFT script)

**What:** The outer `Qwen3ASRForConditionalGeneration` only has a `generate()` method, not a proper `forward()`. The official script patches it to delegate to `thinker.forward()`.
**When to use:** Required for Trainer compatibility.

```python
# Source: https://github.com/QwenLM/Qwen3-ASR/blob/main/finetuning/qwen3_asr_sft.py
def patch_outer_forward(model):
    cls = model.__class__
    if getattr(cls, "_forward_patched", False):
        return
    if not hasattr(model, "thinker") or not hasattr(model.thinker, "forward"):
        raise RuntimeError("Cannot patch forward: model has no .thinker.forward")
    
    def forward(self, input_ids=None, attention_mask=None,
                input_features=None, feature_attention_mask=None,
                labels=None, **kwargs):
        return self.thinker.forward(
            input_ids=input_ids, attention_mask=attention_mask,
            input_features=input_features,
            feature_attention_mask=feature_attention_mask,
            labels=labels, **kwargs,
        )
    
    cls.forward = forward
    cls._forward_patched = True
```

### Pattern 3: LoRA Configuration (decoder-only targeting)

**What:** Apply LoRA only to decoder text model layers, freeze encoder entirely
**When to use:** For all LoRA training on this model

```python
# Source: PEFT docs + model architecture analysis [VERIFIED: source code inspection]
from peft import LoraConfig, get_peft_model, TaskType

# Freeze encoder explicitly
for param in model.thinker.audio_tower.parameters():
    param.requires_grad = False

lora_config = LoraConfig(
    task_type=TaskType.CAUSAL_LM,
    r=16,
    lora_alpha=32,
    lora_dropout=0.05,
    # Target only decoder attention + MLP projections
    target_modules=[
        r"thinker\.model\.layers\.\d+\.self_attn\.(q_proj|k_proj|v_proj|o_proj)",
        r"thinker\.model\.layers\.\d+\.mlp\.(gate_proj|up_proj|down_proj)",
    ],
    # Alternative simpler approach (targets ALL matching modules including encoder):
    # target_modules=["q_proj", "k_proj", "v_proj", "o_proj"]
    # This would also match encoder modules - use regex to be safe
)

model = get_peft_model(model, lora_config)
model.print_trainable_parameters()
```

### Anti-Patterns to Avoid
- **Loading with transformers 5.x directly:** Qwen3ASR is NOT in transformers 5.x. Must use qwen-asr package. [VERIFIED: `Qwen3ASR` not in `dir(transformers)` with 5.5.0]
- **Targeting encoder modules for LoRA:** The audio encoder should stay frozen. LoRA on encoder would change audio feature space and likely degrade performance.
- **Using `device_map="auto"` with PEFT:** Can cause issues with LoRA on split models. Use `device_map=None` and let Trainer/accelerate handle placement. [ASSUMED]
- **Skipping `patch_outer_forward()`:** Without this patch, the HuggingFace Trainer cannot call `model.forward()` for training loss computation.
- **Using `use_reentrant=True` (default) for gradient checkpointing:** Breaks when parameters are frozen (encoder). Must use `use_reentrant=False`. [CITED: https://github.com/huggingface/transformers/issues/21381]

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| LoRA injection | Custom weight matrix injection | `peft.get_peft_model()` | PEFT handles rank decomposition, scaling, dropout, save/load, merge correctly |
| Gradient checkpointing | Manual activation checkpointing | `model.gradient_checkpointing_enable()` | transformers handles layer-level checkpointing with correct hooks |
| Training loop | Custom training loop | HuggingFace `Trainer` | For prototype phase, Trainer handles device, logging, checkpointing. Custom loop comes in Phase 2 (GRPO) |
| Audio preprocessing | Custom mel spectrogram | `Qwen3ASRProcessor` via qwen-asr | Processor handles mel extraction, tokenization, chat template formatting correctly |
| Data collation | Custom batch collation | `DataCollatorForQwen3ASRFinetuning` pattern from official SFT script | Handles prefix masking, label creation, padding correctly |
| Adapter merging | Manual weight addition | `model.merge_and_unload()` | Handles scaling factor, removes adapters cleanly |

**Key insight:** The official fine-tuning script (`qwen3_asr_sft.py`) already solves most infrastructure problems. Reuse its patterns for model loading, forward patching, data collation, and checkpoint management rather than inventing new approaches.

## Common Pitfalls

### Pitfall 1: transformers Version Conflict
**What goes wrong:** qwen-asr 0.0.6 requires transformers ~4.57. If you have transformers 5.x, the qwen-asr modeling code may break due to changed internal APIs.
**Why it happens:** Qwen3-ASR is not yet merged into transformers main (PR #43837 still open). The qwen-asr package bundles its own modeling code that depends on 4.57 internals.
**How to avoid:** Pin `transformers==4.57.6` explicitly. Do not upgrade.
**Warning signs:** ImportError from `transformers.modeling_layers`, `transformers.masking_utils`, or similar internal modules.
[VERIFIED: source code imports from transformers internals that changed in 5.x]

### Pitfall 2: Missing Forward Patch
**What goes wrong:** `Trainer` calls `model()` which hits `Qwen3ASRForConditionalGeneration.forward()` which does NOT exist by default (only `generate()` is defined).
**Why it happens:** The outer model is a generation-only wrapper; the actual forward with loss is in `thinker.forward()`.
**How to avoid:** Call `patch_outer_forward(model)` before wrapping with PEFT or passing to Trainer.
**Warning signs:** `NotImplementedError` or `AttributeError` during first training step.
[VERIFIED: source code inspection shows no `forward()` in `Qwen3ASRForConditionalGeneration`]

### Pitfall 3: LoRA Targeting Encoder Modules
**What goes wrong:** If `target_modules=["q_proj", "v_proj"]` is used without regex scoping, PEFT will also inject LoRA into encoder attention layers (which also have `q_proj`, `k_proj`, `v_proj`, `out_proj`).
**Why it happens:** PEFT name matching is global across all named modules.
**How to avoid:** Use regex patterns that scope to `thinker.model.layers` (decoder) only, OR freeze encoder before `get_peft_model` (PEFT should skip frozen modules, but explicit regex is safer).
**Warning signs:** `model.print_trainable_parameters()` shows more trainable params than expected; encoder parameters showing `requires_grad=True`.
[VERIFIED: encoder has matching module names `self_attn.q_proj` etc.]

### Pitfall 4: Gradient Checkpointing with Frozen Params
**What goes wrong:** Training crashes or gradients are silently zero.
**Why it happens:** PyTorch's default reentrant checkpoint implementation requires all inputs to have `requires_grad=True`. With frozen encoder, this breaks.
**How to avoid:** Use `gradient_checkpointing_enable(gradient_checkpointing_kwargs={"use_reentrant": False})`.
**Warning signs:** `RuntimeError` about tensors not requiring gradients, or loss that never decreases.
[CITED: https://github.com/huggingface/transformers/issues/21381]

### Pitfall 5: VRAM Spike from Optimizer States
**What goes wrong:** OOM despite model fitting in memory.
**Why it happens:** AdamW stores 2 states per parameter (momentum + variance). Even with LoRA's small param count, the base model activations during forward/backward can spike VRAM.
**How to avoid:** Enable gradient checkpointing, use bf16 training, keep batch size small initially (1-2), verify VRAM with `nvidia-smi` during first steps.
**Warning signs:** CUDA OOM on second training step (first step allocates optimizer).
[ASSUMED]

### Pitfall 6: Data Format for Qwen3-ASR Training
**What goes wrong:** Model produces garbage output or loss doesn't decrease.
**Why it happens:** Qwen3-ASR expects a specific chat template format with language prefix: `"language English<asr_text>actual transcript"`. The data collator must mask the prefix (system + user prompt tokens) so loss is only computed on the transcript.
**How to avoid:** Use the `DataCollatorForQwen3ASRFinetuning` pattern from the official SFT script, which creates labels with `-100` for prefix tokens.
**Warning signs:** Training loss is very low from the start (model is learning to predict the prompt, not the transcript).
[VERIFIED: official SFT script data collator masks prefix tokens with -100]

### Complete LoRA Prototype Setup

```python
# Source: Composite of official SFT script + PEFT docs
# [VERIFIED: model loading pattern from official script, PEFT patterns from docs]

import torch
from qwen_asr import Qwen3ASRModel
from peft import LoraConfig, get_peft_model, TaskType

# Step 1: Load model via qwen-asr wrapper
asr_wrapper = Qwen3ASRModel.from_pretrained(
    "Qwen/Qwen3-ASR-1.7B",
    dtype=torch.bfloat16,
    device_map=None,
)
model = asr_wrapper.model
processor = asr_wrapper.processor

# Step 2: Patch forward for Trainer compatibility
def patch_outer_forward(model):
    cls = model.__class__
    if getattr(cls, "_forward_patched", False):
        return
    def forward(self, input_ids=None, attention_mask=None,
                input_features=None, feature_attention_mask=None,
                labels=None, **kwargs):
        return self.thinker.forward(
            input_ids=input_ids, attention_mask=attention_mask,
            input_features=input_features,
            feature_attention_mask=feature_attention_mask,
            labels=labels, **kwargs,
        )
    cls.forward = forward
    cls._forward_patched = True

patch_outer_forward(model)

# Step 3: Freeze encoder
for param in model.thinker.audio_tower.parameters():
    param.requires_grad = False

# Step 4: Enable gradient checkpointing
model.gradient_checkpointing_enable(
    gradient_checkpointing_kwargs={"use_reentrant": False}
)

# Step 5: Apply LoRA to decoder only
lora_config = LoraConfig(
    task_type=TaskType.CAUSAL_LM,
    r=16,
    lora_alpha=32,
    lora_dropout=0.05,
    target_modules=[
        r"thinker\.model\.layers\.\d+\.self_attn\.(q_proj|k_proj|v_proj|o_proj)",
    ],
)

model = get_peft_model(model, lora_config)
model.print_trainable_parameters()
```

## Open Questions

1. **task_type for PEFT**
   - What we know: The model is encoder-decoder but LoRA targets only the decoder (causal LM)
   - What's unclear: Whether `TaskType.CAUSAL_LM` or `TaskType.SEQ_2_SEQ_LM` is correct for PEFT
   - Recommendation: Start with `TaskType.CAUSAL_LM` since the decoder is autoregressive. If issues arise, try without specifying task_type (PEFT can often auto-detect).

2. **LoRA rank selection for VRAM budget**
   - What we know: r=16 is standard. Model is small (1.7B) so r=16 should be fine.
   - What's unclear: Exact VRAM usage; whether r=32 would also fit.
   - Recommendation: Start with r=16, profile VRAM, increase if budget allows.

3. **Whether PEFT regex matching works cleanly with this custom model class**
   - What we know: PEFT supports regex in `target_modules` for standard HF models.
   - What's unclear: Whether custom model classes from qwen-asr register modules in a way PEFT can discover.
   - Recommendation: Print `model.named_modules()` before applying LoRA to verify module names match regex patterns. Fall back to simpler string matching + manual encoder freeze if regex fails.
