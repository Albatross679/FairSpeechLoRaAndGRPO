# Phase 1.1: LoRA Prototype - Research

**Researched:** 2026-04-05
**Domain:** LoRA fine-tuning of Qwen3-ASR-1.7B multimodal speech-language model
**Confidence:** HIGH

## Summary

Qwen3-ASR-1.7B is a multimodal encoder-decoder ASR model with an audio encoder (24 transformer layers, 1024 hidden) and a Qwen3 text decoder (28 transformer layers, 2048 hidden, GQA 16 heads / 8 KV heads). The model is loaded via the `qwen-asr` package (v0.0.6), which registers custom modeling code with HuggingFace AutoModel. The official fine-tuning script (`qwen3_asr_sft.py`) already demonstrates extracting the inner HuggingFace model from the wrapper and training it with the standard HuggingFace Trainer.

The key architectural insight is the model hierarchy: `Qwen3ASRForConditionalGeneration.thinker.audio_tower` (encoder) and `Qwen3ASRForConditionalGeneration.thinker.model` (text decoder) + `thinker.lm_head`. For LoRA, we target only the decoder's attention and MLP projections via PEFT regex patterns, freeze the audio encoder entirely, and use gradient checkpointing to fit within the 14GB VRAM budget. The model at bf16 is ~3.4GB; LoRA rank 16 adds ~2.5M trainable params (~0.15% of total). With gradient checkpointing, peak VRAM should be well under 14GB.

**Primary recommendation:** Use the official fine-tuning script pattern (`qwen_asr.Qwen3ASRModel.from_pretrained()` then extract `.model`), apply PEFT LoRA to decoder-only modules via `target_modules` regex, freeze encoder explicitly, and use HuggingFace Trainer with gradient checkpointing.

## Architecture Patterns

### Model Hierarchy (Critical for LoRA Targeting)

```
Qwen3ASRModel (inference wrapper - NOT a nn.Module)
  .model = Qwen3ASRForConditionalGeneration (outer HF model)
    .thinker = Qwen3ASRThinkerForConditionalGeneration
      .audio_tower = Qwen3ASRAudioEncoder          # 24 layers, FREEZE
      .model = Qwen3ASRThinkerTextModel             # 28 decoder layers, LoRA HERE

        .layers[0..27] = Qwen3ASRThinkerTextDecoderLayer

          .mlp.down_proj     (6144 -> 2048)           # Optional LoRA target
        .norm = RMSNorm
      .lm_head = nn.Linear(2048, 151936)            # Keep frozen
```

### Pattern 2: Forward Patch (from official SFT script)

**What:** The outer `Qwen3ASRForConditionalGeneration` only has a `generate()` method, not a proper `forward()`. The official script patches it to delegate to `thinker.forward()`.
**When to use:** Required for Trainer compatibility.

```python
# Source: https://github.com/QwenLM/Qwen3-ASR/blob/main/finetuning/qwen3_asr_sft.py
def patch_outer_forward(model):
    cls = model.__class__
    if getattr(cls, "_forward_patched", False):
        return
    if not hasattr(model, "thinker") or not hasattr(model.thinker, "forward"):
        raise RuntimeError("Cannot patch forward: model has no .thinker.forward")
    
    def forward(self, input_ids=None, attention_mask=None,
                input_features=None, feature_attention_mask=None,
                labels=None, **kwargs):
        return self.thinker.forward(
            input_ids=input_ids, attention_mask=attention_mask,
            input_features=input_features,
            feature_attention_mask=feature_attention_mask,
            labels=labels, **kwargs,
        )
    
    cls.forward = forward
    cls._forward_patched = True
```

### Pattern 3: LoRA Configuration (decoder-only targeting)

**What:** Apply LoRA only to decoder text model layers, freeze encoder entirely
**When to use:** For all LoRA training on this model

```python
# Source: PEFT docs + model architecture analysis [VERIFIED: source code inspection]
from peft import LoraConfig, get_peft_model, TaskType

# Freeze encoder explicitly
for param in model.thinker.audio_tower.parameters():
    param.requires_grad = False

lora_config = LoraConfig(
    task_type=TaskType.CAUSAL_LM,
    r=16,
    lora_alpha=32,
    lora_dropout=0.05,
    # Target only decoder attention + MLP projections
    target_modules=[
        r"thinker\.model\.layers\.\d+\.self_attn\.(q_proj|k_proj|v_proj|o_proj)",
        r"thinker\.model\.layers\.\d+\.mlp\.(gate_proj|up_proj|down_proj)",
    ],
    # Alternative simpler approach (targets ALL matching modules including encoder):
    # target_modules=["q_proj", "k_proj", "v_proj", "o_proj"]
    # This would also match encoder modules - use regex to be safe
)

model = get_peft_model(model, lora_config)
model.print_trainable_parameters()
```

### Anti-Patterns to Avoid
- **Loading with transformers 5.x directly:** Qwen3ASR is NOT in transformers 5.x. Must use qwen-asr package. [VERIFIED: `Qwen3ASR` not in `dir(transformers)` with 5.5.0]
- **Targeting encoder modules for LoRA:** The audio encoder should stay frozen. LoRA on encoder would change audio feature space and likely degrade performance.
- **Using `device_map="auto"` with PEFT:** Can cause issues with LoRA on split models. Use `device_map=None` and let Trainer/accelerate handle placement. [ASSUMED]
- **Skipping `patch_outer_forward()`:** Without this patch, the HuggingFace Trainer cannot call `model.forward()` for training loss computation.
- **Using `use_reentrant=True` (default) for gradient checkpointing:** Breaks when parameters are frozen (encoder). Must use `use_reentrant=False`. [CITED: https://github.com/huggingface/transformers/issues/21381]

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| LoRA injection | Custom weight matrix injection | `peft.get_peft_model()` | PEFT handles rank decomposition, scaling, dropout, save/load, merge correctly |
| Gradient checkpointing | Manual activation checkpointing | `model.gradient_checkpointing_enable()` | transformers handles layer-level checkpointing with correct hooks |
| Training loop | Custom training loop | HuggingFace `Trainer` | For prototype phase, Trainer handles device, logging, checkpointing. Custom loop comes in Phase 2 (GRPO) |
| Audio preprocessing | Custom mel spectrogram | `Qwen3ASRProcessor` via qwen-asr | Processor handles mel extraction, tokenization, chat template formatting correctly |
| Data collation | Custom batch collation | `DataCollatorForQwen3ASRFinetuning` pattern from official SFT script | Handles prefix masking, label creation, padding correctly |
| Adapter merging | Manual weight addition | `model.merge_and_unload()` | Handles scaling factor, removes adapters cleanly |

**Key insight:** The official fine-tuning script (`qwen3_asr_sft.py`) already solves most infrastructure problems. Reuse its patterns for model loading, forward patching, data collation, and checkpoint management rather than inventing new approaches.

## Common Pitfalls

### Pitfall 1: transformers Version Conflict
**What goes wrong:** qwen-asr 0.0.6 requires transformers ~4.57. If you have transformers 5.x, the qwen-asr modeling code may break due to changed internal APIs.
**Why it happens:** Qwen3-ASR is not yet merged into transformers main (PR #43837 still open). The qwen-asr package bundles its own modeling code that depends on 4.57 internals.
**How to avoid:** Pin `transformers==4.57.6` explicitly. Do not upgrade.
**Warning signs:** ImportError from `transformers.modeling_layers`, `transformers.masking_utils`, or similar internal modules.
[VERIFIED: source code imports from transformers internals that changed in 5.x]

### Pitfall 2: Missing Forward Patch
**What goes wrong:** `Trainer` calls `model()` which hits `Qwen3ASRForConditionalGeneration.forward()` which does NOT exist by default (only `generate()` is defined).
**Why it happens:** The outer model is a generation-only wrapper; the actual forward with loss is in `thinker.forward()`.
**How to avoid:** Call `patch_outer_forward(model)` before wrapping with PEFT or passing to Trainer.
**Warning signs:** `NotImplementedError` or `AttributeError` during first training step.
[VERIFIED: source code inspection shows no `forward()` in `Qwen3ASRForConditionalGeneration`]

### Pitfall 3: LoRA Targeting Encoder Modules
**What goes wrong:** If `target_modules=["q_proj", "v_proj"]` is used without regex scoping, PEFT will also inject LoRA into encoder attention layers (which also have `q_proj`, `k_proj`, `v_proj`, `out_proj`).
**Why it happens:** PEFT name matching is global across all named modules.
**How to avoid:** Use regex patterns that scope to `thinker.model.layers` (decoder) only, OR freeze encoder before `get_peft_model` (PEFT should skip frozen modules, but explicit regex is safer).
**Warning signs:** `model.print_trainable_parameters()` shows more trainable params than expected; encoder parameters showing `requires_grad=True`.
[VERIFIED: encoder has matching module names `self_attn.q_proj` etc.]

### Pitfall 4: Gradient Checkpointing with Frozen Params
**What goes wrong:** Training crashes or gradients are silently zero.
**Why it happens:** PyTorch's default reentrant checkpoint implementation requires all inputs to have `requires_grad=True`. With frozen encoder, this breaks.
**How to avoid:** Use `gradient_checkpointing_enable(gradient_checkpointing_kwargs={"use_reentrant": False})`.
**Warning signs:** `RuntimeError` about tensors not requiring gradients, or loss that never decreases.
[CITED: https://github.com/huggingface/transformers/issues/21381]

### Pitfall 5: VRAM Spike from Optimizer States
**What goes wrong:** OOM despite model fitting in memory.
**Why it happens:** AdamW stores 2 states per parameter (momentum + variance). Even with LoRA's small param count, the base model activations during forward/backward can spike VRAM.
**How to avoid:** Enable gradient checkpointing, use bf16 training, keep batch size small initially (1-2), verify VRAM with `nvidia-smi` during first steps.
**Warning signs:** CUDA OOM on second training step (first step allocates optimizer).
[ASSUMED]

### Pitfall 6: Data Format for Qwen3-ASR Training
**What goes wrong:** Model produces garbage output or loss doesn't decrease.
**Why it happens:** Qwen3-ASR expects a specific chat template format with language prefix: `"language English<asr_text>actual transcript"`. The data collator must mask the prefix (system + user prompt tokens) so loss is only computed on the transcript.
**How to avoid:** Use the `DataCollatorForQwen3ASRFinetuning` pattern from the official SFT script, which creates labels with `-100` for prefix tokens.
**Warning signs:** Training loss is very low from the start (model is learning to predict the prompt, not the transcript).
[VERIFIED: official SFT script data collator masks prefix tokens with -100]

## Open Questions

1. **task_type for PEFT**
   - What we know: The model is encoder-decoder but LoRA targets only the decoder (causal LM)
   - What's unclear: Whether `TaskType.CAUSAL_LM` or `TaskType.SEQ_2_SEQ_LM` is correct for PEFT
   - Recommendation: Start with `TaskType.CAUSAL_LM` since the decoder is autoregressive. If issues arise, try without specifying task_type (PEFT can often auto-detect).

2. **LoRA rank selection for VRAM budget**
   - What we know: r=16 is standard. Model is small (1.7B) so r=16 should be fine.
   - What's unclear: Exact VRAM usage; whether r=32 would also fit.
   - Recommendation: Start with r=16, profile VRAM, increase if budget allows.

3. **Whether PEFT regex matching works cleanly with this custom model class**
   - What we know: PEFT supports regex in `target_modules` for standard HF models.
   - What's unclear: Whether custom model classes from qwen-asr register modules in a way PEFT can discover.
   - Recommendation: Print `model.named_modules()` before applying LoRA to verify module names match regex patterns. Fall back to simpler string matching + manual encoder freeze if regex fails.

## ML Prototyping Methodology

**Supplementary research added:** 2026-04-05
**Domain:** ML training pipeline prototyping best practices
**Confidence:** HIGH

This section documents the established methodology for prototyping a machine learning training pipeline before scaling up. The core principle, articulated by Andrej Karpathy and widely adopted in the ML engineering community, is: **build confidence incrementally by validating the simplest possible thing first, then adding complexity one layer at a time.** Neural networks fail silently -- a misconfigured network will train and produce numbers without raising exceptions, but the results will be garbage. The validation ladder below is designed to catch these silent failures early.

[CITED: http://karpathy.github.io/2019/04/25/recipe/]
[CITED: https://fullstackdeeplearning.com/spring2021/lecture-7/]

### Validation Ladder

Execute these steps in strict order. Do NOT skip ahead. Each step has a go/no-go gate -- if the gate fails, stop and debug before proceeding.

#### Rung 0: Bare Model Smoke Test (no LoRA, no training)

**What:** Load the base Qwen3-ASR-1.7B model, run inference on 1-2 audio samples, verify it produces coherent English transcriptions.

**Why:** Establishes that the model loads correctly, the processor works, and the base model is functional before any modifications.

**Go/no-go gate:**
- Model loads without errors
- `model.generate()` produces recognizable English text (not garbage tokens, not empty)
- VRAM usage after load is in expected range (~3.4GB for bf16)

**If this fails:** The problem is model loading, environment, or CUDA -- not LoRA. Debug here before adding any complexity.

#### Rung 1: Forward Pass with LoRA (no training)

**What:** Apply LoRA adapters, patch forward, run a single forward pass with a preprocessed input. Verify output shape and loss value.

**Why:** Confirms that PEFT injection worked, the forward patch is functional, and the model produces a valid loss. Since LoRA B matrices are initialized to zeros, the initial output should be identical to the base model.

**Go/no-go gate:**
- `model.print_trainable_parameters()` shows expected count (~2.5M for r=16 on attention-only, more if MLP included)
- Forward pass returns a loss (not NaN, not None)
- Output logits have correct shape: `(batch_size, seq_len, vocab_size=151936)`
- VRAM after forward is within budget (<14GB)
- Initial loss is reasonable (cross-entropy on vocab of 151936 means random would be ~ln(151936) ~= 11.9; pretrained model should produce loss ~1-4 range on valid ASR data)

**If this fails:** Check forward patch, LoRA target_modules regex (print `model.named_modules()` to verify), encoder freeze status, and data formatting.

#### Rung 2: Single Backward Pass + Gradient Flow Verification

**What:** Run forward + backward on one batch. Verify gradients exist on LoRA parameters and are zero on frozen parameters.

**Why:** A forward pass succeeding does not mean gradients flow correctly. Gradient checkpointing with frozen params is a known failure mode (Pitfall 4). This step catches it.

**Go/no-go gate:**
- `loss.backward()` completes without error
- LoRA A and B matrices have non-zero `.grad` values
- Encoder parameters have no gradients (`.grad is None`)
- No gradient explosion (grad norms < 100)
- VRAM after backward is within budget (this is typically the peak)

**If this fails:** Most likely `use_reentrant=True` is being used for gradient checkpointing. Switch to `use_reentrant=False`. Also verify that `requires_grad` is correctly set on all parameters.

#### Rung 3: Overfit One Batch (2-5 examples)

**What:** Pick 2-5 short audio samples with known transcriptions. Train for 50-200 steps on ONLY these examples (no shuffling, no data augmentation). The model must memorize them perfectly.

**Why:** This is the single most powerful debugging technique in ML. If the model cannot memorize 2-5 examples, something is fundamentally broken in the training loop, data pipeline, or loss computation. [CITED: http://karpathy.github.io/2019/04/25/recipe/]

**Go/no-go gate:**
- Training loss decreases monotonically toward near-zero (~0.01 or lower)
- After training, `model.generate()` on those same inputs produces exact (or near-exact) transcriptions
- Loss curve is smooth (no oscillation, no plateaus after first few steps)

**If this fails, debug in this order:**
1. **Loss goes UP:** Sign flip in loss function or gradient. Check that labels are formatted correctly (prefix masked with -100).
2. **Loss explodes (NaN/Inf):** Learning rate too high (start with 1e-5 for LoRA), or numerical instability in bf16.
3. **Loss oscillates:** Learning rate too high, or data pipeline is corrupting inputs between steps.
4. **Loss plateaus (does not decrease):** Gradients are not flowing to LoRA params (check Rung 2 again), or learning rate is too low, or loss is computed on wrong tokens.
5. **Loss decreases but model generates garbage:** Data collator is not masking prefix correctly -- loss was being computed on predicting the prompt tokens, not the transcript.

**Recommended hyperparameters for overfit test:**
- Learning rate: 1e-4 (aggressive, fine for memorization)
- Batch size: 1-2
- No warmup, no weight decay, no dropout (set `lora_dropout=0.0` temporarily)
- AdamW optimizer (Trainer default)

#### Rung 4: Small Subset Training (50-100 examples, 50-100 steps)

**What:** Train on a small but representative subset of real data. Verify that loss decreases and the model produces plausible (not necessarily perfect) transcriptions.

**Why:** Validates that the pipeline works beyond memorization. At this scale you can catch data pipeline issues (wrong audio paths, mismatched transcriptions, truncation) that do not show up with 2-5 hand-picked examples.

**Go/no-go gate:**
- Training loss shows a clear downward trend over 50-100 steps
- Loss does not diverge or plateau immediately
- Generated transcriptions are recognizably English (even if WER is high)
- VRAM usage is stable (no memory leak across steps)

**If this fails:** Likely a data pipeline issue. Check that audio files are being loaded and resampled correctly, transcriptions match audio, and the data collator is producing correct label shapes.

#### Rung 5: VRAM Budget Confirmation

**What:** Profile peak VRAM during a training step at the target batch size (likely batch_size=1 or 2 for this project).

**Why:** INFRA-03 requires peak VRAM < 14GB. VRAM spikes on the second training step when the optimizer allocates momentum and variance buffers.

**Go/no-go gate:**
- Peak VRAM (from `torch.cuda.max_memory_allocated()`) is < 14GB
- `nvidia-smi` confirms no CUDA OOM risk (reserved < 15GB, leaving 1GB headroom)
- VRAM is stable across steps 1-10 (no leak)

**If VRAM exceeds budget:**
1. Reduce batch size to 1
2. Reduce LoRA rank from 16 to 8
3. Remove MLP modules from LoRA targets (keep attention-only)
4. Verify gradient checkpointing is actually active (check Pitfall 4)
5. Last resort: Use 8-bit AdamW via `bitsandbytes` (`adamw_bnb_8bit`)

#### Rung 6: Save/Load/Merge Lifecycle

**What:** Save the adapter after a few training steps. Load it onto a fresh base model. Verify outputs match. Merge and verify outputs still match.

**Why:** INFRA-05 requirement. If save/load is broken, all training work is wasted.

**Go/no-go gate:**
- `model.save_pretrained()` creates adapter files (adapter_config.json, adapter_model.safetensors)
- Loading onto fresh base model produces identical outputs to pre-save model
- `merge_and_unload()` produces identical outputs to the adapter-loaded model
- Adapter files are small (~10MB, not gigabytes -- if gigabytes, full model was saved by mistake)

### Common Prototype Traps

Things that work at small scale during prototyping but break when scaling up. The planner should ensure verification steps catch these before Phase 2 (GRPO) begins.

#### Trap 1: Gradient Checkpointing Appears to Work but Does Not Reduce Memory

**What happens:** `model.is_gradient_checkpointing` returns True, training runs without errors, but VRAM usage is identical to non-checkpointed training.

**Why it happens at prototype scale:** With batch_size=1 on short audio, VRAM may fit either way, so the bug is invisible. At larger batch sizes or longer sequences, OOM appears.

**How to detect:** Compare `torch.cuda.max_memory_allocated()` with and without gradient checkpointing on the same input. Checkpointing should reduce peak by 30-60%. If the difference is < 5%, checkpointing is not actually active.

**Root cause:** Known PEFT/transformers interaction bug (GitHub issue #42947). The gradient checkpointing wrapper may not propagate correctly through PEFT's adapter layers.

**Fix:** If not working, try applying gradient checkpointing BEFORE `get_peft_model()`, or use `model.base_model.gradient_checkpointing_enable()` after PEFT wrapping.

[CITED: https://github.com/huggingface/transformers/issues/42947]
[CITED: https://github.com/huggingface/peft/issues/2826]

#### Trap 2: Loss Decreases but Model Output Quality Does Not Improve

**What happens:** Training loss goes down, but `model.generate()` produces the same text as before training (or worse text).

**Why:** Two possible causes:
1. **Loss is on wrong tokens.** The data collator is not masking prefix tokens with -100, so the loss includes predicting the system prompt and audio tokens. The model "improves" at predicting the prompt but not the transcript.
2. **LoRA alpha/r ratio is too low.** The effective LoRA update is `alpha/r * (B @ A)`. If alpha=16 and r=16 (ratio=1), updates may be too small to affect generation at inference scale.

**How to detect:** After overfitting 2-5 examples, run `model.generate()` on those inputs. If the transcription does not closely match the reference, the loss is misleading.

**Fix for (1):** Inspect labels tensor -- positions with -100 should cover all prompt tokens, only transcript tokens should have real IDs. Fix for (2):** Use alpha = 2*r as a starting point (alpha=32 for r=16).

#### Trap 3: VRAM Fits at batch_size=1 but OOM at batch_size=2+

**What happens:** Prototype works at batch_size=1. Scaling to batch_size=2 for better throughput causes OOM.

**Why:** Activations scale linearly with batch size, but the jump from 1 to 2 doubles activation memory while model weights stay constant. On a 16GB card, if model+optimizer takes 8GB and activations take 5GB per sample, batch_size=2 needs 18GB.

**How to detect:** Profile VRAM at batch_size=1 and batch_size=2 during the prototype phase. Record both values.

**Fix:** For this project (RTX A4000, 16GB), batch_size=1 with gradient accumulation steps may be necessary. Plan for `gradient_accumulation_steps=4` to get effective batch_size=4 without the VRAM cost.

#### Trap 4: Training Succeeds but Adapter Does Not Load on Different Machine/Session

**What happens:** Adapter saves and loads in the same Python session, but fails to load in a new session or on a different machine.

**Why:** The PEFT adapter config stores the base model ID and class name. If the qwen-asr package is not installed on the loading machine, or is a different version, `PeftModel.from_pretrained()` fails because it cannot instantiate the base model class.

**How to detect:** Test the full save/load cycle in a separate Python process (not just the same script). The Rung 6 validation should use `subprocess` or a separate script to verify.

**Fix:** Ensure the adapter directory includes all necessary metadata. Pin qwen-asr version in requirements.

#### Trap 5: Overfit Test Passes but Training on Full Data Diverges

**What happens:** Model perfectly memorizes 5 examples at lr=1e-4, but training on the full dataset at the same learning rate causes loss to spike or oscillate.

**Why:** The learning rate that works for memorization (overfit test) is typically 10-100x too high for generalization training. The overfit test deliberately uses an aggressive LR to test the pipeline, not to find the right LR.

**How to detect:** This is expected behavior, not a bug. The prototype phase should document the working LR range but expect to reduce it for full training.

**Fix:** Use lr=1e-5 to 5e-5 for actual training (Phase 1.2). Apply warmup (5-10% of steps). The overfit test LR (1e-4) is for debugging only.

[CITED: https://introl.com/blog/fine-tuning-infrastructure-lora-qlora-peft-scale-guide-2025]

#### Check 1: Verify LoRA Targets Match Intended Modules

**What:** After `get_peft_model()`, enumerate all adapted modules and confirm they are decoder-only.

**Why:** If regex is wrong, LoRA could silently target encoder modules, waste VRAM, and corrupt audio features. [VERIFIED: this is the Pitfall 3 scenario]

#### Check 2: Verify Initial Output Equals Base Model

**What:** Before any training, run `model.generate()` on the same input with and without LoRA. Outputs must be identical.

**Why:** PEFT initializes LoRA B to zeros, so `B @ A = 0` and the adapter is an identity transform. If outputs differ, PEFT initialization is wrong or something else changed (e.g., dropout is active during inference).

```python
# Before training, compare:
model.eval()  # Disable dropout
with torch.no_grad():
    output_with_lora = model.generate(**inputs)
# Temporarily disable adapter:
model.disable_adapter_layers()
with torch.no_grad():
    output_without_lora = model.generate(**inputs)
model.enable_adapter_layers()
assert torch.equal(output_with_lora, output_without_lora), "LoRA changed output before training!"
```

[CITED: https://huggingface.co/docs/peft/main/en/conceptual_guides/lora]

#### Check 4: Verify LoRA Weights Changed After Training

**What:** After the overfit test (Rung 3), confirm that LoRA B matrices are no longer zero.

```python
for name, param in model.named_parameters():
```

#### Gradient Flow Verification Script

```python
# Source: Standard PyTorch debugging + PEFT docs
# [CITED: https://apxml.com/courses/lora-peft-efficient-llm-training/chapter-5-peft-optimization-deployment/debugging-peft-implementations]

def verify_gradient_flow(model, batch):
    """Run one forward+backward and verify gradient flow is correct."""
    model.train()
    model.zero_grad()
    
    outputs = model(**batch)
    loss = outputs.loss
    print(f"Loss value: {loss.item():.4f}")
    
    loss.backward()
    
    # Check 1: LoRA params have gradients
    lora_grads = {}
    for name, param in model.named_parameters():
        if "lora_" in name and param.requires_grad:
            if param.grad is not None:
                grad_norm = param.grad.norm().item()
                lora_grads[name] = grad_norm
            else:
                print(f"FAIL: {name} has requires_grad=True but grad is None")
    
    if lora_grads:
        print(f"\nLoRA gradient norms ({len(lora_grads)} params):")
        norms = list(lora_grads.values())
        print(f"  min={min(norms):.6f}, max={max(norms):.6f}, mean={sum(norms)/len(norms):.6f}")
    
    # Check 2: Encoder has no gradients
    encoder_grad_count = 0
    for name, param in model.named_parameters():
        if "audio_tower" in name and param.grad is not None:
            encoder_grad_count += 1
            print(f"FAIL: Encoder param {name} received gradient!")
    
    if encoder_grad_count == 0:
        print("OK: Encoder received no gradients (frozen correctly)")
    
    # Check 3: VRAM after backward
    peak_gb = torch.cuda.max_memory_allocated() / 1024**3
    print(f"\nPeak VRAM after backward: {peak_gb:.2f}GB")
    
    return loss.item(), lora_grads
```

#### VRAM Profiling at Key Checkpoints

```python
# Source: PyTorch CUDA memory docs
# [CITED: https://docs.pytorch.org/docs/stable/torch_cuda_memory.html]

import torch

def vram_checkpoint(label):
    """Log VRAM at a key point. Call at each validation ladder rung."""
    if not torch.cuda.is_available():
        return
    torch.cuda.synchronize()
    allocated = torch.cuda.memory_allocated() / 1024**3
    reserved = torch.cuda.memory_reserved() / 1024**3
    peak = torch.cuda.max_memory_allocated() / 1024**3
    print(f"[VRAM {label}] allocated={allocated:.2f}GB  reserved={reserved:.2f}GB  peak={peak:.2f}GB")

# Usage throughout prototype:
vram_checkpoint("after model load")        # Rung 0: expect ~3.4GB
vram_checkpoint("after PEFT wrap")         # Rung 1: expect ~3.5GB (minimal overhead)
vram_checkpoint("after first forward")     # Rung 1: expect ~5-7GB
vram_checkpoint("after first backward")    # Rung 2: expect ~8-12GB (peak)
vram_checkpoint("after optimizer step")    # Rung 2: expect ~8-12GB (optimizer states allocated)

# Reset peak tracking between rungs:
torch.cuda.reset_peak_memory_stats()
```
