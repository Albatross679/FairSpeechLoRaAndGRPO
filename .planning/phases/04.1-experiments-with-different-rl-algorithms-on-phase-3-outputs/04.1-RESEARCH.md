## Summary

Phase 04.1 answers the user's three explicit questions:

1. **How to design the experiments?** — Hold the base model, SFT init, dataset subset, reward function, reference model, generation hyperparameters, evaluation protocol, and seeds constant. Vary **only** the RL loss / advantage computation. Run ≥ 3 seeds per algorithm on the same 2 K subset used in Phase 4, report Interquartile Mean (IQM) with bootstrap 95 % CIs over seeds, and anchor everything to GRPO (the Phase 4 reference) so the table is always "GRPO vs X" — never a free-for-all.
2. **How to make conditions consistent?** — Extend the Phase 4 `grpo_step()` into a dispatcher over swappable *loss modules* that share generation, log-prob computation, reference-model handling (PEFT `disable_adapter_layers()`), reward computation, and KL coefficient. Keep ε (clip), β (KL), lr, temperature, G, batch size, gradient accumulation, optimizer, and sample budget identical across algorithms. Only the advantage/loss math changes. Tune learning rate per algorithm on a 3-point grid (1e-6, 1e-5, 3e-5) because lr is the one HP where "same value" is unfair [CITED: Ahmadian 2024 arxiv:2402.14740; Colas et al. 2018 arxiv:1806.08295].
3. **Which algorithms?** — Four on-policy algorithms that all fit in T4 VRAM, all use a scalar-per-sequence reward, all reuse the existing generation loop, and collectively cover the "family tree" of critic-free RLHF:
   - **GRPO (Dr. GRPO variant)** — the Phase 4 reference baseline. [CITED: Liu et al. 2025 arxiv:2503.20783]
   - **RLOO** — leave-one-out baseline, the "clean" critic-free REINFORCE. [CITED: Ahmadian et al. 2024 arxiv:2402.14740]
   - **REINFORCE++** — global batch-level advantage normalization + PPO clip, no per-prompt group. [CITED: Hu et al. 2025 arxiv:2501.03262]
   - **PPO-LoRA (no separate critic)** — standard PPO ratio loss with *group-mean baseline* as value proxy (still critic-free), to isolate "what PPO clipping + KL penalty buys over GRPO's group-relative trick." [CITED: Huang et al. 2022 ICLR Blog Track — "The 37 Implementation Details of PPO"]

**DPO / KTO are NOT recommended for inclusion.** Both require a different data pipeline (offline preferences or binary desirable labels) that would force converting our scalar composite reward into lossy preference pairs, and KTOTrainer is marked experimental in TRL v1.0 [CITED: TRL KTO docs] while neither trainer supports Qwen3-ASR's audio encoder. If the user insists on a preference-style baseline, we can generate K samples per utterance offline from the SFT checkpoint and use `best vs worst` pairs — but this is documented below as a **discretionary extension**, not a core run.

**Primary recommendation:** Refactor `train_grpo.py` into an RL dispatcher (`train_rl.py`) with a `--algorithm {dr_grpo,rloo,reinforce++,ppo}` flag and a `LossModule` ABC. Run 4 algorithms × 3 seeds × 1 λ (0.3) × 200 steps = 12 runs (≈ 8–16 hours wall-clock on T4). Then one additional sweep of the winner across 3 λ values (0, 0.3, 0.7) for completeness. Report: reward curves (mean + IQM over seeds), final dev/test WER, final fairness gap, VRAM peak, wall-clock, trainable params, and bootstrap 95 % CI pairwise against GRPO.

### Pattern 1: LossModule ABC (the core refactor)

**What:** A tiny abstract base class that encapsulates the only thing that varies across RL algorithms — how rewards become advantages and how advantages become a loss.

**When to use:** Every new algorithm added to Phase 04.1. The dispatcher (`train_rl.py`) calls `loss_module.compute_loss(...)` and everything else stays identical.

**Example:**

```python
# scripts/training/rl_losses/base.py
# Source: Refactor of Phase 4 grpo_step() (scripts/training/train_grpo.py:336-500)
# [VERIFIED: existing grpo_step() shape in Phase 4 code]
from abc import ABC, abstractmethod
from dataclasses import dataclass
import torch

@dataclass
class LossInputs:
    rewards: torch.Tensor          # (batch_size, G) — from RewardComputer, HELD CONSTANT
    old_log_probs: list            # length G, each (batch_size, gen_len) — HELD CONSTANT
    current_log_probs: list        # length G, each (batch_size, gen_len) — HELD CONSTANT
    ref_log_probs: list | None     # length G, each (batch_size, gen_len) — HELD CONSTANT (beta > 0)
    response_masks: list           # length G, each (batch_size, gen_len) — HELD CONSTANT
    config: "RLConfig"             # same config dataclass across all algorithms

class LossModule(ABC):
    """Abstract loss module for RL algorithm comparison.

    The contract is intentionally narrow: rewards and log-probs are computed
    identically for every algorithm (shared generation, shared RewardComputer,
    shared reference-model handling via disable_adapter_layers). The only
    degree of freedom is how advantages are computed from rewards and how the
    loss is computed from advantages + ratios.
    """

    name: str  # e.g. "dr_grpo", "rloo", "reinforce_pp", "ppo_groupmean"

    @abstractmethod
    def compute_advantages(self, rewards: torch.Tensor) -> torch.Tensor:
        """Return (batch_size, G) advantage tensor. Must not touch log-probs."""

    @abstractmethod
    def compute_loss(self, inputs: LossInputs, advantages: torch.Tensor) -> tuple[torch.Tensor, dict]:
        """Return (scalar loss, diagnostic dict).

        Diagnostic dict MUST contain (for every algorithm, to make logs comparable):
          - 'loss' : float
          - 'advantage_mean', 'advantage_std' : floats
          - 'ratio_mean', 'ratio_max', 'clip_frac' : floats (ratio=1.0, clip_frac=0.0 if not applicable)
          - 'kl' : float (or 0.0 if beta == 0)
          - 'entropy' : float
        """
```

Every algorithm subclasses this. The planner writes one ~50-line file per algorithm.

### Pattern 2: train_rl.py dispatcher (drop-in replacement for train_grpo.py)

**What:** A single training script that loads the model once per run and calls the appropriate `LossModule` based on `--algorithm`.

**When to use:** For every Phase 04.1 run. Do not copy-paste `train_grpo.py` four times.

**Example:**

```python
# scripts/training/train_rl.py (sketch)
# Source: Refactor of Phase 4 train_grpo.py train_grpo() function.
# [VERIFIED: current train_grpo.grpo_step() at scripts/training/train_grpo.py:336]

from rl_losses import DrGRPOLoss, RLOOLoss, ReinforcePPLoss, PPOGroupMeanLoss

LOSS_REGISTRY = {
    "dr_grpo":      DrGRPOLoss,
    "rloo":         RLOOLoss,
    "reinforce_pp": ReinforcePPLoss,
    "ppo_groupmean": PPOGroupMeanLoss,
}

def rl_step(model, processor, batch, config, reward_computer, loss_module, device):
    # Steps 1-4 are IDENTICAL to Phase 4 grpo_step():
    #   - generate_candidates() -> candidates, generated_ids          [HELD CONSTANT]
    #   - reward_computer(candidates, references, demographics)       [HELD CONSTANT]
    #   - compute old_log_probs via teacher-forcing, no grad          [HELD CONSTANT]
    #   - compute ref_log_probs via disable_adapter_layers, if beta>0 [HELD CONSTANT]
    # Step 5 delegates to loss_module (the ONLY thing that varies):
    advantages = loss_module.compute_advantages(rewards)
    current_log_probs = [compute_per_token_log_probs(model, ..., generated_ids[g]) for g in range(config.G)]
    inputs = LossInputs(rewards, old_log_probs, current_log_probs, ref_log_probs, masks, config)
    loss, metrics = loss_module.compute_loss(inputs, advantages)
    return loss, metrics

def main():
    args = parse_args()
    loss_module = LOSS_REGISTRY[args.algorithm]()
    # Seed ALL RNGs deterministically from args.seed BEFORE model load
    torch.manual_seed(args.seed); np.random.seed(args.seed); random.seed(args.seed)
    torch.cuda.manual_seed_all(args.seed)
    # Reload the Phase 3 SFT adapter fresh for every run
    model = load_phase3_sft_adapter(args.sft_adapter_path)
    # ... same training loop as Phase 4 ...
```

### Pattern 4: Shared reward-hacking detector

**What:** A post-run script that detects reward hacking signals (per-group WER floor breach, mean output length explosion, vocabulary collapse, entropy collapse) across all algorithms uniformly.

**When to use:** After every run, before aggregating metrics. A comparison is only fair if all four algorithms are subjected to the same pathology detector.

**Example:**

```python
# scripts/analysis/detect_reward_hacking.py
# Source: Pitfall 2 (fairness hacking) + Pitfall 8 (entropy collapse) from .planning/research/PITFALLS.md
# [CITED: .planning/research/PITFALLS.md §2, §8]

def detect_reward_hacking(training_log: list, baseline_wer: float) -> dict:
    return {
        "wer_floor_breach": any(e["wer/mean"] > baseline_wer * 1.2 for e in training_log[-10:]),
        "length_explosion": training_log[-1]["completions/mean_length"] > 2.0 * training_log[0]["completions/mean_length"],
        "entropy_collapse": training_log[-1].get("entropy", 1.0) < 0.3 * training_log[0].get("entropy", 1.0),
        "zero_std_fraction_end": sum(e["reward/frac_zero_std"] for e in training_log[-10:]) / 10,
        "verdict": "OK" | "DEGRADED" | "HACKED",
    }
```

### Anti-Patterns to Avoid

- **Copy-paste `train_grpo.py` four times** — introduces drift across algorithms, breaks comparability. Use the dispatcher.
- **Installing TRL just for RLOO** — TRL RLOO is for causal LMs only, will not work on Qwen3-ASR. You would end up building glue code equivalent to rewriting RLOO in ~40 lines anyway.
- **Per-algorithm different KL coefficient** — β silently dominates behavior; holding it fixed is the whole point.
- **Per-algorithm different temperature or G** — changes advantage variance and generation diversity, muddies comparison.
- **Reporting single-seed numbers as "X beats Y"** — single seeds are noise [CITED: Henderson et al. 2018 arxiv:1709.06560; Colas et al. 2018 arxiv:1806.08295]. Minimum 3 seeds.
- **Using different dataset subsets per algorithm** — even sampling from the same manifest with a different seed creates tiny distributional differences. Dump the subset to CSV once and reuse the file.
- **Evaluating at different step counts** — "GRPO at 200 steps vs PPO at 500 steps" is a different question. Equal sample budget = equal max_steps = equal gradient-update count.
- **Skipping the lr grid** — "use the same lr" is the most common source of unfair RL algorithm comparison. Every algorithm needs at least a narrow lr sweep.

## Algorithm Comparison Table

The shortlist below was chosen to satisfy four constraints simultaneously:

1. **Applicable to sequence-level scalar reward on decoder-only ASR** — no pairwise preferences, no reward model.
2. **Fits in T4 VRAM at G=2, batch_size=1** — no separate critic, no extra model copy.
3. **Implementable as a ~50-line LossModule** — reuses Phase 4 generation, log-prob, and ref-model code unchanged.
4. **Scientifically interesting** — each represents a distinct point on the "group baseline vs global baseline vs clipping vs KL regularization" design spectrum.

| # | Algorithm | Canonical paper / impl. | Advantage computation | Loss | Needs critic? | Needs pairwise data? | Needs ref model? | VRAM vs GRPO | Family |
|---|-----------|-------------------------|----------------------|------|---------------|---------------------|-------------------|--------------|--------|
| 1 | **Dr. GRPO** | Liu et al. 2025 [CITED: arxiv:2503.20783] ; code: [sail-sg/understand-r1-zero](https://github.com/sail-sg/understand-r1-zero) | `A_i = r_i − mean_g(r)` (no std, no length normalization) | Clipped ratio: `min(ρ·A, clip(ρ,1−ε,1+ε)·A)` + optional β·KL | No | No | Yes (PEFT disable_adapter_layers) | ≡ (reference) | Group-relative |
| 2 | **RLOO** | Ahmadian et al. 2024 [CITED: arxiv:2402.14740] ; TRL 0.22+ docs (causal LMs only) | `A_i = r_i − (1/(G−1))·Σ_{j≠i} r_j` (leave-one-out baseline) | REINFORCE: `−A_i·log π(o_i\|q)`; optionally clipped if num_iterations>1 | No | No | Yes (β·KL penalty) | ≡ (identical cost) | Leave-one-out baseline |
| 3 | **REINFORCE++** | Hu et al. 2025 [CITED: arxiv:2501.03262] ; OpenRLHF impl. | `A_i = (r_i − mean_batch(r)) / std_batch(r)` — **global** normalization across all prompts × G, not per-prompt | Clipped ratio (PPO-style) + β·KL | No | No | Yes | ≡ (identical cost) | Global baseline + PPO clip |
| 4 | **PPO (group-mean baseline)** | Schulman et al. 2017; impl. details: Huang et al. 2022 [CITED: ICLR Blog Track — "37 Implementation Details of PPO"] | `A_i = r_i − mean_g(r)` (SAME as Dr. GRPO — this is the key control) | Clipped ratio `min(ρ·A, clip(ρ,1−ε,1+ε)·A)` + β·KL + (optional) entropy bonus 0.01·H | **NO — group mean replaces critic** | No | Yes | ≡ (identical cost — no critic model) | Clipped ratio + KL |

**Why only four algorithms:**
- With 4 algorithms × 3 seeds × 200 steps × ≈ 30 min/run, total T4 time is **≈ 6–12 hours**. Plus one lr grid (12 runs × 50 steps × ≈ 7 min) ≈ 1.5 hours. Total: **≈ 8–16 hours** which fits in one overnight session.
- Adding a fifth algorithm roughly linearly increases compute — not worth it for marginal scientific value.
- **Dr. GRPO vs PPO (group-mean baseline) is the most scientifically interesting comparison:** they use the *identical* advantage computation. The only difference is the loss — PPO adds the clipped ratio and the entropy bonus on top of the same advantage. This isolates what the PPO machinery actually buys over a plain REINFORCE with group-mean baseline, which is exactly what the literature has been arguing about since 2024 [CITED: Ahmadian et al. 2024].
- **RLOO vs Dr. GRPO isolates the effect of the baseline:** leave-one-out (unbiased, lower variance for small G) vs plain group mean. Dr. GRPO's advantage is mathematically equivalent to RLOO up to a constant scale factor when G=2 — so at G=2, on paper, they should give identical gradients. Any difference we observe is due to implementation details (importance ratio handling, loss aggregation) which is exactly the kind of "37 implementation details" phenomenon we want to document.
- **REINFORCE++ vs GRPO isolates global vs per-prompt normalization:** REINFORCE++'s claim is that global batch normalization is more robust on heterogeneous prompt sets [CITED: Hu et al. 2025]. Our dataset has two sub-populations (FS and CV) with different accent distributions — this is exactly the setting where the claim should matter. If REINFORCE++ wins, we learn something about ASR heterogeneity. If it doesn't, we learn that GRPO is robust enough.

### Algorithms Considered But Rejected (with rationale)

| Algorithm | Why rejected |
|-----------|--------------|
| **PPO with a separate value / critic network** | Doubles VRAM (adds a second model head). Explicitly out of scope per `.planning/REQUIREMENTS.md`. Also not a "fair" comparison since the other algorithms are critic-free. |
| **DPO / IPO / SimPO / ORPO** | Pairwise preference methods. Require converting our scalar composite reward into (chosen, rejected) pairs — information-lossy. DPO also requires the **same reference model** as the policy initialization, which we already have (the SFT adapter), but the data pipeline is completely different (offline rollouts → preference pairs → separate trainer). Additional complication: TRL DPO has no audio trainer. Including DPO would require building an entire second data pipeline for one algorithm — not worth it. **Fallback path if user insists:** see "Discretionary Extension" section below. [CITED: DPO: Rafailov et al. 2023; KTO paper notes that "DPO performance can even be matched at most scales by running an offline PPO variant on dummy +1/-1 rewards" — which is effectively an RLOO run, and we already have that in the shortlist.] |
| **KTO** | Requires *binary* desirable/undesirable labels, not scalar rewards. Would need `reward > threshold → desirable`, another arbitrary binarization. Also marked *experimental* in TRL v1.0 [CITED: TRL KTO docs]; no audio support. |
| **RAFT / Rejection sampling fine-tuning** | Already planned as Phase 6 ("Rejection Sampling — Fairness-Aware SFT") per ROADMAP.md. Including it here would duplicate work. |
| **ReMax** | Replaces the baseline with a *single* greedy rollout's reward [CITED: Li et al. 2023 arxiv:2310.10505]. At G=2, ReMax is functionally nearly indistinguishable from RLOO with the greedy one as the "other" sample. Marginal scientific value on top of RLOO. |

Including the full DAPO would need per-knob ablation. Deferred unless an entropy-collapse signal appears during Phase 04.1 runs. |
| **GSPO (Group Sequence Policy Optimization)** | GSPO operates at the full-sequence importance ratio rather than per-token. Useful for very long sequences, but ASR transcriptions are short (< 128 tokens in Phase 4). Marginal value. |
| **PPO-Max / TRL `PPOTrainer` (v1 RLHF)** | The legacy TRL PPO loop is being deprecated and was never compatible with non-causal-LM models anyway [CITED: TRL docs on trainer deprecation]. Do not use. |

### Explicitly Not Varied (these are knobs we deliberately do not touch)

- Number of PPO iterations per batch (`num_iterations`): fixed at 1 (single-iteration online) across all algorithms. Matches Phase 4 `mu=1`.
- Advantage whitening beyond what each algorithm prescribes (no extra `(A − mean) / std` pass on top of RLOO's leave-one-out baseline).
- Asymmetric clipping / DAPO clip-higher: fixed symmetric ε=0.2.
- Entropy bonus: off (set to 0.0). Including it introduces another knob and only PPO conventionally has it — we set it to 0 to keep PPO's loss comparable to the others.
- Early stopping: off. All runs go to exactly 200 steps so sample budgets match.

### Sample Budget Equity

Given the four algorithms all consume the same number of generations per step (G=2 candidates per prompt) and the same number of gradient updates (200 steps × gradient_accumulation=4 = 800 optimizer updates), the sample budget is automatically matched per algorithm. This is the most important equity control.

- **Number of environment interactions (rollouts):** 200 × 1 (batch_size) × 2 (G) = 400 per seed
- **Number of reward queries (RewardComputer calls):** 200 × 2 = 400 per seed
- **Number of gradient updates:** 200 per seed (with gradient_accumulation=4)
- **Number of reference-model forward passes:** 200 × 2 (once per G for teacher-forcing) = 400 per seed

Every algorithm sees the same counts. Wall-clock may differ slightly (PPO has a marginally heavier loss computation but should be within ± 10 %), and that is the correct metric to report separately.

### Seed Count Justification

- **Minimum (must have):** 3 seeds per (algorithm × lr) cell. Three is the minimum that allows any form of bootstrap CI [CITED: Henderson et al. 2018 arxiv:1709.06560 — "most deep RL papers use ≤ 5 seeds and this is insufficient"; Colas et al. 2018 arxiv:1806.08295 — statistical power analysis].
- **Preferred (if T4 time permits):** 5 seeds for the final head-to-head, which is the standard in the "Deep RL at the Edge of the Statistical Precipice" paper [CITED: Agarwal et al. NeurIPS 2021 arxiv:2108.13264 — IQM over 5+ seeds].
- **Planner decision rule:** Start with 3 seeds. If total wall-clock for 4 × 3 = 12 runs finishes in under 8 hours, extend to 5 seeds and re-run. If it takes longer, stick with 3 and note the limitation.

### Learning-Rate Grid Protocol

The one HP we tune per algorithm. Everything else is fixed.

1. For each algorithm, run 3 *short* probes at lr ∈ {1e-6, 1e-5, 3e-5} with a single seed (42), 50 steps, 200-sample subset. Total: 4 × 3 = 12 short runs, ≈ 7 min each, ≈ 1.5 hours wall-clock.
2. Pick the lr per algorithm with the highest mean reward in the last 10 logged steps (and NaN-free loss).
3. Use that lr for the full 3-seed 200-step comparison.
4. Log the chosen lr per algorithm in `outputs/rl-comparison/lr_grid_winners.json` so reviewers can audit the comparison.

This protocol matches the "Back to Basics" RLOO paper methodology [CITED: Ahmadian 2024] which tunes lr per algorithm and holds everything else fixed.

## Evaluation Protocol

### Metric Set (reported per run, per algorithm, per seed)

| Metric | Where computed | Purpose |
|--------|---------------|---------|
| Final reward (mean over last 10 training log entries) | `training_log.json` | Training-time signal, NOT a final verdict |
| Final dev WER (overall) | `evaluate_adapter.py` on FS + CV dev subsets | Accuracy, final verdict |
| Final per-group WER | `evaluate_adapter.py` | Fairness, final verdict |
| Final WER gap (max − min across groups) | computed in aggregator | Primary fairness metric |
| Final WER max-min ratio | computed in aggregator | Secondary fairness metric |
| Training wall-clock (min) | `training_summary.json.training_time_sec / 60` | Efficiency |
| Peak VRAM (GB) | `training_summary.json.peak_vram_gb` | Budget compliance (must be < 13 GB) |
| Trainable parameters (count) | Printed at model load | Same for all — sanity check |
| frac_reward_zero_std (mean last 10) | `training_log.json` | Signal quality — unique-answer problem detector |
| advantage_std (mean last 10) | `training_log.json` | Signal quality — gradient magnitude detector |
| Entropy at final step | `training_log.json` | Entropy-collapse detector [CITED: PITFALLS.md §8] |
| Mean completion length (final step) | `training_log.json.completions/mean_length` | Length-bias / reward-hacking detector |
| `clip_frac` (fraction of ratios hitting the clip boundary) | `training_log.json` | PPO / GRPO / REINFORCE++ comparability; should be < 0.2 in a healthy run |
| Sample efficiency proxy: step at which reward reaches 90 % of final reward | derived from `training_log.json` | Efficiency comparison |

### Aggregation Across Seeds

For every (algorithm, metric) pair:

1. Compute per-seed final value.
2. Report the **Interquartile Mean (IQM)** with bootstrap 95 % CI [CITED: Agarwal et al. NeurIPS 2021 arxiv:2108.13264 — "Deep RL at the Edge of the Statistical Precipice" — IQM is more robust than mean, more statistically efficient than median, recommended for few-seed settings].
3. Also report the arithmetic mean and standard error (for readers who want the classic view).
4. Produce a learning-curve plot: reward-vs-step, one curve per algorithm, shaded band = IQM ± 95 % bootstrap CI across seeds.

With only 3 seeds, IQM is equivalent to the mean (you cannot trim with 3 points). The **plan should flag this explicitly and preferred-recommend 5 seeds** if compute allows, because IQM is only meaningful at ≥ 5.

### Statistical Test: Bootstrap Pairwise Comparison vs GRPO

The reference algorithm is Dr. GRPO (Phase 4's prototype). Every other algorithm is tested pairwise against it.

For each metric (final WER, final gap, final reward):

1. Collect the 3 (or 5) per-seed values for GRPO and for the challenger.
2. Compute the **difference of means** `Δ = mean(challenger) − mean(GRPO)`.
3. Bootstrap the distribution of Δ by sampling seeds with replacement 10 000 times.
4. Report the 95 % CI of Δ and the p-value (fraction of bootstrap samples with Δ ≥ 0 if challenger ought to be worse on WER; ≤ 0 if better on fairness).
5. Per Henderson et al. 2018: ALSO report a 2-sample t-test and Mann–Whitney U for robustness [CITED: arxiv:1709.06560].

Use `scipy.stats.bootstrap` and `scipy.stats.mannwhitneyu` — both are in scipy 1.17.1 [VERIFIED: pip show].

**Honest reporting requirement:** With N=3 seeds, no pairwise test will reach traditional p < 0.05 unless the effect is very large. The plan should warn the reader that these bootstraps are descriptive, not conclusive, and that the value of the table lies in showing the *direction* of differences and the *overlap* of CIs. This is the entire point of Henderson 2018 and "statistical precipice" Agarwal 2021 — do not sell underpowered tests as conclusive.

### Plots (produced once at phase end)

1. **Reward learning curves** — one panel, four lines (one per algorithm), x=step, y=mean reward, shaded band = bootstrap 95 % CI across seeds.
2. **Pareto scatter** — one point per (algorithm, seed), x=WER, y=fairness gap, color=algorithm, ellipse = 95 % CI ellipse around the per-algorithm centroid.
3. **Wall-clock efficiency** — bar chart, one bar per algorithm, y=min to reach 90 % of its own final reward.
4. **VRAM bar chart** — one bar per algorithm, peak VRAM across seeds (should be nearly identical — a large difference signals a bug).

### Reward-Hacking Checks (gate)

Per PITFALLS.md §2, §3, §8, run this detector per adapter:

- `wer_floor_breach`: final mean WER > `baseline_wer * 1.2`
- `length_explosion`: final mean completion length > 2 × initial length
- `vocabulary_collapse`: unique token count in final completions < 50 % of initial (proxy for determinism)
- `entropy_collapse`: final entropy < 0.3 × initial entropy

If any check fires for any run, that run is marked `DEGRADED` in the report and excluded from the "winner" decision but still reported. This is a fairness-to-the-reader control, not a fairness-of-comparison control.

## Common Pitfalls

### Pitfall 1: Per-algorithm KL coefficient differences silently determine the winner

**What goes wrong:** PPO literature recommends β ≈ 0.01–0.1 depending on the task; GRPO literature often sets β = 0.04; REINFORCE++ defaults to 0.01; RLOO in TRL defaults to β=0.05 (called `beta` in RLOOConfig) [CITED: TRL RLOOTrainer docs]. If you "use each algorithm's default KL," you are not comparing the algorithms — you are comparing how each author tuned KL on their own benchmark.
**Why it happens:** Every paper has its own "sensible default."
**How to avoid:** Hold β = 0.04 **identical across all algorithms** (matching Phase 4 and Amazon ASRU 2025). Document this prominently. If a reviewer asks "but REINFORCE++ is supposed to run at 0.01," the answer is "we matched it across algorithms for a fair comparison; per-algorithm KL tuning is an orthogonal ablation."
**Warning signs:** One algorithm shows massively lower KL divergence than the others → its effective β is different because its loss is different and the KL term's relative weight changes. Log both raw `kl` and `β * kl` for comparability.
**Confidence:** HIGH [CITED: arxiv:2510.01555 "Rethinking KL Regularization in RLHF" — β dominates behavior].

### Pitfall 2: Per-algorithm learning rate *not* tuned → REINFORCE-family looks bad

**What goes wrong:** Plain REINFORCE and RLOO have different gradient magnitudes than PPO (no clipping → raw gradient) and Dr. GRPO (has implicit std-normalization in the "standard" variant; Dr. GRPO removes it, changing magnitude). A single shared lr will favor whichever algorithm's effective gradient magnitude happens to match that lr.
**Why it happens:** Lazy "use lr=1e-5 for everyone" because Phase 4 used it for GRPO.
**How to avoid:** The lr-grid protocol above. Tune lr per algorithm on a 50-step 1-seed probe before the main comparison. Log chosen lr per algorithm.
**Warning signs:** One algorithm's training loss diverges (NaN) while others are stable → its lr is too high. One algorithm's reward is flat → its lr is too low.
**Confidence:** HIGH [CITED: Ahmadian et al. 2024 — their main experimental claim is that "careful lr tuning per algorithm closes the gap between PPO and REINFORCE"; Huang 2022 "37 details" — lr is the single most important PPO HP].

### Pitfall 3: Single-seed "winners" from the lr grid

**What goes wrong:** The lr-tuning grid uses 1 seed to save time. If the winning lr is a noise peak for one algorithm and a local minimum for another, the downstream 3-seed comparison is biased.
**Why it happens:** Compute budget pressure.
**How to avoid:** (a) Use seed 42 consistently for all lr probes so the bias is at least directionally consistent across algorithms. (b) If the chosen lr is ≥ 2× away from the grid center (1e-5), flag the run for a second-seed confirmation. (c) In the final report, explicitly list "lr grid used seed=42 only" as a limitation.
**Warning signs:** The winning lr differs by > 1 order of magnitude across algorithms → at least one is a noise artifact. Re-run that algorithm's lr probe with seed 43.
**Confidence:** MEDIUM (best-practice, project-specific).

### Pitfall 4: Unique-answer problem is algorithm-independent → same zero-advantage trap as Phase 4

**What goes wrong:** Every algorithm in the shortlist operates on the same reward signal and the same G=2 candidates. If Phase 4's unique-answer problem (frac_reward_zero_std > 0.5) persists, ALL four algorithms will fail equally.
**Why it happens:** The root cause is the reward signal, not the algorithm.
**How to avoid:** (a) REUSE the Phase 4 continuous-WER reward and temperature=0.8, which Phase 4 validated resolves the problem. (b) Include the **Phase 4 prototype signal-validation gate** as a pre-flight check for Phase 04.1: if Phase 4's last run had frac_zero_std > 0.5, Phase 04.1 will inherit the problem and the plan should escalate to the user before launching.
**Warning signs:** All four algorithms show flat reward curves → the reward signal is dead, not the algorithms.
**Confidence:** HIGH [CITED: Phase 4 RESEARCH.md Pitfall 1, Amazon ASRU 2025 arxiv:2509.01939].

### Pitfall 5: Reference-model drift across seeds

**What goes wrong:** Each run reloads the Phase 3 SFT adapter fresh (good), but if the adapter save/load is non-deterministic (e.g., some buffers are not loaded back), different runs start from very slightly different initial weights.
**Why it happens:** PEFT adapter save/load has had quirks with module-name matching across versions; also gradient-checkpointing state can leak.
**How to avoid:** (a) After loading, hash the state_dict of the base model and the LoRA adapter modules and log the hash. If two runs have different hashes, the load is non-deterministic. (b) Use `torch.use_deterministic_algorithms(True)` + `CUBLAS_WORKSPACE_CONFIG=:4096:8` env var for fully deterministic runs (accepting a small speed penalty).
**Warning signs:** Same seed, same algorithm, different runs produce different step-1 losses → non-deterministic load.
**Confidence:** MEDIUM (has happened in published RLHF replications).

### Pitfall 6: Length bias disguised as "PPO is better"

**What goes wrong:** PPO's clipped loss sums tokens, so longer completions get more weight. In ASR, one algorithm may just learn to emit longer transcriptions, and longer transcriptions sometimes accidentally match more reference tokens → slightly better WER for the wrong reason.
**Why it happens:** Token-level loss aggregation without length normalization [CITED: Dr. GRPO paper arxiv:2503.20783 §3.2 — exactly this bug in GRPO].
**How to avoid:** (a) All four algorithms in the shortlist use the Dr. GRPO length normalization (divide by a fixed max length constant, not by the actual sequence length). (b) Log `completions/mean_length` per algorithm per step and flag any run where the final length > 1.5× initial.
**Warning signs:** Mean completion length creeping up across training, especially after step 100.
**Confidence:** HIGH [CITED: Dr. GRPO arxiv:2503.20783].

### Pitfall 7: Entropy collapse from the PPO clip

**What goes wrong:** PPO's symmetric clip `[0.8, 1.2]` compresses rare-token probability updates (a token with p=0.01 can only move to p=0.012 per step), systematically suppressing exploration. The model becomes increasingly deterministic; once entropy drops below ~30 % of initial, all G candidates become identical and the algorithm grinds to a halt [CITED: .planning/research/PITFALLS.md §8; DAPO paper].
**Why it happens:** Symmetric ε=0.2 is lazy; PPO literature has moved to asymmetric `[0.8, 1.28]` (DAPO clip-higher).
**How to avoid:** (a) Since we are holding ε=0.2 constant across algorithms (for comparability), entropy collapse is a RISK for all of them equally. (b) Log entropy every step. (c) If any run drops entropy below 0.3× initial before step 150, mark that run DEGRADED and note in the report that the symmetric clip choice may be causing it. (d) The Phase 04.1 plan should include an explicit recommendation: *if entropy-collapse is observed in the core run, schedule a follow-up phase with asymmetric clipping ε_high=0.28*.
**Warning signs:** Entropy curve monotonically decreasing; `frac_reward_zero_std` increasing back toward 1.0 after step 100.
**Confidence:** HIGH [CITED: DAPO paper; Dr. GRPO paper].

### Pitfall 8: "Comparing to Phase 4 GRPO" requires the *same* Phase 4 code

**What goes wrong:** The refactor from `train_grpo.py` to `train_rl.py` introduces subtle differences (e.g., a different loss aggregation, a different masking policy), so the Dr. GRPO loss module in Phase 04.1 does not exactly match Phase 4's GRPO run. The comparison inherits a confounder.
**Why it happens:** Refactors introduce bugs.
**How to avoid:** (a) As the FIRST task of Phase 04.1, port Phase 4 `grpo_step()` lines 380–500 *verbatim* into `DrGRPOLoss.compute_loss()`. (b) Run `DrGRPOLoss` with seed=42 and verify that the reward curve matches Phase 4's `outputs/grpo-prototype/lambda_0.3/training_log.json` within 0.5 % at every logged step. If not, the refactor is wrong — fix before proceeding. (c) This is the **regression test** for the dispatcher.
**Warning signs:** Dr. GRPO reward curve in Phase 04.1 differs from Phase 4's at step 0 → the initial model is different (adapter load bug). Differs only at step 50+ → the loss aggregation is different.
**Confidence:** HIGH (this is the classic refactor-pitfall pattern).

### Pitfall 9: Sample-budget mismatch via silent NaN skips

**What goes wrong:** Some algorithms (especially PPO with ε=0.2 and low KL) can produce NaN losses when all G candidates get the same reward (zero advantage → zero loss → NaN from a downstream division). The training loop might silently skip those steps, leaving one algorithm with fewer effective gradient updates.
**Why it happens:** Defensive `if loss.isnan(): continue` in training loops.
**How to avoid:** (a) Log a counter `n_nan_skipped` per run. (b) In the comparison, report it as a column. (c) If `n_nan_skipped > 5` for any run, mark it DEGRADED and investigate.
**Warning signs:** Wall-clock for one algorithm is noticeably shorter than the others with identical max_steps → it's skipping steps.
**Confidence:** MEDIUM.

### Pitfall 10: Reporting single-number "winners" from underpowered experiments

**What goes wrong:** With 3 seeds, the 95 % CI for the difference of means between two algorithms will almost always include zero — you cannot statistically distinguish them. Reporting "PPO beats GRPO by 0.3 % WER" is misleading.
**Why it happens:** Researchers want a clean story.
**How to avoid:** Every numeric claim in the final report MUST be accompanied by (a) a 95 % bootstrap CI, and (b) a sentence about whether CIs overlap. The plan should include a lint check that ensures every number in the experiment log has CI shown alongside.
**Warning signs:** Report draft contains a sentence like "X is the best" without a CI.
**Confidence:** HIGH [CITED: Henderson et al. 2018; Agarwal et al. 2021].

```python
# scripts/training/rl_losses/dr_grpo_loss.py
# Source: VERBATIM port of Phase 4 grpo_step() lines 380-500
# [VERIFIED: scripts/training/train_grpo.py:336-500]
import torch
from .base import LossModule, LossInputs


class DrGRPOLoss(LossModule):
    """Dr. GRPO — advantage = reward - group mean, no std division.

    Reference: Liu et al. 2025, "Understanding R1-Zero-Like Training"
    [CITED: arxiv:2503.20783]
    """

    name = "dr_grpo"

    def compute_advantages(self, rewards: torch.Tensor) -> torch.Tensor:
        # Dr. GRPO: subtract group mean, do NOT divide by std
        return rewards - rewards.mean(dim=1, keepdim=True)  # (B, G)

    def compute_loss(self, inputs, advantages):
        total_loss = torch.zeros((), device=advantages.device)
        total_kl = 0.0
        total_clip_frac = 0.0
        total_entropy = 0.0
        n_tokens = 0
        B, G = advantages.shape

        for g in range(G):
            cur_lp = inputs.current_log_probs[g]   # (B, gen_len)
            old_lp = inputs.old_log_probs[g]       # (B, gen_len)
            mask = inputs.response_masks[g]        # (B, gen_len)
            for i in range(B):
                adv_i = advantages[i, g]  # scalar
                cur = cur_lp[i][mask[i].bool()]
                old = old_lp[i][mask[i].bool()]
                if cur.numel() == 0:
                    continue
                # Per-token ratio
                ratio = torch.exp(cur - old)
                # Clipped surrogate (PPO clip, as in Phase 4)
                eps = inputs.config.epsilon
                clipped = torch.clamp(ratio, 1 - eps, 1 + eps)
                pg_loss = -torch.min(ratio * adv_i, clipped * adv_i).mean()
                total_loss = total_loss + pg_loss
                total_clip_frac += (ratio != clipped).float().mean().item()
                total_entropy += (-cur).mean().item()  # proxy: -log π for chosen tokens
                n_tokens += cur.numel()
                if inputs.config.beta > 0 and inputs.ref_log_probs is not None:
                    ref = inputs.ref_log_probs[g][i][mask[i].bool()]
                    # Schulman k3 estimator
                    log_ratio = ref - cur
                    kl = (torch.exp(log_ratio) - log_ratio - 1).mean()
                    total_loss = total_loss + inputs.config.beta * kl
                    total_kl += kl.item()

        total_loss = total_loss / max(B * G, 1)
        metrics = {
            "loss": total_loss.item(),
            "advantage_mean": advantages.mean().item(),
            "advantage_std": advantages.std().item(),
            "ratio_mean": 1.0,  # placeholder — computed per-step in practice
            "clip_frac": total_clip_frac / max(B * G, 1),
            "kl": total_kl / max(B * G, 1),
            "entropy": total_entropy / max(B * G, 1),
        }
        return total_loss, metrics
```

### Example 2: RLOO loss module (leave-one-out baseline)

```python
# scripts/training/rl_losses/rloo_loss.py
# Source: Ahmadian et al. 2024, "Back to Basics: Revisiting REINFORCE Style Optimization"
# [CITED: arxiv:2402.14740, Eq. 6; TRL RLOOTrainer docs (Computing the advantage)]
import torch
from .base import LossModule, LossInputs


class RLOOLoss(LossModule):
    """RLOO — leave-one-out baseline, otherwise same structure as GRPO.

    Advantage: A_i = r_i - (1/(G-1)) * sum_{j != i} r_j
    Loss: REINFORCE (no clipping by default; clipping only if num_iterations > 1)
    """

    name = "rloo"

    def compute_advantages(self, rewards: torch.Tensor) -> torch.Tensor:
        # rewards: (B, G)
        B, G = rewards.shape
        if G < 2:
            return rewards - rewards.mean(dim=1, keepdim=True)
        # sum over G, then subtract the current element, divide by (G-1)
        total = rewards.sum(dim=1, keepdim=True)  # (B, 1)
        loo_baseline = (total - rewards) / (G - 1)  # (B, G)
        return rewards - loo_baseline

    def compute_loss(self, inputs, advantages):
        # RLOO reduces to REINFORCE under num_iterations=1 (our setting).
        # Same outer structure as DrGRPOLoss, but with clipping optional.
        # We keep the PPO clip active because inputs.config.epsilon is held
        # constant across algorithms for comparability.
        # Delegate to the Dr. GRPO loss body — the only thing that changed
        # is compute_advantages() above.
        return DrGRPOLoss().compute_loss(inputs, advantages)
```

### Example 3: REINFORCE++ loss module (global batch normalization)

```python
# scripts/training/rl_losses/reinforce_pp_loss.py
# Source: Hu et al. 2025, "REINFORCE++: An Efficient RLHF Algorithm"
# [CITED: arxiv:2501.03262; OpenRLHF implementation]
import torch
from .base import LossModule


class ReinforcePPLoss(LossModule):
    """REINFORCE++ — normalize advantages globally across the batch.

    Advantage: A_i = (r_i - mean_batch(r)) / (std_batch(r) + eps)
      where mean_batch and std_batch are computed over the ENTIRE batch*G,
      NOT per-prompt (the key difference vs GRPO).
    """

    name = "reinforce_pp"

    def compute_advantages(self, rewards: torch.Tensor) -> torch.Tensor:
        flat = rewards.reshape(-1)  # (B*G,)
        mean = flat.mean()
        std = flat.std().clamp(min=1e-8)
        normalized = (flat - mean) / std
        return normalized.reshape_as(rewards)

    def compute_loss(self, inputs, advantages):
        # Same loss body as Dr. GRPO (PPO clip + KL penalty).
        return DrGRPOLoss().compute_loss(inputs, advantages)
```

### Example 4: PPO (group-mean baseline) loss module

```python
# scripts/training/rl_losses/ppo_groupmean_loss.py
# Source: Schulman et al. 2017 PPO; critic replaced with group mean
# [CITED: Huang et al. 2022 ICLR Blog — "The 37 Implementation Details of PPO"]
import torch
from .base import LossModule


class PPOGroupMeanLoss(LossModule):
    """PPO with group-mean baseline replacing the critic network.

    Advantage: identical to Dr. GRPO (reward - group mean), but the loss
    includes the PPO entropy bonus to isolate "what entropy regularization
    buys you" from "what group-relative advantage buys you."

    NOTE: with entropy_coef=0.0 (our setting to hold things constant), this
    degenerates to Dr. GRPO's loss. That is intentional — it becomes the
    regression test for the refactor.
    """

    name = "ppo_groupmean"

    def compute_advantages(self, rewards: torch.Tensor) -> torch.Tensor:
        return rewards - rewards.mean(dim=1, keepdim=True)  # (B, G)

    def compute_loss(self, inputs, advantages):
        # Same loss body as Dr. GRPO, plus an entropy bonus (coef set to 0.0
        # during the core comparison to isolate the advantage-computation
        # effect; set to 0.01 in a follow-up ablation).
        loss, metrics = DrGRPOLoss().compute_loss(inputs, advantages)
        entropy_coef = getattr(inputs.config, "entropy_coef", 0.0)
        if entropy_coef > 0:
            loss = loss - entropy_coef * metrics["entropy"]
        return loss, metrics
```

### Example 6: Bootstrap pairwise comparison (for the final report)

```python
# scripts/analysis/bootstrap_pairwise.py
# Source: Agarwal et al. 2021 rliable methodology; scipy.stats.bootstrap
# [CITED: arxiv:2108.13264; scipy 1.17.1]
import json
import numpy as np
from scipy import stats


def pairwise_vs_reference(metric_name: str, reference: list[float], challenger: list[float],
                          n_bootstrap: int = 10_000, alpha: float = 0.05) -> dict:
    """Bootstrap CI on mean(challenger) - mean(reference) over seeds."""
    ref = np.asarray(reference, dtype=float)
    cha = np.asarray(challenger, dtype=float)
    diff = cha.mean() - ref.mean()

    rng = np.random.default_rng(42)
    boot_diffs = np.empty(n_bootstrap)
    for i in range(n_bootstrap):
        ref_boot = rng.choice(ref, size=len(ref), replace=True)
        cha_boot = rng.choice(cha, size=len(cha), replace=True)
        boot_diffs[i] = cha_boot.mean() - ref_boot.mean()

    lo, hi = np.percentile(boot_diffs, [100 * alpha / 2, 100 * (1 - alpha / 2)])

    # Classical tests alongside (Henderson 2018 recommendation)
    t_stat, t_p = stats.ttest_ind(cha, ref, equal_var=False)
    u_stat, u_p = stats.mannwhitneyu(cha, ref, alternative="two-sided")

    return {
        "metric": metric_name,
        "n_reference_seeds": int(ref.size),
        "n_challenger_seeds": int(cha.size),
        "reference_mean": float(ref.mean()),
        "reference_std": float(ref.std(ddof=1)) if ref.size > 1 else 0.0,
        "challenger_mean": float(cha.mean()),
        "challenger_std": float(cha.std(ddof=1)) if cha.size > 1 else 0.0,
        "delta_mean": float(diff),
        "delta_95ci": [float(lo), float(hi)],
        "ci_includes_zero": bool(lo <= 0 <= hi),
        "welch_t_p": float(t_p),
        "mannwhitney_u_p": float(u_p),
    }
```

## State of the Art

| Old approach | Current (2025-2026) approach | When changed | Impact for Phase 04.1 |
|--------------|------------------------------|--------------|----------------------|
| PPO with a separate value / critic network for RLHF | Group-relative / leave-one-out baselines (GRPO, RLOO, Dr. GRPO, REINFORCE++) — **no critic** | 2024 (Ahmadian RLOO paper; DeepSeekMath GRPO) | We use critic-free variants only, matching T4 budget and project scope |
| Mean ± std reporting over few seeds | IQM with bootstrap CIs over ≥ 5 seeds | 2021 (Agarwal et al. NeurIPS) | Plan requires ≥ 3 seeds, aspires to 5, reports IQM + bootstrap CIs |
| Symmetric PPO clip ε=0.2 | Asymmetric DAPO clip `[1−0.2, 1+0.28]` (clip-higher) | 2025 (DAPO paper) | Deferred — keep symmetric ε=0.2 for comparability; flag asymmetric clipping as a follow-up if entropy collapse is observed |
| GRPO std normalization of advantages | Dr. GRPO: remove std normalization to eliminate difficulty bias | 2025 (arxiv:2503.20783) | Phase 4 already switched; Phase 04.1 uses Dr. GRPO as its reference |
| KL penalty mandatory (β > 0) | β=0 increasingly common in reasoning RL | 2025 (DAPO, Dr. GRPO) | We keep β=0.04 to maintain ref-model anchoring; ablating β to 0 is a valid follow-up |
| Single-sample REINFORCE with learned value | RLOO with leave-one-out baseline over G samples | 2024 (Ahmadian) | RLOO is in our shortlist |
| Algorithm-specific default HPs | Per-algorithm lr tuning, everything else held fixed | 2024 (Ahmadian)  | We follow this protocol in the lr-grid phase |

**Key insight for Phase 04.1:** The RL-for-LLM comparison literature in 2024–2026 has converged on a simple message — **for a fair comparison, hold everything fixed except per-algorithm lr, use ≥ 5 seeds, use IQM, and report bootstrap CIs**. Our phase is a direct application of this methodology to the ASR fairness setting, which (to my knowledge) has not yet been done in a published paper.

**Deprecated / outdated:**
- TRL `PPOTrainer` (v1 RLHF, pre-v1.0) — use modern trainers (RLOO, GRPO) if using TRL, but see our TRL exclusion note.
- "GRPO with std normalization" — superseded by Dr. GRPO.
- Single-seed RL benchmarks — indefensible in 2026.

## Decisions For User

These are the decision points the user should make before the planner writes tasks. Each has a recommendation and a fallback.

1. **Seed count: 3 or 5?**
   - **Recommend 3 seeds** initially (≈ 8-hour overnight run). Extend to 5 if wall-clock under 8 hours.
   - **Fallback:** 3 only, note limitation in report.
   - **Why it matters:** Statistical power. 3 is the minimum Henderson 2018 considers; 5 is the minimum Agarwal 2021 considers adequate for IQM.

2. **Include a preference-style baseline (DPO / KTO) via offline rollouts?**
   - **Recommend NO** for the core comparison — it requires a separate data-generation step, scalar→binary conversion, a different trainer, and does not integrate with our existing custom loop. Also rejected by the four-constraint filter above.
   - **Fallback if user insists:** Run an **offline best-vs-worst DPO** as a separate secondary comparison, NOT in the main 4-algorithm table. Generate K=8 rollouts from the SFT adapter for every training prompt, pick the highest-reward and lowest-reward, train DPO with β=0.1, and report it as "offline DPO vs GRPO" in a separate section. Time cost: ~2 hours of rollout generation + ~1 hour of DPO training per seed, so 3 seeds ≈ 9 hours extra. Not worth it unless a reviewer specifically asks for it.

3. **λ value for the core comparison: single value or 3 values?**
   - **Recommend single λ=0.3** for the core 4-algorithm comparison to save compute. Then run the winning algorithm across λ ∈ {0, 0.3, 0.7} as a secondary sweep.
   - **Fallback:** All 4 algorithms × all 3 λ values × 3 seeds = 36 runs. Adds ~24 hours. Only do this if the user wants a 3D Pareto.
   - **Why it matters:** With 1 λ and 4 algorithms, we can answer "which RL algorithm is best at the mid-Pareto point." With 3 λ, we can answer "which RL algorithm dominates the Pareto frontier" — a stronger claim but 3× the cost.

4. **Use Phase 3 production SFT adapter or Phase 4's prototype adapter?**
   - **Recommend Phase 3 production SFT adapter** (`outputs/full-lora/adapter/`) — as specified in the user prompt. This is the real starting point; Phase 4's lambda=0 adapter is only a prototype that was itself trained by GRPO and has its own biases.
   - **Fallback:** If Phase 3 has not completed by the time Phase 04.1 launches, use Phase 2.1 locked-config SFT adapter instead. Note the substitution in the report.
   - **Why it matters:** Every algorithm starts from the same checkpoint; changing the starting checkpoint changes the reachable loss landscape. Consistency across phases is the value.

5. **G=2 or G=4?**
   - **Recommend G=2** — matches Phase 4, keeps T4 VRAM safe.
   - **Fallback:** If Phase 4 empirically confirmed G=4 fits in T4 VRAM, upgrade all four algorithms to G=4 together. Never mix (G=4 for one algo, G=2 for another) — that is a catastrophic confounder.
   - **Why it matters:** G affects advantage variance, which affects all four algorithms differently. Holding G=2 is required for comparability.

6. **Include an entropy-collapse asymmetric-clip follow-up?**
   - **Recommend only if entropy-collapse is observed** in the core runs.
   - **Fallback:** Skip; document as future work.
   - **Why it matters:** Scope management.

### Known Threat Patterns for This Stack

| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
| Path traversal via `--sft_adapter_path` or `--output_dir` CLI args | Tampering | Resolve paths with `Path.resolve()`, reject paths outside the project root (`pathlib.Path.is_relative_to(project_root)`) |
| Loading a maliciously crafted LoRA adapter / checkpoint (`torch.load` pickle deserialization) | Tampering | Use `weights_only=True` in `torch.load` (transformers 4.57 default), or use `safetensors` (PEFT 0.18 already writes `adapter_model.safetensors`) |
| W&B API key leakage in training logs | Information Disclosure | `wandb.login()` uses `WANDB_API_KEY` env var, never log it; `training_log.json` never contains env vars |
| Reward function side effects (jiwer computes WER over strings — no eval()) | Tampering | jiwer 4.0.0 is pure-Python string ops, no eval/exec, safe |
| GPU OOM denial-of-service when another user's job is co-located | Denial of Service | CLAUDE.md GPU safety gate: run `nvidia-smi`, do not launch if >80 % VRAM used — already enforced project-wide |
| Deterministic but expensive: symlinking the adapter path to a large directory | DoS | Paths are resolved and checked for size; `os.path.getsize()` sanity check |
| Audit trail of which algorithm/seed produced which adapter | Repudiation | `grpo_config.json` / `rl_config.json` saved per run with algorithm + seed + all HPs; git commit hash of train_rl.py embedded in config |

**Net:** Low security surface — single-user research pipeline on a single GPU. The only non-trivial control is pickle-safe checkpoint loading, which PEFT already handles via safetensors.

## Open Questions

1. **Is the Phase 4 dataset subset seed stable, or will regenerating it produce a different 2 K selection?**
   - What we know: `outputs/grpo-prototype/train_subset.csv` is written once by Phase 4's `run_sweep` / `setup_data` and reused across lambdas.
   - What's unclear: What random seed produced the subset — is it fixed?
   - Recommendation: Phase 04.1 should COPY the CSV from `outputs/grpo-prototype/` rather than regenerate it. If the file is missing, halt and regenerate with `SEED=42` deterministically.

2. **How long does one 200-step GRPO run actually take on T4 with the current Phase 4 code?**
   - What we know: Phase 4 plan estimated 30–90 minutes.
   - What's unclear: The actual number from Phase 4's real execution (not yet done).
   - Recommendation: Before committing to 3 or 5 seeds, wait for Phase 4 wall-clock data. Multiply by 4 algorithms × seed count to check budget.

3. **Does Phase 4's GRPO implementation pass the Pitfall 4 log-prob-consistency check?**
   - What we know: Phase 4 research flagged this as Pitfall 4 and documented the solution.
   - What's unclear: Whether the implementation actually started ratio=1.0 at the first iteration.
   - Recommendation: In the Phase 04.1 regression test (Pitfall 8), assert `ratio_mean ≈ 1.0 ± 1e-3` at step 1. If not, fix in the dispatcher before running the comparison.

4. **Is the fairness reward rolling window (fairness_window=10) actually filled before the fairness signal becomes nonzero?**
   - What we know: RewardComputer accumulates 10 batches before computing fairness reward.
   - What's unclear: Whether step 1–10 of training have zero fairness signal (which would disadvantage all algorithms equally in the first 10 steps, not a comparison issue, but a power issue).
   - Recommendation: Log `reward_computer.window_filled` per step; warmup the reward computer once at start by running 10 eval batches through it.

5. **Which G should be used in the PPO-groupmean loss module's compute_advantages?**
   - What we know: G=2 globally.
   - What's unclear: Whether the PPO variant with G=2 has high enough advantage variance for the PPO clip to matter at all (at G=2, the group mean is just `(r_1+r_2)/2`, making advantages `±(r_1-r_2)/2`, often tiny).
   - Recommendation: Honestly document this as a limitation. With G=2, PPO-groupmean and Dr. GRPO are extremely similar (identical advantages, similar losses); the comparison mostly isolates the role of the PPO entropy bonus, which we set to 0 for fairness. If the empirical difference is < 1 %, that is the scientifically honest finding — "at G=2, PPO machinery does not add value over Dr. GRPO." This is publishable.
